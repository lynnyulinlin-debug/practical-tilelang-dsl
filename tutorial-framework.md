## 第一部分：基础与动机

### 第 1 章：Kernel 与 DSL：AI 编译器的动机

#### 1.1 章节定位
回答"做高性能 GPU kernel 为什么不能直接写 CUDA？"这个基础问题。建立读者对 DSL 必要性的认识。

#### 1.2 核心内容

**1.2.1 AI 计算的硬件挑战**
- 访存墙：GPU 内存带宽固定，计算能力持续增长导致的不匹配
  - 数据：A100 理论计算 312 TFLOPS，峰值带宽 2TB/s，计算密度比 = 156 FLOP/Byte
  - 实际应用中很多算子达不到 80% 的理论性能
- 并行墙：线程数量有限，难以充分利用所有计算单元
  - NVIDIA A100：108 SM × 128 线程/SM = 13,824 并行线程
  - 但如果算法的并行度不足，很多线程闲置

**1.2.2 PyTorch 到硬件的鸿沟**
- PyTorch 的 CUDA 算子库有限，新算子需要手写
- CUDA C++ 代码复杂（3000+ 行很常见），容易出错
- cuBLAS/cuDNN 的"黑盒"问题：针对特定 shape 优化困难
- 例：INT8 量化推理算子，NVIDIA 的库可能不提供，需要自己实现

**1.2.3 DSL 的诞生：解决什么问题**
- 生产力：用更少的代码表达高性能 kernel
- 性能：编译器能做的优化比手写 CUDA 更系统
- 可移植性：同一份代码在 NVIDIA/AMD/昇腾 之间切换

**1.2.4 两条 DSL 路线**
```
Triton（高层自动化）：
├─ 特点：block 级抽象，编译器处理细节
├─ 优点：写法简洁，学习快
├─ 缺点：对硬件细节的控制有限
└─ 适用：快速开发、学术探索

TileLang（底层精细控制）：
├─ 特点：tile + kernel 级抽象，开发者精细控制
├─ 优点：性能可控，跨硬件兼容
├─ 缺点：需要理解硬件
└─ 适用：极致调优、多硬件适配
```

**1.2.5 TileLang 的定位**
- 多层级抽象：从计算逻辑 → 分块控制 → 底层调优
- 多硬件后端：统一的 DSL 代码，不同硬件后端编译
- 与 TVM 生态的关系：基于 TVM 的 TIR，更聚焦 AI kernel

#### 1.3 关键数据对比

| 方面 | 手写 CUDA | Triton | TileLang |
|------|---------|--------|----------|
| 代码行数（GEMM） | 500+ | 200 | 300 |
| 开发周期 | 2-3 周 | 2-3 天 | 3-5 天 |
| 性能上限 | ~95% 峰值 | ~85% 峰值 | ~90% 峰值 |
| 跨硬件改动 | 大量改写 | 部分改动 | 很少改动 |
| 学习曲线 | 陡峭 | 平缓 | 中等 |

- Triton：block 级思维

### 第 2 章：GPU 硬件模型与执行层次

#### 2.1 章节定位
建立必要的硬件知识，确保后续讲解有据可依。目标：3-5 页内让读者理解 GPU 执行模型与内存层次。

#### 2.2 核心内容

**2.2.1 NVIDIA GPU 架构（以 A100 为例）**
```
执行层次：
├─ Grid（网格）：所有 thread block 的集合
├─ Block（块）：一组协作的线程，共享 shared memory
│  ├─ Warp（束）：32 个线程，SIMT 执行
│  └─ Thread（线程）：最小执行单元
└─ 数量关系：1 Grid = N Blocks, 1 Block = M Warps, 1 Warp = 32 Threads

内存层次（从快到慢）：
├─ 寄存器：~1 ns，每线程 255×32bit，不能跨线程访问
├─ Shared Memory：~5 ns，96KB（可配为 96+0 或 64+32 L1），可跨线程访问
├─ L2 Cache：~30 ns，40MB 全 GPU 共享
└─ HBM（DRAM）：~200 ns，80GB，理论带宽 2TB/s

计算单元：
├─ CUDA Core：基础算术逻辑单元
├─ Tensor Core：矩阵乘法专用，16×16×16 规格
└─ RT Core：光线追踪（AI 中不常用）
```

**2.2.2 其他硬件概览**
- AMD MI300X：Wavefront（64 线程）vs Warp（32 线程），Wave64 vs Wave32
- 昇腾 910B：AI Core + L1 Buffer（片上），UB（统一缓冲）
- MUSA（摩尔线程）：架构类似 NVIDIA，但细节有差异

**2.2.3 为什么需要"分块（Tile）"**
- 访存带宽 vs 计算速度的量化对比
  - 例：矩阵乘法 C = A × B（M×K × K×N），不分块的访存成本 = O(n³)
  - 分块后，shared memory 中的数据重用，访存成本可降到 O(n² × sqrt(n))
- 分块与内存层次的对齐
  - 寄存器块：线程内部的数据（最快）
  - 共享内存块：block 内线程共享的数据
  - 全局内存块：不同 block 的数据

**2.2.4 并行执行模型**
- 指令级并行（ILP）：单线程内多条独立指令可并行
- 线程级并行（TLP）：多线程并行执行
- 流水线并行：计算与内存访问可重叠

#### 2.3 关键概念速查表
```
概念              定义                              对性能的影响
---------------------------------------------------------------
Warp             GPU 执行的最小单位                 warp 内的线程必须同步
Block            线程的组织单位                     共享 shared memory
Occupancy        每个 SM 上活跃 block 数 / 最大     低 occupancy 导致内存延迟隐藏差
Shared Memory    块内线程可见的片上内存            容量限制了 block 大小
Bank Conflict    多个线程访问同一 bank              会导致访存串行化
Coalesced Access 多个线程访问连续内存              充分利用内存带宽
```

- C++ 代码展示内存延迟对性能的影响：同一个计算，改变数据访问模式
- Python 代码展示 GPU 架构信息查询

### 第 4 章：分块、内存层次与数据复用

#### 4.1 章节定位
建立 TileLang 原语与硬件的一一映射。这是性能优化的基础。

#### 4.2 核心内容

**4.2.1 全局内存与数据加载**
- `T.alloc` 分配全局张量
- 合并访问（Coalesced Access）的重要性
  - 对齐原则：访问应该按 warp 的自然序列进行
  - 示例：连续访问 vs 步进访问 vs 随机访问的性能差异

**4.2.2 共享内存：显式控制片上缓存**
- `T.alloc_shared` 分配共享内存
- 数据复用模式（以矩阵乘法为例）
  - A 矩阵的一个 tile 被 block 内所有线程读取多次
  - 相比全局内存直接访问，shared memory 的好处：延迟低、带宽高
- Bank Conflict 与布局优化
  - 共享内存分成多个 bank，避免冲突的访问模式
  - `T.annotate_layout` 的使用

**4.2.3 寄存器与碎片化存储**
- `T.alloc_fragment` 分配寄存器级矩阵碎片
- 碎片大小与计算单元的匹配
  - Tensor Core：m16n16k16 / m32n32k16 等规格
  - 实际编程中常用的规格及其性能特点

**4.2.4 硬件映射速查表**

| TileLang 原语 | 硬件对应 | 用途 | 典型大小 |
|--------------|--------|------|---------|
| T.alloc | Global Memory | 输入/输出 | MB-GB |
| T.alloc_shared | Shared Memory | 线程块内数据共享 | 32-96 KB |
| T.alloc_fragment | Register | 矩阵片段 | 16×16-32×32 |
| T.gemm | Tensor Core | 矩阵乘法指令 | m16n16k16 |

**4.2.5 代码示例框架**

### 第 5 章：调度优化与性能分析

#### 5.1 章节定位
从被动"实现可工作的代码"转向主动"把代码调到高性能"。

#### 5.2 核心内容

**5.2.1 并行策略：T.Parallel 的层级选择**
- Block 级并行：多个 block 独立执行
- Warp 级并行（Thread 组级）
- Thread 级并行
- 不同层级对性能的影响（occupancy、同步成本）

**5.2.2 流水线：T.Pipelined 与双缓冲**
- 手动流水线的概念
  - Stage 1：加载数据到 shared memory（异步）
  - Stage 2：在 shared memory 上进行计算
  - 这两个 stage 可以交错执行，计算与加载并行
- 双缓冲实现
  - 用两块 shared memory 交替，一块计算一块加载
  - 隐藏内存延迟的好办法

**5.2.3 内存布局优化**
- `T.annotate_layout`：指定张量的内存排布
  - 行主序 vs 列主序对访问模式的影响
- Swizzle 与 `T.use_swizzle`：缓解 Bank Conflict

**5.2.4 搜索空间与自动调优**
- 超参数：Block Size、Thread Per Block、Tile Size 等
- 搜索空间大小：(64/128/256/512/1024)^3 = 可能有万级别组合
- 调优方法：
  - Roofline 模型：判断是计算受限还是内存受限
  - 网格搜索：枚举配置，实测最优
  - 贝叶斯优化：用代理模型加速搜索

**5.2.5 性能对比与分析**
- 基准测试工具的使用
- FLOPS / Bandwidth 的计算
- 与 cuBLAS 的对比
- 不同 GPU 上的性能差异

#### 5.3 代码示例框架

**9.2.4 选型指南**
