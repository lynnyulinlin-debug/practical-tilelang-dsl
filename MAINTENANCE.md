# TileLang 教程维护文档

## 1. 目的

这份文档用于项目的持续维护和交接配合，目标是让后续接手的人快速知道：

- 项目现在处于什么状态
- 哪些文档是事实来源
- 章节怎么写、怎么拆、怎么验
- 代码和文档怎么同步
- 交接时应该检查什么

它不是项目介绍，也不是进度汇总。  
它只负责把“怎么维护这个项目”说清楚。

## 2. 当前维护原则

### 2.1 文档优先

先补文档骨架，再补代码。  
当前项目的推进顺序是：

1. 先建立章节目录和 `intro.md`
2. 再建立每章 5 个左右的内容块
3. 最后补代码示例和 benchmark

### 2.2 章节先结构化，再填内容

每章建议采用以下结构：

- `intro.md`
- `01-*.md`
- `02-*.md`
- `03-*.md`
- `04-*.md`
- `05-*.md`
- `summary.md`

如果章节后续需要继续扩展，可以增加更多内容块，但不要打乱顺序。

### 2.3 代码以 `.py` 为主

主示例使用 `.py` 文件作为权威版本。  
notebook 可以作为教学层和交互层，但不替代可测试的 `.py` 版本。

建议做法：

- `.py`：权威实现、可复用、可测试
- `.ipynb`：教学展示、交互运行、逐格解释
- 两者内容一致时，以 `.py` 为准，`.ipynb` 负责可视化和讲解节奏

### 2.4 图和公式先占位

文档里允许先放图和公式占位，但必须写清楚：

- 这张图要解释什么问题
- 这个公式要支持什么结论
- 后续要验证什么现象

不要只留“这里放图”这种空占位。

### 2.5 文档职责边界

这几个文档各自只承担一类任务：

- `MAINTENANCE.md`：怎么维护、怎么写、怎么验
- `HANDOFF.md`：当前事实、当前状态、当前决策
- `PROGRESS.md`：任务颗粒、待办、完成度
- `README.md` / `INDEX.md`：入口、导航、阅读路径

如果一段内容在多个文档里都适用，优先放到职责最贴近的那份文档里，不要四份都重复写。

## 4. 章节维护规范

### 4.1 命名规范

章节标题建议统一为：

- 中文：`第 N 章：主题名`
- 英文：`Title Case`，尽量自然，不要逐词直译

章节目录建议统一为：

- `chapters/01-name/`
- `chapters/02-name/`
- `chapters/03-name/`

### 4.2 每章标准文件

建议每章都包含：

- `intro.md`
- `01-*.md` 到 `05-*.md`
- `summary.md`

### 4.3 每章标准内容

每个内容块建议只负责一件事：

- 一个问题
- 一个核心概念
- 一个结论

不要在同一小节里塞太多硬件、编译器、代码和性能分析。

### 4.4 章节编写顺序

建议顺序是：

1. 先写 `intro.md`
2. 再写内容块的提纲
3. 再补占位公式和图片说明
4. 最后补代码和 benchmark

## 5. 代码维护规范

### 5.1 示例优先级

优先级从高到低：

1. 能验证主线结论的示例
2. 能支撑章节说明的示例
3. 能做 benchmark 对比的示例
4. 纯展示型示例

### 5.2 代码形式

- 主示例用 `.py`
- 辅助演示可用 notebook
- 代码示例必须能被独立运行或至少明确运行前提

Notebook 入口如果需要导入仓库内的本地模块，必须自行定位仓库根目录并把它加入 `sys.path`，不要依赖用户手工设置 `PYTHONPATH` 才能运行。

建议做法：

- 用 `Path.cwd()` 和父目录回溯定位包含 `HANDOFF.md` 的仓库根目录
- 将定位到的根目录插入 `sys.path`
- 保持 notebook 在 `examples/` 目录下直接打开时也能运行

### 5.3 验证要求

新增或修改示例后，至少确认：

- 语法可读
- 运行入口明确
- 输出结果可验证
- 错误处理有说明
- notebook 入口对本地模块导入不依赖人工环境变量

## 6. 图和公式维护规范

### 6.1 图的要求

每张图必须回答一个问题，例如：

- 为什么需要分块
- 为什么 shared memory 能提高性能
- 为什么 Tensor Core 会改变优化路径

### 6.2 公式的要求

每个公式必须服务一个结论，例如：

- 算术强度如何计算
- 性能上限如何估计
- 为什么某个算子是 memory-bound 或 compute-bound

### 6.3 占位写法

建议统一成：

```md
**图：XXX**

> 这里要说明 XXX，用来支持 XXX 结论。
```

```md
**公式：XXX**

> 这里要展示 XXX 的计算方式，用来解释 XXX 现象。
```

## 7. 维护流程

### 7.1 改章节

1. 先改章节目录骨架
2. 再补 `intro.md`
3. 再补内容块
4. 最后同步 `README.md`、`MAINTENANCE.md`、`QUICKREF.md`

### 7.2 改示例

1. 先确认该示例属于哪一章
2. 再确认它要支撑哪个结论
3. 修改代码后补说明
4. 最后更新交叉引用

### 7.3 改整体结构

如果章节顺序、命名或平台范围变化：

1. 先改维护规则
2. 再改 `README.md`
3. 再改 `QUICKREF.md`
4. 最后检查各章节正文是否还一致

## 8. 交接时检查清单

交接时至少确认以下事项：

- 当前章节结构是否仍是最新
- 哪些章节已拆成目录，哪些还在单文件状态
- 哪些示例是权威版本
- 哪些图和公式还只是占位
- 哪些平台名是正式口径，哪些只是暂用名
- 哪些章节是风险章节，是否需要后移
- 当前最阻塞的工作是什么

## 9. 常见风险

### 9.1 文档和目录不同步

如果标题已经改了，但目录还没改，会导致读者和维护者看到不同版本的项目结构。

### 9.2 平台名过早写死

如果章节标题写死具体平台，后面平台扩展会很麻烦。

### 9.3 先写代码，后补设计

如果不先把章节骨架写清楚，后面代码会把文档结构带偏。

### 9.4 图和公式没有指向结论

只放图不解释它要证明什么，会让内容变成装饰，不是教学。

## 10. 与交接文档的配合方式

- `README.md` 负责当前状态事实
- `MAINTENANCE.md` 负责长期维护方法
- `QUICKREF.md` 负责快速操作口径

交接时主要查看这三个文档：

1. `README.md` 看当前状态
2. `MAINTENANCE.md` 看维护方法
3. `QUICKREF.md` 看实际怎么跑

## 11. 当前建议的下一步

1. 先补齐第 6-10 章的正文收尾
2. 再继续补充示例验证和性能数据
3. 最后收紧 `README.md` 和 `QUICKREF.md` 的口径

## 12. 文档与代码增强顺序

如果继续做增强，建议按下面顺序推进：

1. 先补 benchmark 记录模板和记录方式
2. 再把章节里的图和公式占位落成正式内容
3. 再补教学 notebook 的交互层
4. 再补 `.py` 里的可测试实现或辅助模板
5. 最后统一根文档和章节正文的表述

### 12.1 代码增强的底座

- `examples/*.py` 里保留可运行、可复用的最小实现
- benchmark 记录模板应单独抽成一个小文件，避免各章重复定义
- 教学 notebook 负责解释流程，脚本负责验证逻辑

### 12.2 文档增强的底座

- 先把“这里应该放的图 / 公式 / 表”补成正式内容
- 再把每章总结页收成最终版
- 最后收紧 README 和 QUICKREF 的状态描述

### 12.3 最值得先做的三个增强

1. 第 6 章和第 7 章的 benchmark 实测表
2. 第 8 章和第 10 章的迁移结果图
3. 统一的 benchmark 记录代码模板
3. 固定平台命名口径
4. 继续补章节占位图、公式和示例
5. 等目录稳定后再补代码示例
