# 4.1 全局内存与数据加载

## 要说明的问题

全局内存是最基础的数据来源，但也是性能最容易出问题的地方。

## 要点

- 全局内存的容量和延迟特性
- `T.alloc` 对应的抽象
- 合并访问为什么重要
- 对齐为什么重要

> 这里要说明为什么 warp 的访问模式会直接影响带宽利用率。

## 这里应该解释的现象

- 为什么同样的算子，不同索引写法性能差很多
- 为什么随机访问会浪费事务
- 为什么“能读到”不等于“读得快”

## 这一节可以进一步追问的问题

- 访问模式和 warp 组织是否一致
- 线程间地址是否连续
- stride 是否导致事务数变多

## 这一节应该留下的判断标准

- 如果线程按连续地址访问，通常更容易合并
- 如果访问跨 stride，事务数通常会上升
- 如果访问模式和 warp 组织方式不一致，带宽利用率会掉得很快

这三条是后面判断 layout 和 tile 是否合理的基础。

## 常见误区

- 误区 1：只要数据在显存里就不算瓶颈
- 误区 2：访问模式不会影响吞吐，只影响代码可读性
- 误区 3：随机访问慢只是实现不够巧

实际上，global memory 的访问模式往往就是性能上限的一部分。
