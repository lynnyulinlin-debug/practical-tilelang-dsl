# 4.1 全局内存与数据加载

很多 kernel 第一次跑起来时，问题不是算得不对，而是数据从哪里来、怎么进来的方式不对。  
在 GPU 上，全局内存往往是最先被碰到的层，但也是最容易把性能拖下来的层。写 kernel 时最容易忽略的一点，就是“能把数据读出来”并不等于“以合适的方式读出来”。

这一节要做的，就是把“读数据”这件看起来平常的事重新讲清楚：线程怎么组织访问、地址怎么对齐、事务怎么合并，这些细节为什么会直接决定带宽利用率。  
如果这层逻辑没讲透，后面的 shared memory、tile 和布局优化就会像是单独的技巧，而不是同一条链路上的不同环节。

## 全局内存为什么先看

全局内存是最基础的数据来源，但也是性能最容易出问题的地方。

## 全局内存的访问判断

- 全局内存的容量和延迟特性
- `T.alloc` 对应的抽象
- 合并访问为什么重要
- 对齐为什么重要

这里的关键不是记住几个名词，而是建立一个最基本的直觉：全局内存的访问代价，通常不是由“读了多少字节”单独决定的，而是由“这一批线程如何组织访问”决定的。  
如果线程访问的是连续地址，硬件更容易把多个请求合并成更少的事务；如果访问是跨 stride 的，事务数会变多，带宽利用率会变差。

## 访问方式为什么会变慢

- 为什么同样的算子，不同索引写法性能差很多
- 为什么随机访问会浪费事务
- 为什么“能读到”不等于“读得快”

这也是为什么全局内存章节要先讲访问模式，而不是先讲复杂优化。  
只要访问模式和 warp 的组织方式不一致，后面无论是 tile、shared memory 还是布局调整，很多收益都会先被这一层的损耗吃掉。

所以这一节的核心不是“显存很慢”这种常识，而是“为什么慢、慢在什么地方、为什么可以通过访问组织把损耗压下去”。  
这个判断一旦建立，后面看任何优化都会先去问：数据是怎么走的。

## 全局内存的判断标准

- 如果线程按连续地址访问，通常更容易合并
- 如果访问跨 stride，事务数通常会上升
- 如果访问模式和 warp 组织方式不一致，带宽利用率会掉得很快

这三条是后面判断 layout 和 tile 是否合理的基础。

如果还要继续追问，先看访问模式和 warp 组织是否一致，再看线程间地址是否连续，最后看 stride 是否把事务数抬高。

这一节不是在讲“显存是什么”，而是在讲“为什么同样的数据，换一种拿法，性能就会变”。  
这也是后面所有片上优化的出发点。

一旦把这条因果链记住了，后面再看任何数据访问问题，就会先问线程如何组织、地址是否连续、事务是否能合并，而不是只看代码写法。

## 全局内存的常见误区

- 误区 1：只要数据在显存里就不算瓶颈
- 误区 2：访问模式不会影响吞吐，只影响代码可读性
- 误区 3：随机访问慢只是实现不够巧

实际上，global memory 的访问模式往往就是性能上限的一部分。

所以这一节真正留下的不是“显存慢”这种结论，而是“第一段访问怎么组织，后面的所有层次都要跟着受影响”。  
这也是为什么第 4 章要从 global memory 开始，而不是从 shared memory 直接跳进去。
