# 4.3 寄存器与 Fragment

shared memory 解决的是 block 内复用，但数据真正进入计算之前，还会落到更近的层次。  
这一节要讲的就是这最后一段：当数据马上就要参与计算时，它不应该再待在 shared memory 里，而应该进入寄存器或 fragment 这种更贴近计算单元的短生命周期区域。

这一步之所以重要，是因为它把“缓存思维”切换成了“计算前最后一段停靠”的思维。
读者一旦接受这个转向，就能理解为什么同样的 tile，在不同硬件上会有不同的寄存器压力和并发结果。

这一节的关键，是把“更近”解释成一种有边界的选择，而不是简单的性能口号。  
数据越靠近计算单元，延迟通常越低，但容量越紧、生命周期越短、对 tile 形状越敏感，所以这里讲的是“就近计算”，不是“越低层越好”。

## 为什么这里要更靠近计算

当数据进一步进入计算核心时，shared memory 还不够，最终需要落到寄存器或 fragment 级别。

## 先看什么

- `T.alloc_fragment` 的角色
- fragment 为什么更接近计算单元
- Tensor Core 相关的块形状为何重要
- 寄存器压力为什么会影响 occupancy

这一层的逻辑和 shared memory 不一样。shared memory 的关键词是“复用”，寄存器和 fragment 的关键词是“就近计算”。  
它们更靠近算术单元，所以延迟更低，但代价是容量更紧、生命周期更短、对 tile 形状更敏感。

如果说 shared memory 解决的是“这一批数据能不能多用几次”，那 fragment 解决的是“这一批数据什么时候该正式进入计算”。  
这一层开始，数据已经不再是缓存意义上的资产，而是即将被算术单元消耗的短寿命中间态。

## 为什么更近不一定更轻松

- 为什么过大的 tile 会增加寄存器压力
- 为什么 fragment 的形状要和硬件计算单元匹配

这里最容易出错的地方，是把“更靠近计算”简单理解成“更快”。实际上，靠近计算单元的前提是资源还能撑住。  
tile 一旦过大，寄存器压力会上升，occupancy 会下降，调度灵活性也会变差。于是原本想省掉的延迟，可能会被资源争用重新找回来。

## 这一节最后要判断什么

- 如果数据马上要参与计算，fragment / register 比 shared memory 更合适
- 如果数据还要被多个线程复用，先停在 shared memory 可能更划算
- 如果寄存器压力过高，occupancy 和调度灵活性都会受影响

## 最容易犯的错

- 误区 1：fragment 只是更小的 shared memory
- 误区 2：寄存器越多越好
- 误区 3：只要靠近计算单元就一定更快

fragment 和 register 的价值在于“更接近计算”，但它们的容量和持续时间都更受限制。

所以这一节要传达的不是“越低层越好”，而是“数据应该在最合适的层停留到最后一刻”。  
shared memory、fragment、register 的区别，核心就在这个停留边界上。

后面的 GEMM 案例之所以要继续落到 fragment，就是因为真正的矩阵计算并不只关心“数据有没有搬进来”，还关心“进入计算核心之前是否已经对齐到合适的形状”。

当读者理解这一点后，就会知道 tile 的设计不是把数据一层层往下压，而是在每一层都做一次“停留时间”的判断。  
这也是 fragment 章节真正要建立的硬件直觉。

换句话说，这一节是在把“离计算更近”翻译成“更快但更贵”。  
它让读者明白，片上层次不是越深越好，而是要看数据在那一层停多久、能不能撑住并发。
