# 第 4 章导言：分块、内存层次与数据复用

## 章节定位

这一章是从“会写 kernel”走向“写得快”的第一道门槛。  
核心任务是把 TileLang 的内存原语和硬件层次对应起来，并解释为什么 tile 是性能设计的基础单位。  
如果说前面第 2 章是在讲 GPU 为什么快慢差很多，那么这一章就是把“为什么差很多”翻译成 TileLang 里的可操作选择。

这一章最重要的地方，不在于让读者记住几个内存名词，而在于让读者开始用“数据应该停在哪一层”来思考问题。  
一旦这个判断建立起来，后面的 shared memory、fragment 和 tile 形状就不再是零散概念，而会变成同一条数据流上的不同停靠点。

这也意味着，第 4 章不是在补一套新术语，而是在给后面的调优章提供“数据怎么流”的默认视角。只要这个视角稳定了，shared memory、fragment 和 GEMM 分块才不会被读成三个孤立技巧。

可以把这一章理解成一次很现实的排查：你先看到 kernel 能跑，但数据一进一出就开始拖慢；接着你会发现，问题不只是“读得慢”，而是数据应该停在哪一层、被谁复用、什么时候进入计算核心。  
这也是为什么第 4 章不是在讲抽象术语，而是在讲一条从 global memory 走到 fragment 的完整路径。

对 TileLang 来说，这一章还是第 1 章那座桥开始真正接住原语映射的地方：`T.alloc`、`T.alloc_shared`、`T.alloc_fragment` 和 `tile` 不是孤立名字，而是会被编译链路继续解释成具体的内存层次、复用范围和计算边界。这个过程也可以理解成一次很典型的 lowering：先把上层的 tile、布局和复用意图保留下来，再逐层压到更具体的层次和边界上。换句话说，TileLang 在这里不是只提供“怎么写”的表面接口，而是把“怎么落到硬件上”一起写进了语言结构里。

## 本章目标

- 建立 `global / shared / fragment` 的对应关系
- 解释 coalescing、alignment、bank conflict
- 说明数据复用如何改变算术强度
- 为后续调度优化和实战案例打底
- 让读者形成一个新的判断单位：tile 不是“切一刀”，而是“决定数据怎么流、在哪一层停留、被谁复用”的优化边界

这一组目标其实是在递进地回答同一个问题：数据为什么不能一直留在慢层、为什么有些块值得被复用、为什么越靠近计算单元越要谨慎。  
只要读者把这几个问题串起来，第 4 章就不再是原语说明，而会变成理解后续调优的起点。

## 本章判断标准

读完这一章，读者应该能判断：

- 哪些数据适合尽早放进更快的层次
- shared memory 什么时候是收益，什么时候是负担
- fragment / register 为什么更接近计算单元
- tile 的大小和形状为什么会影响性能和资源占用

如果读者读完还能说出“为什么同样是搬数据，落在不同层次会改变后面的计算成本”，那说明这一章真正建立起了层次感。  
这比记住术语更重要，因为后面的章节会反复用到这条判断。

## 内容块安排

### 4.1 全局内存与数据加载
- `T.alloc`
- 合并访问
- 对齐
- 从一次最普通的数据搬运开始，解释为什么访问方式会直接影响带宽利用率

### 4.2 共享内存
- `T.alloc_shared`
- block 内数据复用
- bank conflict 的首次定义
- 说明 shared memory 为什么不是“更快的内存”这么简单，而是“把复用留在片上”的机制

### 4.3 寄存器与 fragment
- `T.alloc_fragment`
- 更贴近计算单元的存储层
- 解释为什么越接近计算单元，越需要关注资源占用和并行度折损

### 4.4 原语与硬件映射
- TileLang 原语到硬件层次对照表

### 4.5 GEMM 分块案例
- 朴素版与分块版对比
- 数据复用如何带来收益
- 用 GEMM 把“tile 改变数据流”这件事讲完整

如果前面的几节像是在分别认识不同层次的房间，那 GEMM 就是把这些房间连成一条真正可走的路线。  
只有在一个完整算子里同时看到访存、复用和计算匹配，tile 的意义才会从“切块”变成“重排数据流”。

这一章真正要传达的，不是“内存层次很多”，而是“同一块数据在不同层次停留，后面的计算路径会完全不同”。

对 TileLang 来说，这一章最重要的意义是把原语和硬件层次一一对上：`T.alloc` 先对应数据从哪里来，`T.alloc_shared` 说明哪些数据适合留在片上复用，`T.alloc_fragment` 说明哪些数据已经要靠近计算单元，`tile` 则把这几层的选择统一到同一个切分边界里。也就是说，这一章不是在补内存术语，而是在把 TileLang 的表达方式和硬件结构钉在一起。

也正因为如此，这一章不适合写成“原语清单”。  
最好的写法，是让读者沿着一个完整算子看到：数据怎么进、怎么停、怎么被复用、怎么靠近计算核心，最后为什么性能会变。

### 4.6 本章总结
- 用一句话把 tile 和内存层次的关系说清楚

## 本章最小深度要求

这一章至少要让读者说清楚：

1. 为什么 tile 和内存层次必须一起看
2. 为什么 shared memory 不是所有算子的默认答案
3. 为什么 fragment / register 更接近计算单元
4. 为什么 GEMM 是最好的解释案例

如果只能把 tile 理解成“切块”，这章还没有真正学会。
