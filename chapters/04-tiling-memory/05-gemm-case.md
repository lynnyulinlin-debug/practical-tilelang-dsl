# 4.5 GEMM 分块案例

## 要说明的问题

GEMM 是最适合解释 tile、共享内存和 fragment 的案例，因为它同时有高计算密度和明显的数据复用。

## 要点

- 朴素 GEMM 的问题
- 分块后数据复用如何提高
- shared memory 如何减少全局访问
- 为什么这是后续优化章节的基准案例

## 这一节应该留下的判断标准

- 如果 tile 太小，复用收益不够
- 如果 tile 太大，shared memory 和寄存器压力会变高
- 如果 tile 的形状和访问方向不一致，合并访问也会变差

## 这一节可以进一步追问的问题

- tile 的收益是否覆盖额外的同步和搬运成本
- tile 的形状是否和矩阵乘法的数据流一致
- tile 是否让资源压力超过了收益

## 常见误区

- 误区 1：tile 越大，性能一定越高
- 误区 2：只要分块就能解决 GEMM 性能问题
- 误区 3：GEMM 的最优 tile 在不同硬件上可以直接照搬

所以 GEMM 的分块不是随便切，而是在“复用、并发、资源占用”之间找平衡。

GEMM 之所以是最好的解释案例，是因为它能同时暴露出全局访存、片上复用和计算单元匹配这三类问题。
