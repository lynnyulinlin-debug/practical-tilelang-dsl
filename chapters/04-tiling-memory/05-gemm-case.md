# 4.5 GEMM 分块案例

如果前面几节是在讲“单个层次各自做什么”，那这一节就是把它们全部连起来。  
GEMM 是最适合做这个例子的，因为它同时包含了大规模数据搬运、显式复用和高密度计算，几乎把 tile 设计里最关键的矛盾都摆在了台面上。

这一节的价值，在于它把前面分散的概念汇聚成一个真实算子。  
读者只有在 GEMM 里同时看到全局访存、片上复用和计算单元匹配，才会真正理解 tile 不是“切小块”，而是把数据流和计算流重新对齐。

## 为什么 GEMM 是最好的例子

GEMM 是最适合解释 tile、共享内存和 fragment 的案例，因为它同时有高计算密度和明显的数据复用。

## 先看什么

- 朴素 GEMM 的问题
- 分块后数据复用如何提高
- shared memory 如何减少全局访问
- 为什么这是后续优化章节的基准案例

朴素 GEMM 的问题很典型：同一批矩阵元素会被重复读取很多次，但这些重复读取并没有转化成更多有效计算。  
分块之后，A 和 B 的 tile 被搬到片上，一次加载可以服务多个输出元素的计算，于是复用开始出现，算术强度也开始上升。

这里最容易建立直觉的地方，就是看到“同一份数据不再只服务一次”。  
GEMM 的分块不是先把问题切小，而是先把数据流改成更适合重复使用的样子，所以收益通常不是来自少算，而是来自少搬、少读、少回写。

这也是这一节最该强调的点：
GEMM 分块不是在证明“分块本身有多神奇”，而是在证明“同一份数据如果能在片上多服务几次，性能曲线就会变”。

## 这一节最后要判断什么

- 如果 tile 太小，复用收益不够
- 如果 tile 太大，shared memory 和寄存器压力会变高
- 如果 tile 的形状和访问方向不一致，合并访问也会变差

所以 GEMM 的 tile 不是“随便切一刀”，而是在复用、并发和资源占用之间找平衡。  
tile 太小，复用不够；tile 太大，shared memory 和寄存器压力又会上来；tile 方向不对，访问模式还会拖慢带宽利用率。

这也是为什么 GEMM 经常被拿来做结构性优化的示范案例。  
它的瓶颈足够典型，改动后的收益也足够容易被观察。

## 还能追问什么

- tile 的收益是否覆盖额外的同步和搬运成本
- tile 的形状是否和矩阵乘法的数据流一致
- tile 是否让资源压力超过了收益

## 最容易犯的错

- 误区 1：tile 越大，性能一定越高
- 误区 2：只要分块就能解决 GEMM 性能问题
- 误区 3：GEMM 的最优 tile 在不同硬件上可以直接照搬

所以 GEMM 的分块不是随便切，而是在“复用、并发、资源占用”之间找平衡。  
它之所以适合放在第 4 章收尾，是因为它能同时暴露全局访存、片上复用和计算单元匹配这三类问题。
