# 4.2 共享内存：显式控制片上缓存

如果说全局内存解决的是“把数据搬进来”，那 shared memory 解决的就是“搬进来之后，怎么不让它白白出去”。  
这一节最重要的不是把 shared memory 说成一种更快的存储，而是把它理解成 block 级的数据复用机制：同一块数据一旦会被多个线程重复使用，就值得考虑把它留在片上。

这也是第 4 章真正进入“结构性优化”的地方。  
前一节讲的是数据如何到达，这一节开始回答数据到达之后应该停在哪里、被谁重复使用、什么时候同步才算合理。

## shared memory 为什么开始有复用

shared memory 的价值在于 block 内的数据复用，而不是简单“更快的内存”。

## shared memory 的复用判断

- `T.alloc_shared` 的作用
- 为什么矩阵乘法特别适合用 shared memory
- 同一块数据如何被多个线程重复使用
- 同步为什么必不可少

这里要补一层更具体的因果链：数据从 global memory 进入 shared memory，不是为了“换一个更快的地址”，而是为了让一次加载服务多次计算。  
这在 GEMM 里尤其明显，因为 A 和 B 的 tile 往往会被一个 block 内的多个线程反复消费。如果不做这一步，数据会一次次从更慢的层次重新取来，算术强度很难上去。

## shared memory 的复用收益和代价

可以把 shared memory 的收益理解成：同一块 tile 在 block 内被重复使用的次数越多，搬运一次带来的节省越明显。

- 为什么同一块数据会被 block 内多个线程重复使用
- 为什么 shared memory 能改善复用，但也会增加同步和占用成本
- 为什么不是所有算子都值得搬进 shared memory

这里的“同步”主要指 block 内的协作同步：数据搬到片上之后，哪些线程先等一等、哪些线程可以继续消费，决定了 shared memory 的收益能不能真正兑现。

这里的权衡很直接：shared memory 带来的是复用收益，但也引入了同步点、容量占用和 occupancy 下降的风险。  
所以它从来不是默认答案，而是一个“当复用足够强时才值得付出代价”的选择。

这也是为什么 shared memory 章节必须和 GEMM 绑定着看：
只有在明显可复用的场景里，才能真正看见“搬一次、用多次”的收益。

## shared memory 的判断标准

- 如果同一块数据会被 block 内多个线程重复用，shared memory 值得考虑
- 如果数据只读一次就走，不一定值得搬进 shared memory
- 如果 shared memory 让 occupancy 掉得太多，收益可能被抵消

这也是为什么“shared memory 越多越好”是错误的直觉。

把 shared memory 讲清楚以后，就会发现它和全局内存不是替代关系，而是分工关系：global 负责把数据送进来，shared 负责把这一批数据尽量多用几次。

这也是为什么 shared memory 适合放在 GEMM 这种复用密集的案例里。  
如果一个算子没有明显的重复消费模式，搬进 shared memory 反而可能只是在制造同步和占用成本。

## shared memory 的常见误区

- 误区 1：shared memory 只要用了就会更快
- 误区 2：shared memory 可以替代所有 global memory 访问
- 误区 3：只要把数据搬进片上层，性能问题就解决了

shared memory 的作用是放大复用，不是替代判断。

也正因为如此，这一节的叙事位置很重要：它承接了 global memory 的读入成本，再把“如何不让这笔成本白花”讲清楚。  
如果没有这一步，tile 的价值就只剩“搬运一次”，而不是“搬运一次、服务多次”。

这也为后面的 fragment 章节铺好了路：复用层解决的是“多用几次”，更靠近计算单元的层解决的是“最后怎么用完”。
