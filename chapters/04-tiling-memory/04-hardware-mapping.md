# 4.4 TileLang 原语与硬件映射

## 要说明的问题

这一节要把前面的抽象落到显式映射上，形成一张“语法到硬件”的对照表。

## 要点

- `T.alloc`
- `T.alloc_shared`
- `T.alloc_fragment`
- `T.gemm`
- `T.Parallel`

## 这里应该放的表

**表：TileLang 原语与硬件层次对应表**

> 这里要明确每个原语对应该用哪一层内存或计算单元。

## 这里应该解释的现象

- 为什么抽象越明确，编译器越容易优化
- 为什么硬件映射是性能设计的一部分

## 这一节可以进一步追问的问题

- 这个原语描述的是结果落点、片上复用，还是计算中间态
- 它的生命周期长还是短
- 它更接近全局内存、shared memory 还是计算单元

## 这一节应该留下的判断标准

- `T.alloc` 更偏全局存储和结果落点
- `T.alloc_shared` 更偏 block 内复用
- `T.alloc_fragment` 更偏靠近计算单元的短寿命数据

把这三者分开后，后面看任何 kernel 的存储结构都会更快。

## 常见误区

- 误区 1：原语越多越复杂，越不利于优化
- 误区 2：只要语法有区分，性能上就一定有差异
- 误区 3：所有存储都应该统一成同一种抽象

明确映射的目的，是让“该放哪一层”变成可讨论的问题，而不是靠经验猜。
