# 4.4 TileLang 原语与硬件映射

前面几节把内存和计算层次拆开看了，这一节要做的，是把这些层次重新收拢到 TileLang 原语上。  
这样读者才不会只记住“有 global、shared、fragment”，而是能在看到原语时，立即判断它把数据放到了哪一层、停留了多久、服务了哪种复用或计算。

对 TileLang 来说，这一步尤其关键，因为它不是把原语当成“语法糖”，而是把硬件层次显式写进语言接口里。换句话说，TileLang 的特色不只是能写 kernel，而是能在代码里直接表达“这段数据应该停在哪一层、谁来复用、计算从哪一层开始”。这就是为什么这节不能只讲名字，必须讲映射。

这一节其实是在给后面的所有章节定“语言边界”。  
只要原语和硬件层次的对应关系明确了，后面不管是 GEMM、attention 还是跨平台迁移，读者都能从原语本身看出这段代码在硬件上想做什么。

## 为什么要把原语和硬件对齐

这一节要把前面的抽象落到显式映射上，形成一张“语法到硬件”的对照表。

## 这张映射表在看什么

- `T.alloc`
- `T.alloc_shared`
- `T.alloc_fragment`
- `T.gemm`
- `T.Parallel`

这些原语之所以要分开，不是为了让语法更复杂，而是为了让这座桥和开发者都能看见层次。  
一旦层次可见，优化就不再是“猜编译器会不会做对”，而是“明确告诉系统这段数据该去哪里、该怎么复用、该靠近谁”。

这一步相当于把前面几节讲过的层次，重新压回到一套可写、可读、可优化的语言里。  
也就是说，第 4 章前半段是在建立直觉，后半段是在把直觉固化成原语。

如果把这一步讲成一句更实用的话，那就是：
前面你已经知道“数据该在哪一层停”，这一节开始回答“该怎么把这个停靠点写进代码里”。

## 抽象为什么会变成可优化

- 为什么抽象越明确，编译器越容易优化
- 为什么硬件映射是性能设计的一部分

这里要强调一个容易被忽略的点：原语不是附带的记号，而是性能约束的一部分。  
你把 `T.alloc`、`T.alloc_shared`、`T.alloc_fragment` 分开写，其实就是在告诉后端，这段数据的寿命、位置和使用范围不同。

## 还能追问什么

- 这个原语描述的是结果落点、片上复用，还是计算中间态
- 它的生命周期长还是短
- 它更接近全局内存、shared memory 还是计算单元

## 这一节最后要判断什么

- `T.alloc` 更偏全局存储和结果落点
- `T.alloc_shared` 更偏 block 内复用
- `T.alloc_fragment` 更偏靠近计算单元的短寿命数据

把这三者分开后，后面看任何 kernel 的存储结构都会更快。

也就是说，这一节的作用不是给读者再加一套术语，而是把抽象和硬件的关系固定下来。  
从这一刻开始，读者看到的就不只是“代码怎么写”，而是“数据到底在哪一层完成它的任务”。

这也是第 4 章的一个转折点：前面讲的是层次本身，这一节开始讲“怎样把层次写进程序”。  
只有把这个映射固定住，后面的 GEMM 案例才不会只剩经验描述。

这一步很重要，因为后面的优化不再是抽象判断，而是具体实现选择。  
如果这一层没建立起来，所有原语都会退化成语法记忆。

所以这一节最好和原语映射表一起看：
表负责告诉你“每个原语去哪一层”，正文负责告诉你“为什么要这样写”。

## 最容易犯的错

- 误区 1：原语越多越复杂，越不利于优化
- 误区 2：只要语法有区分，性能上就一定有差异
- 误区 3：所有存储都应该统一成同一种抽象

明确映射的目的，是让“该放哪一层”变成可讨论的问题，而不是靠经验猜。
