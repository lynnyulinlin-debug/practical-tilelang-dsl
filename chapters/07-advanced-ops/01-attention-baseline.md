# 7.1 Attention baseline

Attention 的 baseline 必须先讲清楚，不是因为它简单，而是因为它把后面所有优化的成本来源暴露得最完整。  
一旦序列变长，真正开始膨胀的往往不是公式本身，而是 `scores` 和 `probs` 这些中间矩阵的写回、读取和再利用方式。

这一点很容易被低估。  
如果只看公式，attention 像是几步标准矩阵运算；但一旦把它放进真实执行路径，最先变贵的往往不是乘法，而是中间矩阵在内存层次里来回移动的成本。

如果你第一次把 attention 跑到稍长一点的序列上，最先冒出来的往往不是“公式错了”，而是内存开始变得沉。  
公式看起来还是那几步矩阵运算，但程序背后已经悄悄多出了一张 `S x S` 的中间表，这张表会在 softmax 前后各走一遍，把写回和读取的压力一并带出来。  
读者真正需要先看见的，就是这个压力是怎么一点点长出来的。

所以这一页不是为了把 attention 一次讲完，而是为了把问题摊开。  
你先看见成本从哪里冒出来，后面 FlashAttention、融合和调度才不会像是“额外再加一层技巧”，而会像是顺着同一个瓶颈继续往下挖。  
换句话说，这里是在把地基挖开，让后面的优化都有明确的落点。

## 先看成本

标准 attention 的结构很固定：Q / K / V 先进入 score 计算，再经过 softmax，最后做输出加权求和。  
这个结构看上去是几步标准算子串联，但它的关键成本不在“做了几次乘法”，而在“每一步都生成了需要暂存的中间状态”。  
所以 attention 的 baseline 不是为了证明它跑不动，而是为了把中间状态的代价明确写出来。

这也是为什么 baseline 不是“起步版本”，而是“参照版本”。  
它的任务不是最省事，而是最容易把后续优化真正要解决的问题摆到台面上。

最小形状例子也很直接。`Q, K, V` 通常写成 `B x S x D`，`scores = QK^T / sqrt(D)` 的形状会变成 `B x S x S`，`softmax(scores)` 仍然是 `B x S x S`，最后 `out = softmax(scores) V` 又回到 `B x S x D`。  
这个形状链路本身就说明了问题：当 `S` 变大时，`S x S` 的中间矩阵会迅速膨胀，而这部分膨胀直接对应显存流量和中间状态压力。

换成更直观的说法就是：每一行表示“某个 query 对所有 key 的权重”，每一行都要做一次归一化。  
所以 attention 的成本不是一次矩阵乘法，而是一整串中间状态的生成、归一化和回写。  
当这些中间结果必须反复写回慢层内存时，后面的优化空间就已经出现了。

这里真正值得强调的是，attention 的问题不是复杂度本身，而是复杂度被放在哪一层处理。  
如果这些中间状态都留在慢层内存里，它们就会把所有后续阶段都拖慢；如果后面的优化能把它们压回片上，整个数据流就会明显变轻。

换句话说，attention 的难点不是“算不算得动”，而是“这些中间状态是不是被放在了太贵的层次上处理”。  
一旦把这个问题看清楚，后面的 FlashAttention 就不是技巧展示，而是针对具体成本的重排。

这也是 baseline 的意义所在。  
它不是最“原始”的版本，而是最适合暴露问题的版本：哪里生成了中间张量，哪里发生了写回，哪里开始让序列长度变成真正的成本放大器，这些都要先在这里看清楚。  
只有把这些点标出来，后面每一种优化才知道自己到底是在替哪一部分成本买单。

如果读者能在这里说清楚 `scores` 为什么会成为成本中心，那后面看 FlashAttention 时就不会只把它理解成“更快的 attention”，而会理解成“把成本中心换了一种组织方式”。

## 为什么先看 baseline

这一页的价值，不是证明 attention 很差，而是把“为什么需要重构数据流”说清楚。  
标准 attention 不是错的，而是太依赖中间结果写回内存。  
这正是后面 FlashAttention 要解决的核心问题。

所以这一页要让读者先看到成本，再允许优化登场。  
如果不先看到 `scores` 和 `probs` 的代价，FlashAttention 只会像一个聪明但突兀的技巧；看过 baseline 之后，读者才会明白它到底是在替哪一类成本买单。

## 这一节最后要判断什么

读完这一节，读者至少应该能回答三件事：

1. 为什么序列变长后，`scores` 会迅速变贵
2. 为什么 `B x S x S` 中间矩阵会成为成本中心
3. 为什么 baseline 不是陪跑，而是整个章节的参照系

如果当前阶段还看不出瓶颈，就先别急着优化，先把 baseline 的数据流看清楚。  
这一页的任务不是让读者背公式，而是让读者知道 attention 的成本是从哪里长出来的。

更进一步说，这一页真正要留下的判断是：只要 `B x S x S` 这一层存在，序列长度一变长，内存流量和中间状态压力就会先于算式复杂度成为主问题。  
这个判断一旦建立，整个章节后面的优化顺序就都能顺着它往下推。
