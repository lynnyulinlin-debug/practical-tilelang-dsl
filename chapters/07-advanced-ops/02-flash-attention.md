# 7.2 FlashAttention

FlashAttention 不是把 attention 改得“更花”，而是把 attention 的数据流改得更省 IO。  
它真正解决的问题，是让中间状态尽量停留在片上，而不是每一轮都把 score 矩阵写回慢层内存。

所以它的重点不是“算得更聪明”，而是“搬得更少”。  
这一点很关键，因为在长序列场景里，搬运成本往往比算式本身更先成为主瓶颈。

如果你把 attention 的 baseline 和 FlashAttention 并排看，差别并不在“算式变没变”，而在“中间状态到底要不要每一步都落回去”。  
baseline 里，那张 `S x S` 的 score 矩阵会先被算出来，再被 softmax 处理，再被拿去乘 `V`；而 FlashAttention 试图做的，是把这条路径改成一边读块、一边更新、一边把局部统计量留在片上。  
这就是它真正像故事转折的地方：同一个问题，不再用同一种搬运方式去解决。

从教学角度看，这个转折最值得强调的，是它没有改变 attention 的数学目标，只是改变了计算结果被保存和更新的方式。  
也就是说，它不是“换了一个答案”，而是“换了一种更适合硬件的答题顺序”。

## 为什么要先压 IO

FlashAttention 的价值，不在于多做了什么，而在于少做了什么。  
它通过重新组织计算过程，减少 attention 对中间矩阵和显存的依赖。  
只要中间状态可以在线维护，就不必把完整 `scores` 长期落盘，这就是 IO-aware 的真正含义。

这也是为什么 IO-aware 不是口号，而是具体的执行约束：只保留当前块真正需要的统计量，把一次性膨胀的大矩阵拆成能在线维护的小状态。  
一旦做到这一点，序列越长，节省出来的写回和读取就越明显。

这一点和第 4、5 章的方法是连着的。  
先分块，再控制片上数据，再把计算顺序改成更省 IO 的形式。  
也就是说，FlashAttention 不是一个孤立技巧，而是前面分块和复用思路在复杂算子上的自然延伸。

这里真正值得注意的，是 FlashAttention 并没有改变 attention 在数学上做什么，而是改变了它怎么被组织。  
这也是它和普通“加速版 softmax”最大的区别：后者只是在同一条路径上更快一点，前者是在重新安排路径本身。

把它讲成故事的话，baseline 像是先把整块路面铺开，再在路面上慢慢处理每一段；FlashAttention 则更像是边铺边处理，只保留当前真正需要的局部状态。  
它不是少算了一步，而是把那些原本必须完整落盘的中间结果，改成了可以在线更新的片上变量。  
所以读者看到的不是“速度更快的 attention”，而是“执行路径被换了一种写法的 attention”。

换个角度说，FlashAttention 解决的不是“算不算得动”，而是“中间状态要不要一定经过慢层内存”。  
这个差异非常关键，因为它决定了优化能不能随着序列长度继续放大收益。

## 块化之后，成本怎么变

tile 化之后，attention 不再是“先把整张矩阵算完，再统一 softmax”，而是“边读块、边更新、边累积”。  
Q / K / V 按块处理，block-wise softmax 先算局部最大值和局部和，再做稳定归一化。  
这样一来，原本一次性膨胀的中间矩阵，就被改造成了片上的局部状态。

这个流程的关键不是“省掉一条乘法”，而是把原来要落到慢层内存的中间矩阵，改成只在片上维护的局部变量。  
当序列越长时，这种改法的收益会越明显，因为 baseline 里那张 `S x S` 的中间矩阵会变得越来越贵。

所以这里的判断标准也变了：不是看某个 kernel 有没有做更多操作，而是看它有没有把“必须一次性落盘”的东西改成“可以在线滚动更新”的东西。  
这个变化才是 FlashAttention 的真正核心。

从叙事上看，这一步是第 7 章的第一次明显转向。  
baseline 只是把成本摆出来，FlashAttention 则告诉读者：这个成本不是只能接受的，它可以通过重排计算被压缩掉。

## 为什么长序列最能看出差异

FlashAttention 最典型的使用场景，就是长序列。  
序列越长，baseline 里那张 `S x S` 的中间矩阵越昂贵，块化以后省下来的写回和读取也就越多。  
所以这里真正要看的不是“它是不是更快”，而是“它为什么在长序列上更有优势”。

如果块太小，控制开销可能抵消收益；如果块太大，中间状态又会变重。  
这说明 FlashAttention 不是“块越大越好”，而是在数值稳定、IO 压缩和执行开销之间找平衡。

换句话说，FlashAttention 不是把 block 化简单套上去，而是在“更少 IO”和“更多管理开销”之间找一个可复用的平衡点。  
这也是它为什么需要和序列长度一起看，而不能脱离规模单独谈优劣。

这一段也正好说明了为什么 FlashAttention 不能被理解成“单纯加速”。  
它的目标不是把原来的公式做得更快一点，而是把公式的执行路径改成更适合硬件的方式。

从叙事上看，这里最重要的变化是：读者开始看到“一个大问题可以被拆成一段段小更新”。  
之前 baseline 里那张完整的大矩阵像是必须一次性处理完的东西；到了这里，问题被拆成了块内更新、块间累积和最终归一化三个阶段。  
一旦读者接受这种拆法，就会明白 FlashAttention 的收益为什么会随着序列增长而放大，因为每次被省掉的，都是原本最容易膨胀的那部分中间状态。

如果把这一节压成一句话，那就是：baseline 让成本显形，FlashAttention 让成本不再必须经过慢层内存。  
后面的融合和调度，其实都只是沿着这个方向继续收缩。

## 这一节最后要判断什么

读完这一节，读者至少应该能回答三件事：

1. 为什么 FlashAttention 的核心是重排计算，而不是某个更快的 softmax
2. 为什么 block-wise softmax 能成立
3. 为什么序列越长，收益往往越明显

如果读者能把这一页理解成“改变计算组织方式”，就会明白它和前面分块、共享内存的逻辑是连着的，而不是单独一招。  
这一步把第 7 章的故事往前推进了一大截：baseline 暴露了成本，FlashAttention 说明这类成本可以通过重排计算被压缩掉。
