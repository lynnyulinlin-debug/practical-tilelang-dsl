# 7.5 性能分析的条件对齐

前面几节已经把优化手段摆出来了，但复杂算子最容易出的问题，是“看起来快了，实际上收益不稳定”。  
这一节不讲新优化，只把收益、代价和适用范围固定下来。

这一页的任务，是把前面的判断变成能复核的结论。  
如果输入规模、硬件环境和统计方式没有对齐，收缩数据流、收紧边界和重排顺序这些说法就很容易停在“听起来对”，而不是“测出来对”。

前面几页是在回答“怎么改”，这一页是在回答“改完之后怎么证明它真的值得”。  
FlashAttention、融合和调度只有放回同一输入规模、同一硬件环境、同一统计方式里比较，才算真正完成了收口。

复杂 attention 的收益常常不是线性的。  
有些优化在短序列上几乎看不出来，但一旦序列变长，差异就会迅速放大；有些实现单看延迟并不占优，但在显存和吞吐上更平衡。  
性能分析要做的，就是把这种交换关系说清楚。

这一页先做的，不是挑一个“最好看”的数字，而是决定比较维度：长序列还是短序列，峰值显存还是平均显存，延迟还是吞吐。  
维度定错，后面的结论往往只是把局部优势包装成整体优势。

可以先用短序列看实现有没有把成本压下去，再用长序列看这个收益会不会继续放大。  
前者确认“有没有效果”，后者确认“效果是不是来自 IO 压缩和边界收紧”，而不是某次测试里的偶然波动。

## 先对齐 benchmark 条件

复杂算子不能只看“写没写出来”，必须看收益是否真实。  
这一节要把显存占用、吞吐和延迟一起看，因为 attention 这种算子常常不是单点最优，而是存在明显的取舍：有的实现显存更省但延迟更高，有的实现吞吐更好但峰值占用更大。

这里最重要的前提，是 baseline 和优化版必须在同一输入规模、同一硬件、同一统计方式下测量。  
如果比较条件不一样，后面的结论就很容易失真。  
attention 的结果又特别容易受序列长度影响，所以短序列和长序列不能混着看；只看一个规模，容易把局部收益误当成普遍结论。

实践里最稳的写法，是先给一个最小的对照表，再给一个规模变化图。  
表负责回答“条件是不是一致”，图负责回答“收益是不是稳定放大”。  
表里条件不一致，图再漂亮也很难说明问题；图里趋势不清楚，表里信息再全也只是一堆背景参数。

在写作上，建议把 benchmark 拆成两层：

1. 先给一张横向比较表，确认不同实现到底交换了什么
2. 再给一张规模变化图，确认收益是不是随着序列长度稳定放大

如果这两层都没有，只会看到若干孤立数字，很难把它们和前面的 FlashAttention、融合、调度对应起来。  
所以这一页最重要的动作，不是展示结果，而是把结果放回同一把尺子上。

这里也可以顺手给一个很实用的判断：短序列先看实现有没有把成本压下去，长序列再看收益是不是继续放大。  
如果短序列就已经把延迟和显存都收住了，那说明实现至少没有把问题做大；如果长序列还能继续拉开差距，才说明前面的 IO 压缩和边界收紧真的起作用。  
这样读结果时，就不会把一张图里的局部优点误当成整体结论。

也正因为如此，性能分析真正做的不是“再跑一遍 benchmark”，而是把 benchmark 变成一个判断工具。  
比较的不是单次数字，而是这一组数字能不能在相同输入、相同硬件和相同统计方式下站得住。

这一步还意味着，性能分析必须和前面的正文内容连起来看。  
前面说的“压 IO”“收边界”“排顺序”，到这里都要落到数值上；如果数值不能把这些变化证明出来，前面的解释就还不算闭合。  
所以这一页不是附录，而是把前面几页的推理链收口。

## 结果如何写成结论

这一节最后要形成的，不是“这个实现更快”，而是“这个实现在哪些输入规模和测量条件下更值得用”。  
只有把收益来源和边界条件一起写出来，优化结论才算完整。

所以这里真正要做的，是把“感觉快了”变成“数据上真的快了”。  
如果同一套 benchmark 能在不同规模、相同硬件和相同统计方式下复现，那么前面的 FlashAttention、融合和调度才不是单次展示，而是可复用的方法论。

这里还要再往前想一步：如果 benchmark 不能稳定复现，那就说明前面的推理链还没闭合。  
可能是输入规模不对，可能是统计方式不一致，也可能是实现还没有真正落到需要的场景里。  
所以这一页看似是在总结结果，实际上是在倒查前面每一步是否真的站得住。

这也是为什么这一页会同时看显存、吞吐和延迟。  
复杂算子往往不是单点最优，而是有明显取舍：有的实现显存更省但延迟更高，有的实现吞吐更好但峰值占用更大。  
把这几项一起看，才能知道一个实现到底是在交换什么。

如果最终结果只是在某个小规模上更快，但显存和吞吐都没有形成稳定趋势，那它更像一次局部优化展示，而不是可复用结论。  
这一节的标准，恰恰是要把这种“偶然更快”和“稳定更值得用”区分开。

所以最终结果最好写成“在什么规模、什么测量条件下更值得用”，而不是只写“比 baseline 快”。  
这样会逼着把收益、边界和比较条件一起看，也更容易把 benchmark 的数字回填到正文里。  
对这一章来说，能被复测、能被比较、还能说清为什么值得用，才算真正完成了性能分析。  
性能分析要证明的，不是“这个实现有一次很漂亮的峰值”，而是“这个实现的收益是否能在不同规模和相同统计方式下说得通”。

这一页最后要看的，还是同一组复核条件：比较的是显存、吞吐还是延迟；baseline 和优化版是否在同一输入、同一硬件、同一统计方式下测量；长序列和短序列是否应该分开看；以及这个版本到底更适合什么场景。  
如果这些问题说不清楚，性能分析就只是把数字摆出来，还不算真正完成。  
能被复测、能被比较、能说清边界的结果，才配得上放到章节里当结论。  
这也是第 7 章最后还要回到 benchmark 模板的原因。  
复杂算子的价值不在于“做过”，而在于“做完之后，能不能稳定地说清楚它在什么条件下有收益”。  
只要这句话能成立，前面的 FlashAttention、融合和调度才真正闭合。  

把这一页放到整章里看，它的作用就不只是报告结果，而是给整章方法论做最后一次验收。  
前面讲的是“怎么改”，这里讲的是“为什么相信改动有效”；前面讲的是“数据流和调度怎么变”，这里讲的是“这些变化对应的成本和收益是否真的可见”。  
这一层验收完成后，整个章节才不会停在实现层，而会落到可复核、可比较、可复用的结论层。

这一步完成以后，第 7 章前面那些“怎么改”的内容才真正闭合。  
因为复杂算子的价值不是“做了很多优化动作”，而是“这些动作在相同输入规模、相同硬件和相同统计方式下确实带来了稳定收益”。

## 表格要记录什么

**表：Attention benchmark 记录表**

> 这里要统一记录模型规模、序列长度、batch size、显存占用、延迟和吞吐。
> 表格的目的，是让后面任何一次复测都能对齐输入规模、硬件和统计方式，也能看出收益到底来自哪里。

如果要给出最小样例，可以直接引用统一模板，先给出一个 attention 场景的记录字段。

```python
from examples.benchmark_record_template import set_benchmark_record

record = set_benchmark_record(
    scenario="chapter 7 attention benchmark",

    platform="NVIDIA",
    target="cuda",

    baseline="standard attention",
    optimization="FlashAttention + fusion",
)
```

## 图表和代码回填

这张图要承担“单点结果”和“规模趋势”的双重说明，一张图看趋势，一张表看条件，避免只看数值不看上下文。  
真正重要的不是某一次测得多快，而是这套结果能不能被复测、能不能被比较、能不能回填正文。  
如果图表本身只能证明“有变化”，却不能证明“变化来自哪里”，那这页就还没有完成它的任务。

更进一步说，这一页的图表不是为了装饰结论，而是为了让结论可以被重新核对。  
当图里能清楚看到短序列和长序列的差异、不同实现之间的边界、以及显存和吞吐的交换关系时，就能知道这不是一次偶然跑出来的数字，而是一组能回到正文里继续解释的证据。
这也是性能分析为什么必须占据一整节，而不是作为一句“后面 benchmark 证明了效果”草草带过。
