# 7.4 复杂调度与优化

## 要解决什么问题

复杂 attention 之所以难，不在算式本身，而在分块、布局、同步和调度的组合复杂度。

换句话说，难点不是“怎么写一个 attention”，而是“怎么让 attention 在真实硬件上稳定跑快”。

## 这一节要讲什么

### 1. shared memory

- 把块内热点数据留在片上
- 优先缓存会被重复访问的 `Q/K/V` 子块

### 2. pipeline

- 让加载和计算重叠
- 把数据搬运和计算阶段交错起来

### 3. layout

- 减少冲突
- 改善访问模式
- 让 block 的访问方向和硬件更匹配

### 4. 复杂度与收益

- 代码复杂度增加
- 性能和显存占用通常显著改善

### 5. 当前写作口径

- 这节要延续第 5 章的调优方法论
- 不要只列优化项，要说明这些优化项各自解决的瓶颈是什么
- 最后要给读者一个可执行的调优顺序

### 6. 一个可执行的顺序

1. 先找出最热的数据块
2. 再判断这些数据块能不能留在 shared memory
3. 如果搬运和计算都重，继续做 pipeline
4. 如果访问模式仍然不理想，再调 layout
5. 最后回到 benchmark 记录表里复核收益

## 这一节的结论

复杂调度的价值在于把长链路算子中的等待和冗余尽量压缩掉。
它是把高级算子从“能实现”推到“能上线”的关键一步。

## 这一节应该留下的判断标准

- 如果加载和计算重叠明显，pipeline 往往比单纯改布局更有效
- 如果 shared memory 告急，可能应该先缩 tile 再谈加速
- 如果 layout 优化带来收益但不稳定，说明访问模式还没固定下来

## 这一节可以进一步追问的问题

- 瓶颈到底是在加载、计算还是同步
- 先改 shared memory、pipeline 还是 layout 更合理
- 为什么复杂算子更需要固定的调优顺序

## 常见误区

- 误区 1：调度只是在做性能微调
- 误区 2：复杂算子可以靠单一优化解决
- 误区 3：只要加上 pipeline，问题就会自动消失

复杂调度的关键，是把多种优化手段按瓶颈顺序排好，而不是把它们堆在一起。

这节真正要教会的，不是记住三种优化，而是知道它们各自解决哪类瓶颈。

## 讲义提示

这一节最好不要抽象地谈“优化方法论”，而要直接给出一个判断表：

- 哪一步是瓶颈
- 哪一步应该先改
- 改完以后怎么验证收益

这样读者才会把它当成可复用模板，而不是一组术语清单。
