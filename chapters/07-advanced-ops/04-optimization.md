# 7.4 复杂调度与优化

复杂 attention 真正难的地方，不是某一个算子不好写，而是多个优化手段要按正确顺序叠起来。  
这也是为什么这一节不能只列优化项，而要把它们放进同一条瓶颈链路里看。

这一页要解决的，不是“能不能继续优化”，而是“接下来该先动哪一层”。  
只要这一层判断不清，后面的 shared memory、pipeline 和 layout 很容易变成按感觉堆技巧。

这一页的作用，是把“复杂”拆回“顺序”。  
shared memory、pipeline、layout 看上去是三种技巧，实际上对应的是复用、等待和访问模式三个层面的瓶颈。

如果把这一节先压成一句话，就是先找出最热的数据块，再确认它能不能留在片上，接着判断能不能把加载和计算交错起来，最后才修访问布局。  
这样写的好处是，读者不会把三种技巧看成并列菜单，而会把它们看成一条逐层收紧的调优路径。

如果只从术语上看，这三者很容易被理解成并列的优化项；但放到真实执行路径里，它们其实是在处理不同阶段的同一条成本链。  
shared memory 先解决“数据能不能稳定留在片上”，pipeline 再解决“这些数据和计算能不能交错起来”，layout 则是在前两步成立以后，进一步把访问模式调整到更顺手的状态。  
也正因为它们处理的是不同阶段，所以这三步不能乱序，否则前一步还没立住，后一步就只能在空中优化。

## 先找瓶颈，再排顺序

复杂 attention 之所以难，不在算式本身，而在分块、布局、同步和调度的组合复杂度。  
换句话说，难点不是“怎么写一个 attention”，而是“怎么让 attention 在真实硬件上稳定跑快”。

最先要看的，还是 shared memory。  
如果某些 `Q/K/V` 子块会被 block 内反复使用，就应该优先把它们放进 shared memory，而不是每次都重新从慢层取。  
这一层解决的是复用问题，是整条链路里最基础的前提。

shared memory 之所以放在最前面，不是因为它最复杂，而是因为它最像前提条件。  
如果数据还没在片上被稳定复用，后面的 pipeline 和 layout 就没有牢固的落点。  
换句话说，这一步不是最后的优化，而是让后面优化成立的基础。

可以把这个判断想成一个很具体的场景：如果某个 `Q/K/V` 子块在同一个 block 内会被读很多次，那先把它搬到 shared memory，通常比先去改访问路径更划算。  
因为这时真正贵的不是“怎么读得更顺”，而是“要不要每次都重新从慢层把同一批数据取回来”。  
只有先把这层复用立住，后面的 pipeline 才有可能把等待藏起来，layout 也才有意义去修访问方向。

这也解释了为什么复杂 attention 往往不能靠单一技巧直接解决。  
如果先谈 layout，很多时候只是把访问路径修得更顺一点，但真正贵的那部分数据还在慢层里反复进出；如果先谈 pipeline，等待可能会被压缩，但复用没有建立起来，收益也会变得很薄。  
所以 shared memory 不是一个附带选项，而是整个顺序里最先要立住的支点。

如果要再具体一点，调优时可以先问自己一句：这一步优化之后，数据是不是还会在同一个 block 里继续被用到。  
如果答案是肯定的，就优先想 shared memory；如果答案是否定的，才去考虑 pipeline 是否能把等待压掉，或者 layout 是否能把访问变顺。  
这个判断不是为了限制选择，而是为了避免把“还没复用起来”的问题误判成布局问题。

比如在一个 attention block 里，当前 `Q` 块会反复配合好几个 `K/V` 子块计算时，先把这批数据放进 shared memory，通常比先去调 layout 更划算。  
因为这时真正的瓶颈不是“访问顺不顺”，而是“同一批数据会不会被反复从慢层取回来”。  
只有当这层复用先站稳，后面的 pipeline 才有机会把等待藏起来，layout 也才有空间去修访问方式。

当复用已经有了，下一步就是把时间压平。  
pipeline 的意义不是让算子更复杂，而是隐藏等待：让加载和计算交错起来，从而减少空泡。  
如果加载和计算还能明显重叠，pipeline 往往比单纯改布局更有效。

这里还有一个很现实的判断：如果数据本来就反复被使用，那么先把它稳定放到片上，比急着去压缩等待更重要。  
因为等待只有在“数据已经能被重复利用”的前提下，才值得继续压；否则你只是让一个更快的空转去掩盖一个还没解决的搬运问题。  
所以这一节真正教的不是“三种技巧都要用”，而是先确认哪一层最值得被改动，再决定下一层怎么接上。

如果再换成另一个场景：当当前块已经能稳定复用，但下一块也能提前准备时，pipeline 才会真正开始显出收益。  
这时它压掉的不是“多余计算”，而是块与块之间原本空着的等待时间。  
所以 pipeline 的价值，前提仍然是前一层复用已经成立。

这也意味着，这一页不是在给出固定答案，而是在给出一个稳定的诊断顺序：先看复用，再看重叠，最后看布局。  
顺序一旦明确，读者以后面对别的复杂算子，也能先判断“成本在哪一层”，再决定“优化从哪一层开始”。

最后再看 layout。  
如果数据已经留在片上，但访问方式仍然别扭，那收益还是会被访问冲突吃掉。  
所以 layout 往往是最后一层修正，用来把前面做好的复用和流水线，真正落到硬件喜欢的访问模式上。

这里最容易看出差别的，就是“已经留在片上但还是读得别扭”的情况。  
这时如果只看布局，确实能把访问顺序修顺一点；但如果前面的 shared memory 还没站稳，layout 的收益就会被搬运成本抵消掉。  
所以它更像最后一公里，而不是第一步。

如果前两步都成立，layout 才是在“优化最后一公里”；如果前两步还没站稳，layout 往往只是在修表面现象。  
所以这一页强调顺序，不是为了限制选择，而是为了避免把还没解决的瓶颈误判成布局问题。

把这三层放在一起看，就会发现复杂调度真正想解决的，并不是单点性能，而是让 attention 的各个瓶颈按照正确顺序被消化掉。  
先把数据放近，再把等待压平，最后把访问路径修顺，复杂算子才不会变成一堆局部最优的拼接。

从结果上看，这种顺序感会直接决定最终收益是不是稳定。  
如果只改 layout，可能只是把访问冲突换一种形式；如果只上 pipeline，可能只是把等待藏得更深；只有当 shared memory、pipeline、layout 三者都按顺序推进时，attention 的整体路径才会真正变顺。

这也是复杂算子和基础算子最大的不同。  
基础算子里，单点优化的效果通常比较直接；复杂算子里，任何一个局部收益都可能被其他阶段的成本抵消掉。  
因此这一页最后想让读者形成的，不只是“有这三个工具”，而是“知道它们各自应该在什么时候出现，以及为什么不能把顺序反过来”。

## 为什么顺序不能反过来

这个顺序的含义很直接：先找最热的数据，再确认它能不能留在片上，再看能不能用 pipeline 压掉等待，最后才用 layout 把访问模式修顺。  
如果顺序反过来，通常只会把复杂度堆上去，而不会把瓶颈真的解决掉。

读者真正需要带走的，不是三个名字，而是一种排障习惯：先判断“贵的是读、等还是冲突”，再决定“先改哪一层”。  
这就是复杂算子调优和基础算子调优最大的分界线。

换句话说，这一页最后留下的不是一组技巧，而是一条固定的调优路径。  
只要路径稳，后面的 benchmark 才能真正说明问题，而不是给局部操作贴成绩单。

这也是为什么这一节要延续第 5 章的调优方法论。  
如果读者只记住“shared memory、pipeline、layout”三个词，那还是停在术语层；真正学会这节，是能在一个 attention 里判断出先后顺序。  
也就是说，这页要教的不是“有什么工具”，而是“先用哪个工具”。

## 这一节最后要判断什么

读完这一节，读者至少应该能回答三件事：

1. 瓶颈到底是在加载、计算还是同步
2. 先改 shared memory、pipeline 还是 layout 更合理
3. 为什么复杂算子更需要固定的调优顺序

如果这三件事说清楚了，复杂调度就不再是“把技巧堆在一起”，而是“把技巧按瓶颈顺序排好”。  
这也是它能把高级算子从“能实现”推到“能上线”的原因。

从写作上说，这一节其实是在把第 5 章的判断模型重新带回 attention 场景。  
第 5 章里，读者先学会了怎么判断并行、流水线和布局各自对应什么问题；到了这里，读者要进一步学会的是，在更长的数据流里，这些判断仍然如何成立。  
一旦这层关系建立起来，复杂调度就不再像是一个新的知识点，而是前面方法论的延伸。

这一页的作用，和第 5 章里的并行、流水线、布局其实是同一条逻辑的延伸，只是复杂度更高。  
第 5 章是在讲调优顺序的骨架，第 7 章是在复杂 attention 里验证这套骨架还成立，而且必须成立。

从内容推进上看，这一页也是在把第 5 章的判断逻辑迁移到更长的数据流里。  
第 5 章讲的是怎么判断先改什么，第 7 章讲的是当算子复杂到一定程度后，这个判断为什么仍然有效。  
这样一来，复杂调度就不再像是多了几种技巧，而是调优方法本身在更难场景里的再验证。

如果再往下想一步，就会发现这页其实是在回答一个更现实的问题：当你面对一个复杂 attention 时，为什么不能凭经验“什么都试一遍”。  
因为如果没有顺序，结果往往只是把复杂度一点点堆上去，而没有把瓶颈真正消掉。  
这就是这一页为什么必须讲顺序，而不能只讲工具名。
