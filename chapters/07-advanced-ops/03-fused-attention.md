# 7.3 Fused Attention

## 这一节要解决什么问题

真正的工程场景里，attention 往往不是单独出现的，而是会和 RoPE、RMSNorm、masking 等步骤连在一起。  
这一节要说明融合为什么能省时间、省内存。

它的价值和第 5 章的布局优化很像：不是单步更炫，而是把多个步骤之间的等待和往返搬运压掉。

## 这一节要讲什么

### 1. 融合目标

- Attention + RoPE
- Attention + RMSNorm
- Attention + masking

### 1b. 为什么这些适合融合

- RoPE 只是在 `Q / K` 上做位置编码，通常不需要单独落盘
- RMSNorm 往往在进入 attention 之前或之后出现，和输入/输出张量生命周期很近
- masking 只是在 `scores` 或 `probs` 上做条件裁剪，不应该单独作为一个完整阶段存在

这三类操作的共同点是：  
它们都不太依赖长寿命中间张量，适合和 attention 主路径绑在一起。

### 2. 中间张量消除

- 少一次写回
- 少一次读取
- 少一次同步

### 2b. 一个更具体的理解

融合之前，流程通常像这样：

```text
load -> preprocess -> attention -> postprocess -> store
```

融合之后，更像这样：

```text
load -> inline preprocess -> attention -> inline postprocess -> store
```

差别不在名字，而在中间状态是否被迫写回到慢层内存。

### 3. 融合的代价

- 代码更复杂
- 调试更难
- 但通常收益明显

### 4. 当前写作口径

- 先讲为什么这些步骤天然适合融合
- 再讲中间张量为什么是成本来源
- 最后讲融合的复杂度为什么值得付出

## 这一节的结论

Fused attention 的关键不是“把代码写到一块”，而是“把中间状态尽量留在片上”。
这类优化通常会同时改善吞吐、显存和端到端延迟。

## 这一节可以进一步追问的问题

- 哪些步骤只是在张量生命周期上很近，适合内联
- 哪些步骤会引入过多复杂度，反而不适合融合
- 融合后的收益是来自少写回，还是来自少同步

## 常见误区

- 误区 1：融合越多越好
- 误区 2：只要名字里带 fused，就一定更快
- 误区 3：融合只是代码重构，不是性能策略

融合的边界，应该由数据流和生命周期来决定，而不是由“能不能塞进一个 kernel”来决定。
