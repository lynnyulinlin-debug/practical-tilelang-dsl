# 7.3 Fused Attention

如果说 FlashAttention 解决的是 attention 内部的数据流问题，那 fused attention 解决的就是 attention 周边那些“本来可以顺手做掉，却被拆成独立阶段”的事情。  
它的核心不是把代码写在一起，而是把中间张量尽量留在片上，减少不必要的往返搬运。

从同步角度看，融合也在减少边界上的等待：原本每个独立阶段都可能引入一次写回、一次读入和一次额外同步，融合之后这些边界就更容易被压缩掉。

这也是为什么融合和 FlashAttention 经常一起出现：前者收紧边界，后者压缩核心路径。  
它们处理的不是同一个细节，但目标都很一致，就是少让中间状态在慢层内存里多跑一圈。

这一页的重点，是把“融合”从代码组织问题拉回到生命周期问题。  
只要某个步骤的中间结果短、近、轻，就应该优先考虑内联掉，而不是再让它单独走一遍慢层内存。

这里的核心判断其实很朴素：如果一个步骤的结果既不会长期保存，也不会被很多地方重复消费，那它就不值得独立承担一次完整的读写往返。  
融合正是把这种“短命中间状态”收回来，避免它们在慢层内存里多走一圈。

换个角度说，融合不是在追求“更大的 kernel”，而是在追求“更短的中间生命周期”。  
只要生命周期够短，单独开一个阶段就往往不划算。

从实践角度看，这类融合最容易被忽略的地方，不是“能不能合并”，而是“合并之后是不是还保留了原本的依赖关系”。  
RoPE、RMSNorm、masking 这些步骤之所以值得合并，不只是因为它们短命，更因为它们都贴着主路径，单独跑一遍其实是在重复制造中间状态。

## 为什么这些边界值得合并

真正的工程场景里，attention 往往不是单独出现的，而是会和 RoPE、RMSNorm、masking 等步骤连在一起。  
这些步骤的共同点很明显：它们都只是对 attention 主路径上的张量做一次轻量变换，而且生命周期都很短。  
如果把它们单独拆成独立阶段，中间结果就很容易被迫写回慢层内存。

也正因为它们贴着主路径，所以它们最容易被写成“看起来独立、实际上很近”的阶段。  
这里的判断不是“能不能写成一个函数”，而是“是否值得为这个函数单独付一次完整内存往返的成本”。

所以 fused attention 的关键，不是“把代码写到一块”，而是“把中间状态尽量留在片上”。  
它和 FlashAttention 的关系也很紧：前者压缩周边阶段，后者压缩核心注意力阶段，最终目标都是少让中间状态落到慢层内存。

这也意味着，融合并不是一味追求更大的 kernel，而是在找一个更合适的边界。  
边界太碎，会导致中间张量反复落盘；边界太大，又会让调试和维护成本迅速上升。  
所以真正值得做的，是把那些短命、紧邻主路径、而且重复走一遍就很浪费的步骤合并起来。

所以融合的判断标准其实有三条：生命周期短、离主路径近、单独跑一遍的往返成本高。  
三条里只满足一条，通常还不够；三条同时满足，融合才更像是一次真正有效的收口。

## 融合前后，路径怎么变

融合之前，流程通常像这样：

```text
load -> preprocess -> attention -> postprocess -> store
```

融合之后，更像这样：

```text
load -> inline preprocess -> attention -> inline postprocess -> store
```

差别不在名字，而在中间状态是否被迫写回到慢层内存。  
RoPE 只是在 `Q / K` 上做位置编码，通常不需要单独落盘；RMSNorm 往往和输入或输出张量生命周期很近；masking 只是在 `scores` 或 `probs` 上做条件裁剪，也不应该单独成为一个完整阶段。  
只要这些步骤的边界足够近，融合就能同时减少写回、减少读取、减少同步。

## 代价从哪里来

融合不是免费午餐。  
它会让 kernel 更难拆分、更难调试，所以真正值得融合的，是那些中间状态很轻、但往返成本很高的步骤。  
如果融合的代价已经超过收益，那它就不再是优化，而是在制造复杂度。

这也是为什么这一节最重要的判断，不是“能不能融合”，而是“什么样的边界值得被合并”。  
融合的边界应该由数据流和生命周期来决定，而不是由“能不能塞进一个 kernel”来决定。

这类判断一旦建立起来，读者以后看别的算子也会更清楚：不是所有能连在一起的步骤都值得融合，也不是所有生命周期短的步骤都一定值得融合。  
真正值得合并的，是那些既短命、又紧邻主路径、又能明显减少往返成本的步骤。

换句话说，这一页解决的不是“有没有办法把算子拼起来”，而是“哪些边界保留更划算，哪些边界合并更划算”。  
这比单纯追求更多融合更重要，因为它直接决定了代码复杂度会不会超过收益。

## 这一节最后要判断什么

读完这一节，读者至少应该能回答三件事：

1. 哪些步骤只是生命周期很近，适合内联
2. 哪些步骤会引入过多复杂度，反而不适合融合
3. 融合后的收益是来自少写回，还是来自少同步

从故事线看，这一页是在 FlashAttention 之后继续收边界：核心路径已经被压缩了，周边的 RoPE、RMSNorm 和 masking 再单独跑一遍就显得浪费。  
所以融合不是另起炉灶，而是在已经收窄的路径上继续删掉不必要的往返。

如果把这一页和前面那一页连起来看，读者会发现第 7 章在做的是两次收缩：先把 attention 核心路径压小，再把周边步骤往里收。  
这就是为什么 fused attention 不是一个独立技巧，而是前一轮重排之后的第二轮收口。

从写作顺序上看，这一节也承担了一个很明确的角色：先把“哪些步骤值得收进来”说清楚，再把“收进来以后为什么更省”说清楚。  
这比直接列出可融合算子更重要，因为它让读者学到的是判断方式，而不是名单。
