# 2.3 计算单元：CUDA Core 与 Tensor Core

## CUDA Core 和 Tensor Core 的分工

GPU 不是只有一种计算单元。
不同单元适合不同类型的计算，尤其是矩阵乘法类任务。

## 计算单元要抓住的判断

- `CUDA Core` 适合通用算术逻辑
- `Tensor Core` 适合矩阵乘加
- TileLang 后面的 `T.gemm`、fragment 和 tile 设计，都要围绕专用计算单元展开
- 不同计算单元对应不同的数据布局和最优 tile 形状
- 计算单元越专用，越依赖输入形状和布局匹配

## 计算单元的常见现象

- 为什么小矩阵不一定能吃满 Tensor Core
- 为什么数据布局会影响单元利用率
- 为什么 fragment 对矩阵计算很关键
- 为什么同样是 GEMM，不同 tile 形状会出现很大的性能差异

## 计算单元的判断标准

读完这一节，应该能够判断：

- 什么时候应优先使用通用计算单元
- 什么时候可以把问题映射到更专用的矩阵单元
- 为什么“理论峰值很高”不等于“实际就能达到”
- 为什么 fragment 和 tile 形状要跟硬件规格对齐

## 计算单元的常见误区

- 误区 1：Tensor Core 总是更快
- 误区 2：只要算子是 GEMM，就一定能直接吃满矩阵单元
- 误区 3：只看 FLOPs，不看数据布局和访存路径

专用单元的优势成立前提是“问题形状和数据布局匹配”。  
如果布局不对，硬件能力就会被浪费掉。

这也是为什么同样是 GEMM，有些形状能很自然地映射到 Tensor Core，有些形状却只能先在通用单元上跑。矩阵单元更吃输入规整度：尺寸太小、边界太碎、布局不顺，都会让理论峰值很难兑现。对比起来，CUDA Core 更像“什么都能算”，Tensor Core 更像“形状对了就特别快”，所以后者更依赖 fragment 和 tile 的配合。

比如在一个规整的 `m16n16k16` 类 tile 上，Tensor Core 更容易把 fragment 直接吃进去；但如果 tile 很碎、边界很多、布局不顺，很多时候就得先在通用单元上做更多整理工作，最后才能喂给矩阵单元。算子不是先看“有没有 Tensor Core”，而是先看“当前形状是不是它擅长的那种问题”。

这在实际排查里很常见：有些 GEMM 看起来规模不小，但因为 M / N / K 里有一边特别碎，最后只能一部分走矩阵单元、一部分走通用单元；也有些算子本来数据量够大，却因为布局没整理好，让矩阵单元吃不到连续、规整的 fragment。先知道“形状先对齐，单元才吃得上”，后面看到 Tensor Core 才不会只想着“开没开”，而会先看“值不值得用、能不能用好”。

这一节只需要先知道：
不是所有算子都应该去追 Tensor Core，形状和布局对了才谈得上吃满优势。
