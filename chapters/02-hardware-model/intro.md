# 第 2 章导言：GPU 硬件模型与执行层次

## 章节定位

这一章把“为什么性能会差很多”转换成硬件语言。  
后面的分块、shared memory、pipeline 和后端适配，都建立在这一章的概念底座上。

## 本章最小深度要求

读完这一章，应该能够把性能问题拆成四类：

- 是执行层次没用对
- 是内存层次没对齐
- 是计算单元没有匹配到合适的 tile
- 还是数据布局和访问方式本身就不合理

## 本章目标

- 建立 GPU 执行层次
- 建立 GPU 内存层次
- 说明 CUDA Core 与 Tensor Core 的差异
- 把 tile 的必要性和硬件约束连接起来
- 让后续优化判断有一个硬件基准

## 本章判断标准

写得足够深时，读者不只是知道术语，而是能判断：

- 哪一层的粒度决定了同步边界
- 哪一层的内存最值得复用
- 为什么某些 kernel 在单卡上很快，在换硬件后就失效
- tile 大小为什么会影响性能、正确性和可维护性

## 内容块安排

### 2.1 执行层次
- `Thread / Warp / Block / Grid`
- warp divergence
- block 内同步和 block 间隔离
- 哪些通信成本是“组内低成本”，哪些必须走更慢的路径

### 2.2 内存层次
- `Register / Shared / L2 / Global`
- 延迟和带宽
- 对齐与合并访问
- 哪些数据应该尽早留在片上，哪些数据只能按需搬运

### 2.3 计算单元
- CUDA Core
- Tensor Core
- 片上矩阵计算单元的意义
- 为什么“更多 FLOP”不一定意味着“更快”

### 2.4 为什么需要 tile
- 数据复用
- 并行粒度
- 和内存层次对齐
- tile 不是为了把代码写得更复杂，而是为了让数据停留在更合适的层次

### 2.5 本章总结
- 用硬件约束解释后续设计选择
