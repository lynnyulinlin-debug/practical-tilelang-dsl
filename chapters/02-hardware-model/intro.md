# 第 2 章导言：GPU 硬件模型与执行层次

## 章节定位

这一章把“为什么性能会差很多”转换成硬件语言。最容易遇到的问题不是“我想先背 GPU 术语”，而是“为什么同样的 kernel，改了一点布局或 tile，性能就会差一截”。这里讨论的是 GPGPU 场景，也就是把 GPU 当成通用并行计算平台来用。后面的分块、shared memory、pipeline 和后端适配，都建立在这一章的概念底座上。比如同样是 attention，前面一版还能靠简单布局跑通，换成更长的序列或更大的 head 后，原来的组织方式就会突然暴露出同步和搬运开销。

GPGPU 这条路线之所以重要，是因为 GPU 从图形渲染走向通用并行计算后，硬件层次、编程模型和生态工具就开始一起影响“怎么写、怎么编、怎么跑、怎么调”。

如果第 1 章已经让你看到“为什么要分层”，那第 2 章要补上的就是：这些层到底落在硬件的什么位置上，为什么一旦放错层，性能就会掉得很明显。  
从整套教程来看，这一章更像是桥前的硬件因果层：先把地形看清楚，后面的编译、分块和调优才知道该往哪里落。

这一章还承担一个重要任务：把看起来像参数问题的事情，重新解释成硬件层次不匹配的问题。一旦这个视角建立起来，后面的调优就不会停留在猜参数，而会转向看执行层次、数据流和计算单元是否真的对齐。比如同一个 attention kernel，只是把 tile 改小、把布局换一下，慢下来的原因往往不是“算少了”，而是数据开始在更不合适的层次之间来回移动；再往前一步，warp 怎么分工、shared memory 够不够放、Tensor Core 能不能吃到合适形状，都会直接影响结果。

对初学者来说，最重要的不是记住全部硬件名词，而是先形成一个直觉：同样是写 kernel，为什么换一种组织方式，性能就会差很多。直觉先建立起来，后面的名词就不会显得那么硬；相反，如果一上来只背术语，后面看到性能波动时还是很难把“到底卡在哪一层”说清楚。

从计算机体系结构的角度看，这其实是在看一个老问题：程序的速度不只是算术本身决定的，还取决于执行方式和数据移动方式。GPU 把这个问题放大了，所以我们必须同时看执行层次、内存层次和计算单元。SIMT 只是这一套组织方式里的一个关键接口，它决定了一批线程怎么一起推进，也决定了为什么分支发散会显著影响吞吐。

这一章覆盖的 GPGPU 优化技术，主要放在“执行怎么组织、数据怎么停靠、计算单元怎么匹配、tile 怎么对齐”这四类问题上。像分支发散、内存层次、bank conflict、coalescing、alignment、Tensor Core 匹配这些内容，会在这一章里作为硬件判断框架讲清楚；更细的布局修正、swizzle、跨平台迁移和平台专有指令差异，则放到后面的调优章和多后端章展开。

至于国产硬件的加速指令和编程模型差异，这一章只先把它们放进“多后端适配”这个背景里，不单独展开。真正系统的比较，会留到第 8 章。

## 本章最小深度要求

读完这一章，应该能够把性能问题拆成四类：

- 是执行层次没用对
- 是内存层次没对齐
- 是计算单元没有匹配到合适的 tile
- 还是数据布局和访问方式本身就不合理

## 本章目标

- 建立 GPU 执行层次
- 建立 GPU 内存层次
- 说明 CUDA Core 与 Tensor Core 的差异
- 把 tile 的必要性和硬件约束连接起来
- 让后续优化判断有一个硬件基准

这一章不是要把硬件讲全，而是要先让读者知道：性能问题常常不是“代码写错了”，而是“数据和计算没有放在合适的层次上”。

## 本章判断标准

写得足够深时，读者不只是知道术语，而是能判断：

- 哪一层的粒度决定了同步边界
- 哪一层的内存最值得复用
- 为什么某些 kernel 在单卡上很快，在换硬件后就失效
- tile 大小为什么会影响性能、正确性和可维护性

如果读者还做不到这些判断，也没关系。先能说出“这一段更像卡在执行层、内存层还是 tile 匹配上”，就已经比单纯背名词更有用了。

## 内容块安排

### 2.1 执行层次
- `Thread / Warp / Block / Grid`
- warp divergence
- block 内同步和 block 间隔离
- 哪些通信成本是“组内低成本”，哪些必须走更慢的路径
- 先把执行层次讲成一条故事线：从单个线程如何发出指令，到一组线程如何协作，再到更大范围如何分工

### 2.2 内存层次
- `Register / Shared / L2 / Global`
- 延迟和带宽
- 对齐与合并访问
- 哪些数据应该尽早留在片上，哪些数据只能按需搬运
- 解释为什么“离计算越近越快”不只是口号，而是数据复用和访问代价共同决定的结果

### 2.3 计算单元
- CUDA Core
- Tensor Core
- 片上矩阵计算单元的意义
- 为什么“更多 FLOP”不一定意味着“更快”
- 从 GEMM 和矩阵类算子切入，说明计算单元的类型会直接改变 tile 的最优形状

### 2.4 为什么需要 tile
- 数据复用
- 并行粒度
- 和内存层次对齐
- tile 不是为了把代码写得更复杂，而是为了让数据停留在更合适的层次

### 2.5 本章总结
- 用硬件约束解释后续设计选择

## 写作提醒

这一章要尽量避免把执行层次和内存层次写成孤立名词表。更好的写法是让读者看到一条连续因果链：线程怎么组织，数据怎么流，算子为什么卡住，tile 为什么能把这些问题重新对齐。也就是说，这一章的目标不是“记住所有层次名词”，而是“学会用层次关系解释性能差异”。对初学者来说，只要能先把“慢”拆成层次问题，这一章就算入门成功。
