# 2.5 本章总结

## GPU 层次、单元和 tile 的边界

1. GPU 有明确的执行层次，不是“很多线程随机并行”
2. 内存层次决定了数据是否是瓶颈
3. 专用矩阵计算单元决定了 GEMM 这类算子的上限
4. tile 是把计算逻辑和硬件层次对齐的基本方法

这一章真正想建立的不是“GPU 术语记忆”，而是性能直觉。当一个 kernel 被看成“在哪一层发起并行、在哪一层搬运数据、在哪一层做主要计算”的组合时，后面的优化判断才算有了起点。

这一章的完成标志，不是把概念背熟，而是能在看到性能异常时先问出三个问题：

- 问题卡在执行层次吗
- 问题卡在内存层次吗
- 问题卡在 tile 和硬件的匹配上吗

能先问对这三个问题，后面第 4 章到第 10 章的所有案例才有统一的硬件解释框架。对初学者来说，这一章到这里已经够用了：先学会问“卡在哪一层”，比先背完硬件名词更重要。比如看到一个慢的 kernel，先别急着改参数，先判断它更像是执行层次错了、数据停靠层次不对，还是 tile 和硬件没有对齐。图表的作用也是一样，是把“执行层次 / 内存层次 / 计算单元 / tile”这四个判断点固定成后续的硬件坐标系。

## GPU 层次、单元和 tile 的判断

- 哪一层的粒度决定了同步边界
- 哪一层的内存最值得复用
- 哪些访问模式会浪费带宽
- 哪些算子需要更强的布局和 tile 匹配
- 为什么“更多线程”或“更大 tile”都不是天然更优

## 第 2 章的收束判断

这一章尽量少讲晦涩术语的堆叠，多讲“层次关系”和“性能原因”。后面的分块、调度和多后端适配，都依赖这里的概念底座。只要已经能拿着一个 kernel 先判断它更可能卡在哪一层，后面的优化就不会只剩参数记忆。
