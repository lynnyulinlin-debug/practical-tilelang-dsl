# 2.2 GPU 内存层次

## 要说明的问题

GPU 性能瓶颈通常不在“算得不够快”，而在“数据来得不够快”。

从体系结构角度看，这就是“执行单元在等数据”的问题。GPU 并不是把等待消掉，而是尽量用更多并行去隐藏等待；所以内存层次、带宽和访问模式会直接决定这种隐藏能不能成功。

## 要点

- `Register` 最快，但容量最小，线程私有
- `Shared Memory` 允许 block 内复用
- `L2 Cache` 由整个 GPU 共享
- `HBM / Global Memory` 容量大，但延迟最高
- 内存越快，通常越小、越局部、越需要显式管理
- 同样的数据如果能在更高层被复用一次，整体性能通常就会改善

这四层不是简单的“越上越好”，而是各自对应不同的复用范围。
真正的工程问题不是“有没有更快的内存”，而是“当前这批数据应该停在哪一层最划算”。

在一个分块后的 GEMM 里，这个问题会变得很具体：如果同一块矩阵片段要被多个线程反复读取，那么每次都回到 Global Memory，代价就会被放大；如果能先搬到 Shared Memory，再让 block 内的线程共享，访问次数就会明显下降。Register 更像最后一段短暂停靠的地方，Global Memory 则是容量兜底，真正高频重复的数据通常不该总是绕回去。

但“搬进来”不代表“就一定快”。如果多个线程访问 Shared Memory 时落到同一组 bank 上，硬件就得把这些请求排队处理，这就是 bank conflict。它的直觉很简单：数据已经在更快的层里了，但访问方式还是没排对，结果共享内存的优势被打了折扣。对初学者来说，只要先记住“搬对层还不够，还要访问得顺”，就已经够用了。

这里还要顺手区分两个经常一起出现、但不完全一样的概念：`alignment` 和 `coalescing`。对齐关注的是地址和边界是不是落在硬件更容易处理的位置；合并访问关注的是一批相邻线程的请求能不能被硬件尽量合成更少的事务。它们通常会一起影响带宽，但不是同一件事：对齐更像“起点是否摆正”，合并更像“这一排请求能不能顺着走”。如果起点没摆正，或者线程访问不是连续的，硬件就更难把这些请求合在一起，事务数也就更容易上升。

如果再往前走一步，这里也能看到“同步”的必要性：数据搬到片上之后，往往要先确认它已经准备好，后面的线程才能继续读；而“异步”的意义，就是尽量把等待藏到加载和计算的交错里。

比如在一个 stride 很大的访问里，线程虽然都在读同一批逻辑数据，但实际落点却很分散，带宽就会被浪费掉；这时如果把数据搬进 shared memory，问题又会转成“搬进来之后怎么排访问”，否则 bank conflict 还是会把收益吃掉。所以这一节最关键的，不是“有没有更快的内存”，而是“访问模式有没有先对齐，再谈复用”。

## 这里应该解释的现象

- 为什么对齐访问更快
- 为什么对齐和合并访问经常一起出现
- 为什么随机访问会浪费带宽
- 为什么 bank conflict 会让 shared memory 退化
- 为什么“把数据搬上来”并不自动等于“性能提升”

## 判断标准

读到这里，读者应该能做出下面几类判断：

- 哪些数据值得放进 shared memory
- 哪些数据只适合短暂驻留在 register
- 哪些访问模式会因为 stride 或随机访问而浪费带宽
- 哪些场景下，复用比单纯并行更重要

如果能做到这一步，读者已经能开始把一个 kernel 看成“数据在哪里停留最久、在哪里搬运最多、在哪里复用最少”的过程，而不只是看它做了多少计算。

对初学者来说，这一节的关键体会是：
慢的不一定是算，很多时候是数据来回搬得太多。

这也意味着，后面看到任何优化时，第一反应不该是“要不要再堆线程”，而应该是“数据该换一层停靠了吗”。

## 常见误区

- 误区 1：shared memory 越多越好
- 误区 2：只要显存够大，访问代价就不重要
- 误区 3：所有瓶颈都能靠增加线程解决

实际上，内存层次的关键不是“有多少层”，而是“数据是否落在合适的层上”。Tile 的作用，就是把数据尽量放到该放的层次里。
