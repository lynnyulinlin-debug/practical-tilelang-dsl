# 2.1 GPU 执行层次

## 执行层次和同步边界

GPU 不是“很多线程随便同时跑”，而是有明确的层次组织方式。
理解这个层次，才能理解 TileLang 为什么要显式表达并行。

这一层组织方式也和 `SIMT` 有关：编程时看起来像很多线程各自执行，但硬件往往会把一批线程按 warp 一起推进。它和 `SIMD` 很像，都是“同一批执行单元一起干一类事”，只是 GPU 里更强调线程模型和分支行为。先不用纠结术语差别，先记住一件事就够了：warp 不是抽象名词，而是会直接影响分支、同步和吞吐的观察单位。

## 执行层次要抓住的判断

- `Thread` 是最小执行单元
- `Warp` 是 SIMT 调度的基本单位
- `Block` 允许线程共享片上资源
- `Grid` 是整个 kernel 的启动范围
- 同一层级内的协同成本通常更低，跨层级通信成本通常更高
- 这也是为什么 tile 和同步边界必须一起设计

## 执行层次的常见现象

- 为什么 warp divergence 会拖慢性能
- 为什么 block 之间无法直接同步
- 为什么并行数量不等于有效吞吐
- 为什么“线程更多”不一定“更快”

先记住一个直觉：
并行不是越细越好，而是要把协作放在合适的层级。

这几条现象可以归结成同一个判断：执行层次不是越细越好，而是要和协作边界对齐。
一旦边界定错，后面的 tile 和同步都会跟着偏。

在实际 kernel 里，这通常会表现成两种很常见的卡点：一种是 warp 里每个线程都走了不同分支，结果同一批指令被拆开执行；另一种是 block 之间频繁交换中间结果，导致本来可以局部完成的事情被迫绕到更慢的路径上。执行层次的价值，不在于把线程堆得更多，而在于把需要一起行动的东西放到同一层，把不该一起行动的东西尽量隔开。

比如一个 warp 里如果每个线程走了不同分支，执行就会被迫分段进行；如果 block 之间还要频繁交换中间结果，就只能走更慢的路径。执行层次的价值，不在于把线程堆得更多，而在于让需要一起行动的东西停在同一层，让不该一起行动的东西尽量少交叉。

这里也可以顺手把“同步”先理解成协作边界上的等待：哪些线程必须先等一等，哪些线程可以继续往前走，决定了执行层次里的真实代价。

比如在一个带条件分支的 kernel 里，如果某些线程先处理有效区、另一些线程先处理边界，warp 就会在不同路径之间来回切换；这时候看起来是“代码能跑”，实际上是同一批执行单元被拆成了几段。对初学者来说，先把这种“同一层里行为不一致”的代价看出来，比先背线程数更重要。
