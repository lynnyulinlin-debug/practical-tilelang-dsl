# 2.4 为什么需要 Tile

## 要说明的问题

Tile 不是随便切块，而是为了把计算、数据和硬件层次对齐。

这也是这一章里最容易被低估的一点：tile 不只是性能技巧，它其实是在把一个大问题拆成更容易验证、也更容易复用的局部问题。

## 要点

- 通过 tile 提高数据复用
- 通过 tile 把热点数据搬到更快的内存层
- 通过 tile 控制并行粒度
- 通过 tile 让编译器更容易生成高效代码
- 通过 tile 把“访问模式”改写成更适合硬件的局部模式
- 通过 tile 把大问题拆成可验证、可调优的小问题

所以 tile 既是表达策略，也是工程策略。
它让“问题太大不好查”变成“可以先查局部，再查整体”，这对后面的调优章节非常关键。

这里要顺手区分一下 `block` 和 `tile`：`block` 更像执行和调度的边界，决定哪些线程可以协作、同步和共享片上资源；`tile` 更像数据和计算的划分方式，决定一块数据怎么被切开、怎么复用、怎么和内存层次对齐。两者经常一起出现，但职责不同：block 解决“谁一起干活”，tile 解决“活怎么切、数据怎么放”。

比如在一个分块后的 attention 里，一个 block 可能负责一小块 query/key/value 的计算协作，而 tile 决定这块数据到底怎么切、哪些中间结果先放进片上、哪些结果需要下一轮再读。也就是说，block 决定协作单位，tile 决定数据单位；前者更靠近执行层次，后者更靠近数据和计算的组织方式。

换到 GEMM 里也一样：一个 block 通常对应一块输出区域的协作范围，但 tile 决定这一块输出背后要怎样把 A / B 的子块搬进来、怎样复用、怎样把中间结果送到更靠近计算单元的位置。读者只要记住这一点就够了：block 更像“工作小组”，tile 更像“工作切分表”。

在实际判断里，这通常意味着先问一句：这批数据是不是会被 block 内多次消费？如果会，tile 就不是可选修饰，而是复用的前置条件；如果不会，硬塞 tile 只会让同步和资源压力更重。也就是说，tile 的存在感越强，往往说明“这块数据值得被多用几次”。

在真实优化里，tile 还是一个必须折中的参数：太小会让调度和同步的相对成本变高，太大又会把寄存器压力、共享内存压力和边界处理一起抬上来。也就是说，tile 不是单纯追求切得细，而是在数据复用、并行粒度和硬件资源之间找一个更稳的平衡点。

对初学者来说，这一节最重要的感受是：
tile 不是先把问题拆碎，而是先把问题放到更容易复用、也更容易对齐硬件的位置。

## 这里应该解释的现象

- 为什么大多数高性能 kernel 都离不开 tile
- 为什么 tile 大小不是越大越好
- 为什么 tile 需要结合寄存器、共享内存和 block 粒度一起设计
- 为什么 tile 是性能、正确性和可维护性的共同折中点
