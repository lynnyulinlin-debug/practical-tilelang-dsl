# 1.4 TileLang 的位置

## TileLang 解决什么问题

TileLang 介于“高层算子表达”和“底层 kernel 手写”之间。

它的目标不是把硬件细节藏起来，而是让开发者在需要时可以明确表达：

- 怎么分块
- 怎么分配内存层次
- 怎么安排并行
- 怎么下钻到更低层控制
- 怎么把一个 AI kernel 的结构和优化边界显式写出来

还是回到 attention 的例子：
第一版 kernel 能跑，不代表下一版换了 shape 之后还能轻松改；如果没有一个能承接分块、布局和后端差异的表达层，这些变化就会不断回到手写代码里。
这时候，TileLang 的作用就很具体：它让你在改结构的时候，不必立刻把所有硬件细节一起重写。
有些决定可以先留在更高层，等到编译链路和后端再去处理。
比如这次改 shape，你可能只想先把 tile 重新排一遍，确认哪些数据还能复用、哪些数据要重新搬运；如果每次都直接落到手写 kernel 里，你就不得不同时改布局、同步和边界处理。
TileLang 的意义就在于，它让这些改动先停在“结构怎么变”这一层，而不是一上来就把“实现怎么写”全盘推倒。
如果再往前走一步，你会发现它还会影响你后面的判断顺序：先决定 tile 怎么放，再看 memory tier 怎么承接，最后才决定要不要下沉到更细的 kernel 代码。
这比一开始就把所有细节写死要灵活得多，因为你可以先保留一部分余地，等真正需要的时候再收紧。
对同一个 attention 场景来说，这意味着你可以先调 tile 和 layout，再看要不要把某些中间状态放进 shared memory，最后才决定是不是继续往更低层实现。
TileLang 让你把这些决策拆开，而不是每次都在同一层里把问题一次性打完；这样你就能先验证一件事，再决定下一件事。

这也是它和前面几层的关系：
1.1 让你看到返工问题，1.2 让你看到 DSL 为什么有价值，1.3 让你知道编译器只是桥。
TileLang 出现在这里，是因为它要把这些东西真正接到可用的工程实践上。
所以它不是在前面三层之外再加一层负担，而是把前面三层真正变成可用工作的那层。

## 这一节的结论

TileLang 不是“另一个 kernel 语法”，而是把结构、调度和编译链路放进同一条更可控的路径里。

对初学者来说，只要先记住一句话就够了：
TileLang 不是用来替代手写 kernel 的，而是让后面的组织、迁移和调优更可控。
如果你以后经常遇到“同一个算子要在不同硬件上跑”“同一套逻辑要反复调 tile”“同一份计算要被不同后端接住”这类问题，TileLang 才会显出明显价值。
换句话说，它最适合的不是“第一次把东西写出来”的阶段，而是“后面还要反复改”的阶段。
如果你把它放到这个位置上理解，很多设计就会顺很多。
也正因为这样，TileLang 更像是给“持续优化”准备的一层工作台，而不是给“第一次写完”准备的一套语法包装。
如果你现在只是想先把一个固定 kernel 写出来，它的重要性还不一定立刻显现；但只要你开始考虑第二次、第三次修改，它的位置感就会变得很清楚。它和前面的 DSL、编译器是连着看的：DSL 负责把结构抽出来，编译器负责把结构接到后端，而 TileLang 负责把这些东西真正用到算子改动里。

## 判断标准

如果一个场景符合下面几条，TileLang 的价值会更高：

- 你需要显式控制 tile、layout 和 memory tier
- 你需要跨硬件迁移同一类 kernel
- 你需要把调优过程写成可复用流程
- 你希望编译链路帮你承接一部分实现复杂度

如果这些需求都不强，直接手写 kernel 或使用更高层框架可能更合适。
