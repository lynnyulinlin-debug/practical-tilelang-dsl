# 1.4 TileLang 的位置

## TileLang 解决什么问题

TileLang 介于“高层算子表达”和“底层 kernel 手写”之间。

它的目标不是把硬件细节藏起来，而是让开发者在需要时可以明确表达：

- 怎么分块
- 怎么分配内存层次
- 怎么安排并行
- 怎么下钻到更低层控制
- 怎么把一个 AI kernel 的结构和优化边界显式写出来

## 和其他工具的关系

### 和 CUDA

- CUDA 更像直接写实现
- TileLang 更像先写可编译的结构化描述，再让编译链路落地
- CUDA 的强项是细粒度控制
- TileLang 的强项是把控制和表达放进同一条可维护链路

### 和 Triton

- Triton 更偏高层抽象和自动处理
- TileLang 更偏显式控制和多后端适配
- Triton 适合快速得到一个高性能实现
- TileLang 更适合要迁移、要下钻、要控制边界的工程场景

### 和 TVM

- TileLang 依托 TVM 的编译基础设施
- 但表达重点更聚焦 AI kernel，而不是通用张量编程
- TVM 提供编译和后端能力
- TileLang 提供更聚焦的 AI kernel 表达方式

## 这一节的结论

TileLang 的价值不在于“又一种语法”，而在于它把 kernel、调度和编译链路组织成了一个更适合 AI 工程的层次结构。

## 判断标准

如果一个场景符合下面几条，TileLang 的价值会更高：

- 你需要显式控制 tile、layout 和 memory tier
- 你需要跨硬件迁移同一类 kernel
- 你需要把调优过程写成可复用流程
- 你希望编译链路帮你承接一部分实现复杂度

如果这些需求都不强，直接手写 kernel 或使用更高层框架可能更合适。
