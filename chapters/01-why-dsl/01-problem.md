# 1.1 为什么只写 kernel 不够

## 核心问题

### 一个具体场景

假设你已经写出一个 attention kernel，它在固定的 batch、sequence length 和 head dimension 下可以正常跑通。  
这时候代码大体是稳定的，block 怎么切、访存怎么排、同步怎么放，都已经和当前 shape 对上了。
如果你回头看这份实现，它往往会很像一张已经画好的线路图：每一步都能对上当前输入，但也正因为如此，它对输入之外的变化并不宽容。

第一次返工发生得很自然：产品把 head dimension 改了一档。  
这时你会发现，原来能直接复用的 tile 切法不一定还合适，shared memory 的分配也可能要重算，某些边界分支甚至要重新写一遍。
如果这个 kernel 原本就是按某个固定块大小写出来的，head dimension 一变，很多原本对齐得很顺的地方都会散开：有的块装不满，有的块要拆开，有的地方要补边界处理。
这时候改动看起来只是“改了一项参数”，实际却是在重新整理一整套组织方式。
更麻烦的是，这类变化通常不是只发生一次。今天是 head dimension，明天可能就是 batch，后天又可能是 sequence length。你每次都在重新确认同样的结构是否还成立。

第二次返工通常更麻烦：后端换了。  
同一份计算逻辑还在，但底层对访存、并行粒度和同步方式的要求变了，原来那套“能跑”的实现开始暴露出更多硬编码细节。
比如原来某些访存顺序在当前后端下是合理的，换了后端之后就可能变成瓶颈；原来默认的同步边界也可能不再适合新的执行方式。
这些东西平时藏得很深，一旦环境变化，就会集中冒出来。
这时候你会发现，原来写进去的不是纯粹的“算法”，而是很多默认前提：当前硬件是什么、当前 block 怎么组织、当前数据怎么流。前提不变时它们很方便，前提变了时它们就会变成负担。

### 为什么会反复返工

这时“写得出来”依然成立，但“每次都手改一遍”已经开始不划算了。
kernel 本身当然很直接，它贴近硬件，也最容易控制；问题在于，一旦同类变化反复出现，直接写 kernel 的代价会迅速抬高。
你改的是一个 shape，但要碰的却往往是一整条数据路径。

问题不只是变了，而是变的时候总会碰到同一批实现细节。
这就是为什么很多 kernel 在第一次写完后看起来很轻，后面维护起来却越来越重。
你不是每次都在重新解决一个全新的问题，而是在重复处理同一类变化。
tile 的大小、访存顺序、边界分支、同步位置，这些都不是改一次就彻底结束的地方。
它们会随着输入形状和后端要求反复被碰到，所以代码看起来只是小修小补，实际每次都在触碰核心组织逻辑。
如果一个 kernel 的改动总是落在这些位置上，你就会越来越清楚地意识到：真正难的不是“算”，而是“组织这份算”。

### 为什么 AI kernel 更容易放大这个问题

AI kernel 之所以更容易放大这个问题，是因为它通常不是单纯的算术循环，而是还要同时处理分块、多级缓存、数据复用、融合算子和硬件约束。
这些因素叠在一起后，单点实现会越来越像一次性手工活。
换句话说，写一个 AI kernel 不只是写算法步骤，还要同时记住很多和性能有关的前提。
这些前提短期内能帮你把性能做上去，长期看却会让后续变化越来越难接。
尤其是当你已经为了某个 shape 把数据路径、同步方式和 tile 结构都调顺后，任何新的变化都会让你重新面对一遍这些前提。
这也是为什么 AI 场景里，手写 kernel 很容易从“可控”慢慢变成“只能小心维护”。

第一次做的时候，很多细节可以靠经验补上；第二次还可以继续修；但当相同类型的变化开始不断出现时，维护成本就会明显上升。
尤其是在实际项目里，只要 shape、后端或者目标硬件不是完全固定，这类重复修改就会变成常态，而不是例外。
所以真正的问题不是“kernel 写得对不对”，而是“这个 kernel 能不能继续承受变化”。

### 这一节的结论

所以这节真正要让人记住的不是术语，而是这个判断：
如果问题只出现一次，手写 kernel 可能够用；如果问题会反复出现，表达层就需要往上抽。
