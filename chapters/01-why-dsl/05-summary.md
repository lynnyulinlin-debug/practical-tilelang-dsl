# 1.5 本章总结

## 你应该记住什么

1. kernel 是执行单位，DSL 是表达和组织 kernel 的方式
2. AI 场景对调优、迁移和融合的需求，放大了 DSL 的价值
3. AI 编译器是 DSL 走向实际性能和部署的关键桥梁
4. TileLang 的定位是更可控的 AI kernel 编程层

前面四节其实是在重复同一条线：一个 kernel 如果只是写一次，手写往往就够了；一旦它开始频繁变化、迁移和调优，DSL 和编译器就会变得重要。算子优化问的是“这一块怎么更快”，AI 编译器问的是“这一块怎么接到可运行、可部署的结果上”。

如果把这一章压缩成一句话，那就是：当 kernel 开始频繁遇到表达、优化和迁移问题时，DSL 和编译器就不再是可选项。attention 之所以一直被拿来讲，就是因为它最容易把“变化 -> 返工 -> 分层”这条线看清楚。

## 你还应该能判断什么

- 如果问题只服务一个固定实现，手写 kernel 可能够用
- 如果问题要覆盖多种 shape、多个后端或多种优化路径，DSL 更有价值
- 如果你主要在处理普通程序的语义和通用优化，传统编译器视角更合适
- 如果你主要在处理算子、布局、调度和后端映射，AI 编译器视角更合适
- 如果还要把调优和迁移变成可复用流程，编译器和 DSL 必须一起看
- 如果编译器无法兜底，最终仍然要靠 benchmark 和 profiler 说话
这些判断不要求你现在就做得很准，但至少应该知道自己是在看哪一层的问题。
只要能先把问题分层，后面的调优和迁移就不会完全凭感觉。
这也是第 1 章最实际的收获：不是把概念背下来，而是先知道哪一层该先动。最简单的决策树可以先记成这样：
- 固定、不会频繁变化的局部逻辑，先看 kernel
- 反复改 shape、布局或后端的同类问题，先上 DSL
- 需要把 DSL 变成可运行、可部署结果时，把 AI 编译器一起纳入视野

## 写作提醒

这一章不应该堆太多 API，也不需要太多代码。
它的任务是把概念边界和判断标准讲清楚，为后面的硬件、编译和优化章节铺路。
如果这一章写得对，读者会先看到“为什么问题会出现”，再看到“为什么 DSL 是合理答案”，而不是反过来先背定义。
编译器内部怎么展开，后面再慢慢学也不迟；这一章只需要知道它是一座桥。
