# 1.2 Kernel 与 DSL 的区别

## 定义边界

- `Kernel` 是可执行的计算单元
- `DSL` 是用来描述、生成或组织 kernel 的专用语言或抽象层

## 对比维度

### 1. 抽象层级

- Kernel 更接近硬件
- DSL 更接近问题定义
- Kernel 适合表达单次实现细节，DSL 适合表达一类问题的结构

### 2. 关注重点

- Kernel 关注线程、block、同步、内存、性能
- DSL 关注计算结构、分块方式、调度表达、可移植性
- Kernel 的目标是“把这一版跑快”
- DSL 的目标是“把这一类问题稳定地跑快”

### 3. 变化成本

- Kernel 的局部修改往往会牵动更多实现细节
- DSL 通常允许在更高层次上修改策略，再由编译器下沉到实现
- 当 shape、硬件或后端变化频繁时，DSL 的边际价值会上升得更快

## 这一区别的意义

如果把 DSL 和 kernel 混为一谈，就容易把“语言设计问题”误解成“只是换个写法”。  
实际上，DSL 的价值在于把某些优化决策从手工编码变成可表达、可分析、可自动化的对象。

## 判断标准

读到这里，应该能形成一个实用判断：

- 如果问题只服务一个固定 kernel，手写实现可能足够
- 如果问题要覆盖多种 shape 或多个后端，DSL 更有长期价值
- 如果后续还要调优、迁移、复用，DSL 能把一次性劳动变成可重复流程

## 常见误区

- 误区 1：DSL 只是更漂亮的语法
- 误区 2：DSL 会自动消灭所有性能问题
- 误区 3：只要能写出 kernel，就不需要 DSL

这三种理解都过于简化。  
真正有价值的 DSL，应该能把“要优化什么”变成可分析的结构，而不是只换一种写法。
