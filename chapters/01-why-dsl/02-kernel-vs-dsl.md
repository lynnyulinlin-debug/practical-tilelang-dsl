# 1.2 Kernel 与 DSL 的区别

## Kernel 与 DSL 的分工

- `Kernel` 是可执行的计算单元
- `DSL` 是用来描述、生成或组织 kernel 的专用语言或抽象层

如果先不用术语包住它们，可以先记成一句更直白的话：
kernel 负责“这一版怎么跑”，DSL 负责“这一类怎么写得更稳”。

### attention kernel 的变化场景

沿着上一节那个 attention kernel 的例子往下看，就会很清楚。
第一版你只是想把它写出来，第二版 head dimension 变了，第三版后端又换了。
这时最难受的地方不是“能不能再写一个 kernel”，而是每次变化都要重新碰同一批实现细节：tile 怎么切、访存怎么排、边界怎么写、同步怎么放。
如果你把这些东西都放在一个手写 kernel 里，变化一多，代码就会很快变成“看起来还能用，但谁都不敢轻易改”的状态。
这时候问题已经不是单个实现是否正确，而是这份实现能不能承受后续变化。
问题不在于这次算没算对，而在于下一次改的时候会不会又回到原点。
更现实一点说，每次面对的都不是“新问题”，而是“同一问题的不同版本”：输入形状变了、约束变了、后端变了，但要重新处理的仍然是那几个组织细节。

### kernel 在变化场景里为什么先吃力

这就是 kernel 和 DSL 开始分开的地方。
kernel 更擅长把“这一版”跑出来，DSL 更擅长把“这一类”固定成稳定的方法。
前者适合一次性实现，后者适合反复变化的场景。

还是拿 attention 来看：如果只是一次演示，手写 kernel 当然能做；如果同一个结构还要继续改 shape、换后端、调 tile，改动就不再只是局部细节，而会变成一整套实现策略的变化。
这意味着你每次改动都不只是“改几个参数”，而是在重新决定这段计算应该如何分块、如何复用数据、如何和硬件对齐。
一旦这些决定都被写死在 kernel 里，后面的调整就会越来越重。
这也是为什么 kernel 在短期内看起来很灵活，但一旦进入持续维护阶段，反而会越来越依赖经验和人工判断。
从阅读角度看，这里最重要的转折是：原本应该由“具体实现”解决的问题，开始逐渐变成“怎么组织这类实现”的问题。
一旦问题变成后者，单纯的 kernel 就开始不够用了。

### 为什么 DSL 更适合反复变化的场景

DSL 的价值就在这里：它先把一些优化决策放到结构层，再交给编译链路下沉到实现。
这样一来，变化不只是被写出来，而是可以先在更高一层组织好，再落到具体实现里。
比如同一个 attention 结构，DSL 可以先表达“这里要按某种 tile 组织”“这里要保留哪些中间状态”“这里要给后端留出优化空间”，而不必把所有细节一开始就写死。
这样后续真正变化的地方，就能先改结构，再改实现，而不是每次都从头拆一遍 kernel。
这样做的好处是，变化开始有层次了：结构先动，细节后动，很多重复工作就不会每次都从底层重来。
可以把它理解成：kernel 更像在写一份具体配方，DSL 更像在写一套可以反复改配方的框架。
前者对一次性任务很好用，后者对反复变化的任务更稳。

所以判断很简单：
只做一次固定实现时，手写 kernel 可能够用；要持续改 shape、换后端、追求可复用时，DSL 就更合适。
如果一个 DSL 只是换了写法，却不能帮助你更容易地调整 layout、tile 或后端策略，那它就还不够像一门工程 DSL。
