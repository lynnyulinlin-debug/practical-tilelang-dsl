# 第 1 章导言：Kernel 与 DSL：AI 编译器的动机

## English Title

From Kernel to DSL

## 章节定位

这一章负责建立整本教程的入口认知：为什么 AI kernel 不能只靠手写 CUDA，为什么需要 DSL，为什么还要把编译器放进来一起理解。

## 本章最小深度要求

读完这一章，应该能够回答三个问题：

- 为什么“能写 kernel”不等于“能维护 kernel”
- 为什么 AI kernel 的难点不只是计算，还包括表达、优化和迁移
- 为什么 TileLang 选择把 kernel、DSL 和编译器放在同一条链路里

## 本章目标

- 讲清 `kernel`、`DSL`、`compiler` 的边界
- 说明 AI 场景为什么放大抽象层的价值
- 给出 TileLang 在整套工具链里的位置
- 为后续硬件、调度和算子章节统一术语

## 本章判断标准

如果这章写得足够深，读者不仅知道“DSL 是什么”，还应该知道：

- 什么时候继续手写 kernel 更合适
- 什么时候应该把问题上提到 DSL 层
- 编译器在哪些地方帮忙，在哪些地方不会替你兜底
- TileLang 和 CUDA、Triton、TVM 的关系分别是什么

## 内容块安排

### 1.1 问题背景
- 为什么“能写 kernel”不等于“写得好”
- 为什么 AI 计算会反复碰到表达、调优、迁移三个问题
- 为什么这些问题在算子数量多、硬件变化快的项目里会被放大

### 1.2 Kernel 与 DSL
- Kernel 是执行单元
- DSL 是描述和组织 kernel 的方式
- 这里建立清晰的概念边界
- Kernel 关注“怎么执行”，DSL 关注“怎么描述和约束执行”
- 只有把表达层和执行层分开，后面才有编译和优化空间

### 1.3 AI 编译器的角色
- 编译器做什么
- 编译器不做什么
- 为什么 TileLang 需要依赖编译链路
- 编译器负责把高层意图转成可执行实现，同时保留优化空间
- 编译器不是万能调优器，它只是在约束内搜索更好的实现

### 1.4 TileLang 的位置
- 和 CUDA 的关系
- 和 Triton 的关系
- 和 TVM 的关系
- TileLang 更强调显式控制和跨硬件迁移
- 这章要让读者知道 TileLang 不是“另一个 kernel 语法”，而是一个工程组织方式
