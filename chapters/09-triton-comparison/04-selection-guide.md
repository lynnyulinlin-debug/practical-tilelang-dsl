# 9.4 选型指南

## Triton 与 TileLang 的选型判断

这一节不再讲原理，而是回答“什么场景选谁”。

这也是整章最接近工程决策的一节：不要追求“最优答案”，而要追求“在当前约束下最合适的答案”。

这一节不再关心抽象差异本身，而是关心这些差异如何映射到项目约束上。  
只要约束说清楚，选型就不再是偏好问题。

这一节真正要交出的，不是一张推荐表，而是一条稳定的判断路径。  
只要路径成立，选型就不会被热门程度、个人偏好或者概念印象带偏。

### 1. 适合选 Triton 的场景

- 在 NVIDIA 上快速验证新算法
- 需求更偏开发效率
- 以常见算子为主

### 2. 适合选 TileLang 的场景

- 需要跨硬件适配
- 需要更强的显式控制
- 需要在性能和可迁移性之间做结构化优化

### 3. 选型时的判断顺序

- 先看目标平台
- 再看算子复杂度
- 再看团队对硬件和调度的控制需求

这个顺序很重要，因为它把“平台”放回第一位，把“喜欢哪条路线”放到最后。  
如果顺序反了，选型就会变成先看手感，再找理由。

这几步其实是在把“平台、目标、复杂度、迁移需求、控制需求”逐层排开。  
如果前面几步已经决定了答案，后面就不要再用手感去推翻它。

### 4. Triton / TileLang 的选型判断标准

- 先给判断顺序，再给建议
- 不要把推荐写成绝对结论
- 选型的重点是约束匹配，不是品牌偏好

这条规则的意思很明确：先解决当下问题，再考虑长期扩展，最后才考虑个人体验。  
工程选型如果倒过来，通常就会偏离项目目标。

选型顺序其实是在把“最重要的约束”摆到最前面。  
如果团队今天最缺的是验证速度，就不要先追求最强控制；如果今天最缺的是迁移能力，就不要只盯着局部开发效率。

先选能最快验证目标的路线，再选能最稳扩展到目标平台的路线，最后才把“写起来顺不顺手”作为微调因素。

## Triton / TileLang 的选型判断

选型不是“谁更好”，而是“哪条路线和当前约束更匹配”。
这一句就是本章最重要的落点。

这一节可以收束为：先看目标，再看约束，再看路线。  
一旦这三步走顺了，选型就不再是争论题，而是一个可以重复的工程判断。

## 选型常见误区

- 误区 1：选型靠个人偏好
- 误区 2：只有一种正确答案
- 误区 3：平台、目标、团队经验可以只看其中一个

选型的关键，是把约束摆出来，再看哪条路线最匹配。

选型判断表应该直接写成四个条件：

- 平台
- 目标
- 团队经验
- 可迁移性

把这四项写实，决策树才能给出稳定建议。
