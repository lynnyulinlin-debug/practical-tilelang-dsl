# 9.3 自动优化 vs 显式控制

## 要解决什么问题

这一节比较 TileLang 和 Triton 的核心差异，尤其是“谁控制什么”。

它不是在做阵营对立，而是在讲两条路线各自把复杂度放在哪里。

换句话说，这一节要回答的是：复杂度是被编译器吞掉了，还是被开发者显式拿在手里。  
这个差别，会直接影响调优方式、迁移能力和教学表达。

## 要讲什么

### 1. Triton 偏自动

- 更高层的抽象
- 编译器处理更多实现细节
- 写法更紧凑
- 更适合先把 kernel 跑起来，再慢慢看性能

### 2. TileLang 偏显式

- 开发者显式控制更多内存层次
- 调度和布局可见
- 更适合细粒度优化和跨硬件迁移
- 更适合把内存流、计算流和调度流拆开看

这并不意味着 TileLang 一定“更难用”，而是它把控制边界放得更靠前。  
一旦控制边界显式化，读者就更容易在迁移、调度和内存布局上做结构化优化。

### 3. 差异带来的后果

- Triton 更快进入实验
- TileLang 更容易做结构化调优
- 两者的优势不在同一维度

这里最重要的判断，是不要拿“开发效率”和“结构控制”去做单维度比较。  
它们本来就不在同一条轴上，所以更合理的问法是：当前问题更需要哪一侧的能力。

### 4. 该怎么看这个差异

- 如果你要快速试算子，先看 Triton 的开发效率
- 如果你要跨平台迁移或压榨硬件，先看 TileLang 的控制边界
- 如果你要教学，TileLang 更容易把“哪里可控”讲清楚

### 4. 当前写作口径

- 先对比控制边界，再对比适用场景
- 不要把“自动优化”和“显式控制”写成价值判断
- 这节的目标是帮助读者建立选择标准

### 5. 一个更实用的判断

```text
need quick experiment?
  -> Triton
need cross-platform migration?
  -> TileLang
need explicit layout / schedule control?
  -> TileLang
need minimal code to verify a hot path?
  -> Triton
```

## 小结

自动优化和显式控制不是对错关系，而是不同的工程策略。
读者真正需要的是根据目标平台、团队能力和迁移需求来选路线。

如果把这一节压成一句话，那就是：Triton 把更多实现复杂度交给编译器，TileLang 把更多实现复杂度交给开发者。  
谁更适合，不看抽象名义，看当前问题更需要哪种复杂度分配。

## 这一节可以进一步追问的问题

- 哪些复杂度被 Triton 处理掉了，哪些留给开发者
- 哪些复杂度被 TileLang 显式暴露了，为什么这有价值
- 为什么两条路线的复杂度分布不同

## 常见误区

- 误区 1：自动优化一定更好
- 误区 2：显式控制一定更强
- 误区 3：两条路线只是编码风格不同

这节真正要讲的，是复杂度放在谁手里、放在哪一层。

## 讲义提示

这一节最适合用“问题 -> 路线 -> 代价”的方式来讲：

- 问题是什么
- 哪条路线更快解决
- 代价是什么

这样读者不会把它看成抽象哲学，而会把它看成工程决策表。
