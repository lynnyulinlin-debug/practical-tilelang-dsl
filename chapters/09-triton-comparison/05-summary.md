# 9.5 本章总结

## 你应该记住什么

1. Triton 代表的是高层、快速、偏自动的 DSL 路线
2. TileLang 代表的是更显式、更可控、适合迁移和深度调优的路线
3. 两者的差异主要体现在抽象层级、内存控制和适用场景
4. 选型应该先看约束，再看偏好
5. 不要把两者写成阵营对立，应该写成不同工程目标下的不同解法

## 进入下一章之前

下一章会把前面这些方法落到端到端工程实战里，看一套算子和迁移方法如何真正服务一个大模型相关场景。

## 写作提醒

这一章最适合的收束方式，是把“路线差异”转成“场景选择”。  
不要停在概念对比，要把读者带到可以做判断的位置。

## 本章最小闭环

如果只保留一个最小闭环，这一章至少要做到：

1. 读者知道 Triton 的 tile 视角
2. 读者知道 TileLang 的显式控制边界
3. 读者知道什么时候优先选 Triton
4. 读者知道什么时候优先选 TileLang

这四点比完整的特性清单更重要，因为它们直接决定读者能不能做出工程选择。

## 你还应该能判断什么

- 哪条路线更适合快速验证
- 哪条路线更适合跨平台迁移
- 哪条路线更适合显式调度控制
- 选型时应该先看平台、目标还是团队能力

## 常见误区

- 误区 1：Triton 和 TileLang 是阵营对立
- 误区 2：选型只看性能数字
- 误区 3：如果一条路线更热门，就一定更适合当前问题

这一章的收束应该把读者带到“我现在该选谁”这个问题上，而不是停在概念对比上。
