# 3.5 常见错误与排错

## 要说明的问题

初学者最常见的阻塞不是“不会写”，而是“跑不起来”。

## 要点

- 环境缺失
- 后端不可用
- shape 不匹配
- dtype 不一致
- 结果误差过大
- 排错顺序要和错误层级对应

## 当前仓库优先级

- 先排查 `import tvm` 是否成功
- 再确认是否有可用 CUDA / NVIDIA 驱动
- 如果只有 CPU 环境，就先验证 CPU fallback 路径

## 这里应该配的占位

**表：常见报错与处理方式**

> 这里要把错误类型和处理建议对应起来，尽量让读者能快速定位问题属于安装、编译还是代码逻辑。
> 当前仓库里最常见的两类是：
> - `ModuleNotFoundError: No module named 'tvm'`
> - `CUDA not available` / NVIDIA 驱动不可用

## 这里应该解释的现象

- 为什么很多错误是环境错误而不是算法错误
- 为什么要把报错信息读完整
- 为什么排错是学习流程的一部分
- 为什么“先缩小问题再扩大问题”比盲查更有效

## 这一节可以进一步追问的问题

- 这条报错是安装阶段、编译阶段还是运行阶段
- 这个错误是输入约束不满足，还是后端能力不满足
- 有没有一个更小的输入规模可以先验证路径

## 这一节应该留下的排错原则

- 先确认依赖，再确认后端
- 先确认能否 import，再确认能否 build
- 先确认结果是否正确，再确认性能是否合理
- 先缩小输入规模，再追查更深层的问题
- 先看报错属于哪一层，再决定是改环境、改编译还是改代码

## 常见误区

- 误区 1：报错信息可以忽略
- 误区 2：一个错误只可能有一个原因
- 误区 3：编译成功就说明运行逻辑没有问题

如果把顺序搞反，初学者很容易在一个无关紧要的地方卡很久。
