# 3.5 常见错误与排错

## 排错和问题分层

初学者最常见的阻塞不是“不会写”，而是“跑不起来”。

所以这节不是附录，而是学习路径的一部分。
它要把“报错”从坏消息变成可定位信号，避免把所有问题都混成一团。

它和第 5 章的共同点，是都要求先判断、再行动；区别只在于这里判断的是“为什么跑不起来”，第 5 章判断的是“为什么跑不快”。

## 排错要抓住的判断

- 环境缺失
- 后端不可用
- shape 不匹配
- dtype 不一致
- 结果误差过大
- 排错顺序要和错误层级对应
- 很多错误一开始看起来像代码问题，实际先是环境或后端问题

## 当前仓库优先级

- 先排查 `import tvm` 是否成功
- 再确认是否有可用 CUDA / NVIDIA 驱动
- 如果只有 CPU 环境，就先验证 CPU fallback 路径

## 排错的常见现象

- 为什么很多错误是环境错误而不是算法错误
- 为什么要把报错信息读完整
- 为什么排错是学习流程的一部分
- 为什么“先缩小问题再扩大问题”比盲查更有效

在实际排错里，最常见的顺序是：
1. 先确认是不是依赖没装全或版本不兼容
2. 再确认是不是后端不可用，比如 GPU 驱动或 target 没准备好
3. 如果能编译但结果不对，再去看 shape、dtype、边界和索引

这不是死板流程，而是一种节省时间的优先级。越靠前的问题，越容易一眼排掉；越靠后的问题，越接近代码逻辑本身。

## 排错的原则

- 先确认依赖，再确认后端
- 先确认能否 import，再确认能否 build
- 先确认结果是否正确，再确认性能是否合理
- 先缩小输入规模，再追查更深层的问题
- 先看报错属于哪一层，再决定是改环境、改编译还是改代码

这些原则本质上也在给第 5 章铺路：先判断瓶颈属于哪一层，再决定要不要继续搜索或优化。

如果报错发生在 `import` 或依赖加载阶段，先看环境；如果发生在 `build` 阶段，先看 target 和后端支持；如果能跑但结果不对，再看 shape、dtype 和输入输出约束。顺序搞对了，很多“无从下手”的问题都会变成“该先查哪一层”的问题。

如果还要继续追问，先判断报错发生在安装、编译还是运行阶段，再决定是改环境、改后端还是改代码。

## 排错的常见误区

- 误区 1：报错信息可以忽略
- 误区 2：一个错误只可能有一个原因
- 误区 3：编译成功就说明运行逻辑没有问题

如果把顺序搞反，初学者很容易在一个无关紧要的地方卡很久。

所以这一节的价值，不只是解决报错，而是把排错节奏固定下来。

对初学者来说，这一节最实用的收获是：
报错不是坏消息，而是提示应该先查哪一层。

等后面的报错对照表补上，这一节就会从“经验说明”变成“查表路径”。
