# 第 3 章导言：TileLang 快速入门与第一个 Kernel

## 章节定位

这一章负责建立“可跑通”的最小闭环。  
它的故事不是“学习一个新语法”，而是“把第一次失败、第一次编译通过、第一次跑出正确结果的过程走完整”。  
目标不是把编译器讲深，而是让读者第一次写出、编译出、运行出一个 kernel。

如果第 1 章已经回答了“为什么要分层”，第 2 章回答了“层次怎么落在硬件上”，那第 3 章要做的就是把这些理解变成一次真正跑通的结果。

如果你是按最短路径读这套教程，这一章也很适合作为第 1 章之后的第二站：先把“能跑通、能验证、能排错”的手感建立起来，再回头补第 2 章的硬件层次，会更容易把前面的抽象和后面的调优接起来。

这一章还有一个更现实的作用：把“学习教程”变成“排错流程”。
如果读者知道先看环境、再看编译、最后看结果，那么后面遇到的很多问题都不会停留在情绪层，而是能直接定位到具体环节。

在 TileLang 里，这一章也第一次把编译器的作用讲实：你写下来的不是最终机器码，而是会先经过中间表示、lowering 和 backend 选择的一条链路。这里的 lowering，简单说就是把较高层的表达一层层压低到更接近目标硬件的表示上。编译器在这里的职责，不是替你猜结果，而是把 DSL 里已经表达清楚的结构和约束，稳定地传到后端，让读者看到“写法”和“落地”之间到底隔着什么。TVM 在这里扮演的就是这条链路的运行和验证底座，所以这一章先把环境、编译和结果跑通，再谈后面的分块和优化会更顺。

这也是它和第 5 章的关系：第 5 章讨论“已经写对之后怎么调快”，第 3 章先讨论“怎么确认自己真的写对了”。
如果第 3 章没有把这个起点立稳，后面的调优只会缺少可靠基线。

对初学者来说，这一章最重要的是建立信心：
你现在就能先跑起来，而不是必须先把整套编译体系学完。

## 本章目标

- 完成环境准备
- 跑通第一个 Vector Add
- 看懂 DSL 到目标代码的最小编译路径
- 形成结果验证和排错习惯
- 把“能跑”与“能判断”分开

这几项看起来很基础，但它们其实是在搭一条最小工程链：
环境先成立，编译链路再成立，结果验证最后成立，读者才能继续往后面的分块和性能优化走。

这一章也会先碰到几个最常用的 TileLang 原语：`T.alloc`、`T.alloc_shared`、`T.alloc_fragment` 和 `T.gemm`。它们不是一次性把所有 API 讲完，而是先让读者知道“数据落点、片上复用、中间态和计算入口”分别长什么样。

如果把它们放在一起看，读者其实只需要记住一个最小序列：先分配位置，再决定是否片上复用，再把中间结果放到合适的容器里，最后交给计算入口去执行。这个序列会贯穿后面的分块和调优章节。

如果你在别处看到 `tile.copy` 这类说法，可以先把它理解成“数据搬运或复制路径”的一种抽象，但本仓库当前教程主线并没有把它当作独立入口展开；更靠前的入口还是这几个基础原语，后面第 4 章会把它们和硬件层次一一对上。

## Python / PyTorch / TileLang 的关系

这一章里，Python 主要是教程和脚本的承载语言，负责把示例写出来、跑起来、测出来。  
PyTorch 主要是参考实现和结果对照的来源，负责告诉你“正确答案大概应该长什么样”。  
TileLang 则是你真正要学习的表达层：它负责把算子、调度和硬件映射放到同一条路径里。

对初学者来说，最稳的理解顺序是：
- 先用 Python 把教程跑通
- 再用 PyTorch 或 NumPy 当参考答案
- 最后把 TileLang 当成需要掌握的 DSL

这样读者就不会把“写教程的语言”“对照结果的工具”和“真正要学的表达层”混在一起。

在真正进入第一个 kernel 之前，读者还需要先认识最小的张量入口：shape、dtype、输入输出和最简单的逐元素操作。这里不打算展开成张量库教程，只是先保证读者能看懂“一个张量长什么样、怎么被传进去、怎么被拿出来验证”。

换句话说，这一章先不要求你掌握完整的张量编程，只要求你能认出最小可跑路径里的四个东西：输入长什么样、输出长什么样、计算对每个元素做了什么、结果该怎么比对。

## 本章判断标准

读完这一章，读者应该能独立判断：

- 一个 kernel 是代码问题、编译问题还是环境问题
- 结果验证和性能验证为什么不能混在一起
- `target`、shape、dtype 三者分别影响什么

如果做不到这些判断，后面即使能照着写，也很难知道自己是在修环境、修表达，还是修后端。
