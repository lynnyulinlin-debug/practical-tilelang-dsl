# 第 3 章导言：TileLang 快速入门与第一个 Kernel

## 章节定位

这一章负责建立“可跑通”的最小闭环。  
目标不是把编译器讲深，而是让读者第一次写出、编译出、运行出一个 kernel。

## 本章目标

- 完成环境准备
- 跑通第一个 Vector Add
- 看懂 DSL 到目标代码的最小编译路径
- 形成结果验证和排错习惯
- 把“能跑”与“能判断”分开

## 本章判断标准

读完这一章，读者应该能独立判断：

- 一个 kernel 是代码问题、编译问题还是环境问题
- 结果验证和性能验证为什么不能混在一起
- `target`、shape、dtype 三者分别影响什么

## 当前可执行口径

- 本仓库当前已确认 `tilelang-tvm` conda 环境可用
- `examples/03_vector_add.py` 和 `examples/04_gemm_basic.py` 已能在 CPU fallback 下跑通
- 当前机器没有可用 NVIDIA 驱动时，GPU 路径会被跳过，而不是强行失败
- 这一章的最低目标不是“会写很多 kernel”，而是“能独立判断一个 kernel 是编译问题、运行问题还是结果问题”
- notebook 版适合逐格讲解，脚本版适合验证和复现

## 本章最小深度要求

这一章至少要让读者回答三个问题：

1. 代码从 DSL 到后端代码到底经历了什么
2. 正确性验证和性能测量为什么要分开
3. 失败时先看环境、编译还是 shape / dtype

如果这三个问题答不出来，这章还只是“看过”，没有到“会用”。
