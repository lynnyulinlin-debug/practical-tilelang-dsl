# 3.4 运行与验证

## 要说明的问题

一个 kernel 能编译出来不代表它正确。  
这一节要说明怎么确认结果是对的，性能是否有基本预期。

## 要点

- 如何准备输入数据
- 如何调用 kernel
- 如何把结果拉回主机侧
- 如何和 NumPy / PyTorch 结果对照
- 如何把“正确性验证”和“性能测试”拆开执行

## 当前仓库的验证方式

- `examples/03_vector_add.py` 已在 CPU fallback 路径上通过正确性验证
- `examples/04_gemm_basic.py` 已在 CPU fallback 路径上完成最小验证
- GPU 正确性和性能验证需要可用 NVIDIA 驱动，当前机器不满足这个前提
- 这意味着“先跑通 CPU，再补 GPU”是当前仓库最稳的学习节奏

## 这里应该放的占位

**表：验证流程清单**

> 这里要列出“输入准备、执行、结果比对、误差阈值、性能测量”的完整验证顺序。
> 对当前仓库，建议明确区分：
> - CPU 验证：确认接口、编译和结果正确
> - GPU 验证：确认后端代码和性能数据
> - 没有 GPU 驱动：只做 CPU 验证，不强行继续

## 这里应该解释的现象

- 为什么结果验证和性能测量必须分开
- 为什么先做正确性，再做性能
- 为什么初学阶段误差阈值比绝对性能更重要
- 为什么 CPU fallback 是很重要的学习路径

## 这一节可以进一步追问的问题

- 验证时比对的是哪一层输出
- 误差是来自数据类型、边界处理还是实现本身
- 如果只换 target，不改代码，结果为什么仍可能变化

## 这一节应该留下的检查顺序

1. 输入 shape 是否对
2. dtype 是否对
3. 输出 shape 是否对
4. 数值误差是否在可接受范围
5. 再看性能是否符合预期

## 常见误区

- 误区 1：能跑就等于正确
- 误区 2：性能低一定是优化不够
- 误区 3：只要结果不报错，验证就完成了

正确的顺序应该是先把接口和数值对齐，再去谈吞吐和峰值利用率。

这个顺序很重要，因为它能把很多“看起来像性能问题”的错误先归到正确性层面。
