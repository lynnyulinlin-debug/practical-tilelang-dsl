# 3.4 运行与验证

## 运行与验证的分工

一个 kernel 能编译出来不代表它正确。  
这一节要说明怎么确认结果是对的，性能是否有基本预期。

这一步其实是在把“能跑”拆成两个层次：
先确认数值正确，再确认性能合理。
如果顺序反过来，很多实际错误都会被误判成性能问题。

## 运行与验证要抓住的判断

- 如何准备输入数据
- 如何调用 kernel
- 如何把结果拉回主机侧
- 如何和 NumPy / PyTorch 结果对照
- 如何把“正确性验证”和“性能测试”拆开执行
- 对初学者来说，先确认 shape 和 dtype，再看数值是否落在可接受误差里

这里的 PyTorch 只扮演参考答案的角色，不是这一章的主线对象。它的价值在于帮你确认“TileLang 跑出来的结果和已知正确实现是不是一致”，而不是让你先学一遍 PyTorch 再回来学 TileLang。

## 当前仓库的验证方式

- `examples/03_vector_add.py` 已在 CPU fallback 路径上通过正确性验证
- `examples/04_gemm_basic.py` 已在 CPU fallback 路径上完成最小验证
- GPU 正确性和性能验证需要可用 NVIDIA 驱动，当前机器不满足这个前提
- 这意味着“先跑通 CPU，再补 GPU”是当前仓库最稳的学习节奏

## 运行与验证的常见现象

- 为什么结果验证和性能测量必须分开
- 为什么先做正确性，再做性能
- 为什么初学阶段误差阈值比绝对性能更重要
- 为什么 CPU fallback 是很重要的学习路径

一个比较稳的做法是先准备一份“参考答案”，比如 NumPy 或 PyTorch 的结果，再让 TileLang 的输出和它逐元素对照。对于最小例子，这个比对通常只需要确认数值是否在容忍范围内；对于更复杂的 kernel，才需要再进一步看误差分布、边界项和 dtype 带来的偏差。

这样做的价值，是把“结果不对”拆成更小的子问题：是输入数据准备错了，还是 kernel 本身写错了，还是只是数值精度不同。初学阶段最怕的就是把这三件事混在一起。

## 运行验证的检查顺序

1. 输入 shape 是否对
2. dtype 是否对
3. 输出 shape 是否对
4. 数值误差是否在可接受范围
5. 再看性能是否符合预期

这五步就是这一节的真正骨架。
它的作用，是让验证先服务正确性，再服务性能分析。

如果是 float32，一般能直接做逐元素对照；如果是 lower precision，就更要先确认比较基线和误差阈值，而不是拿一个“看起来差不多”的结果去判断成功。验证的意义不是证明“它很快”，而是先证明“它真的是对的”。

对初学者来说，这一节最重要的习惯是：
先确认结果对，再去看快不快。

## 运行验证的常见误区

- 误区 1：能跑就等于正确
- 误区 2：性能低一定是优化不够
- 误区 3：只要结果不报错，验证就完成了

正确的顺序应该是先把接口和数值对齐，再去谈吞吐和峰值利用率。

这个顺序很重要，因为它能把很多“看起来像性能问题”的错误先归到正确性层面。

一旦这个顺序固定下来，后面做 benchmark 时就不会把未验证的结果当成优化成果。

如果还要继续追问，重点应该放在比对的是哪一层输出、误差来自数据类型还是边界处理，而不是先把性能当成主要问题。
