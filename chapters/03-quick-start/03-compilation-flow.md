# 3.3 编译流程

## 编译链路和中间层

不仅要会写 kernel，还要知道编译器在中间做了什么。

这一步很重要，因为很多初学者会把编译器看成黑盒，结果一旦出错就只能盲试参数。
如果知道中间经历了哪些阶段，至少能判断问题更像出现在表达、 lowering 还是 backend 选择上。

## 编译链路要抓住的判断

- 高层表达如何进入 IR
- TIR / lowering 的位置
- target 如何影响最终生成结果
- 为什么生成代码可能是 PTX、ROCm code 或其它后端产物
- 编译链路里每一步都可能引入约束，也可能保留优化空间
- 先抓住一条线：表达先变成中间表示，再由 target 决定落到哪种后端

如果只需要一个最小的 PTX 直觉，可以先把它理解成“NVIDIA 后端的中间代码表示”。它不是最终机器指令，但已经比高层 DSL 更接近硬件；很多编译和性能判断，都会先在这层看形状、布局和资源约束有没有被保留下来。先知道 PTX 在链路里的位置就够了，不需要在这一章里钻进指令细节。

## 当前实现口径

- 本仓库示例里，TE 先通过 `create_prim_func` 降到 `PrimFunc`
- 再交给 `tvm.build(...)` 生成目标后端代码
- 在当前 `apache-tvm 0.25` 环境里，这条路径已被 `examples/03_vector_add.py` 验证
- 这条路径的关键不是 API 名字，而是“先定义计算，再交给后端决定怎么实现”

如果把这条路径再拆细一点，可以先把它理解成三步：
1. 把“我要算什么”变成一个中间表达；
2. 把中间表达压到目标后端能接受的形式；
3. 让后端把它落成真正可执行的代码。

这三步里，前两步决定的是“语义怎么被保留和限制”，最后一步决定的是“结果到底跑在哪儿、怎么跑”。这也是为什么同一个 kernel，只是换一个 target，最终产物就可能完全不同。

## 编译链路的常见现象

- 为什么 DSL 不是直接执行
- 为什么编译器会自动补齐一部分细节
- 为什么不同 target 生成不同结果
- 为什么同一份高层表达在不同后端上的表现会不同

这一节真正要建立的是“中间层思维”：
不是只看源代码和最终代码，而是把 IR / lowering / target 这些中间层当成判断问题的依据。

如果你调的是 shape、dtype 或 target，问题通常不会只停在“写法对不对”这一层；它会一路影响 IR 约束、lowering 选择和后端代码形态。先把这条链看懂，后面遇到 build 失败或性能异常时，才知道该从哪一层回看。

举个最简单的理解方式：如果只改了一个输入 shape，程序未必会“逻辑错误”，但编译器可能会因为 shape 不再满足某些假设而改写中间层；如果再把 target 从 CPU 换成 GPU，最终代码的生成路径还会再变一次。这一节要说明：代码层面的小改动，往往会沿着编译链路被放大成不同的后端行为。

对初学者来说，这一节要解决的其实只有一个问题：
为什么“写出来”到“跑起来”之间，还隔着一层需要理解的东西。

## 编译链路的判断标准

- 如果你改的是 `target`，通常改变的是后端代码生成路径
- 如果你改的是 shape / dtype，通常改变的是 IR 约束和运行前提
- 如果你改的是计算表达，通常改变的是编译器看到的语义本身
- 如果你不清楚改动落在哪一层，就很难判断问题应该怎么排

如果还要继续追问，优先区分改动的是表达、生成目标还是后端代码，这比直接盯着结果更容易定位问题。

## 编译链路的常见误区

- 误区 1：编译器只是机械翻译
- 误区 2：只要源码一样，不同 target 的结果就一定一样
- 误区 3：所有性能差异都发生在运行阶段

实际上，很多性能和可用性差异在 lowering 之前就已经埋下了。

理解这三者的差异，后面排错会快很多。
