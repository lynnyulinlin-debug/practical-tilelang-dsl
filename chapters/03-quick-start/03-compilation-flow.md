# 3.3 编译流程

## 要说明的问题

读者不仅要会写 kernel，还要知道编译器在中间做了什么。

## 要点

- 高层表达如何进入 IR
- TIR / lowering 的位置
- target 如何影响最终生成结果
- 为什么生成代码可能是 PTX、ROCm code 或其它后端产物
- 编译链路里每一步都可能引入约束，也可能保留优化空间

## 当前实现口径

- 本仓库示例里，TE 先通过 `create_prim_func` 降到 `PrimFunc`
- 再交给 `tvm.build(...)` 生成目标后端代码
- 在当前 `apache-tvm 0.25` 环境里，这条路径已被 `examples/03_vector_add.py` 验证
- 这条路径的关键不是 API 名字，而是“先定义计算，再交给后端决定怎么实现”

## 这里应该解释的现象

- 为什么 DSL 不是直接执行
- 为什么编译器会自动补齐一部分细节
- 为什么不同 target 生成不同结果
- 为什么同一份高层表达在不同后端上的表现会不同

## 这一节可以进一步追问的问题

- 你改的是表达，还是改的是生成目标
- 你看到的是 IR，还是最终后端代码
- 你遇到的是编译失败，还是运行时行为变化

## 这一节应该留下的判断标准

- 如果你改的是 `target`，通常改变的是后端代码生成路径
- 如果你改的是 shape / dtype，通常改变的是 IR 约束和运行前提
- 如果你改的是计算表达，通常改变的是编译器看到的语义本身
- 如果你不清楚改动落在哪一层，就很难判断问题应该怎么排

## 常见误区

- 误区 1：编译器只是机械翻译
- 误区 2：只要源码一样，不同 target 的结果就一定一样
- 误区 3：所有性能差异都发生在运行阶段

实际上，很多性能和可用性差异在 lowering 之前就已经埋下了。

理解这三者的差异，后面排错会快很多。
