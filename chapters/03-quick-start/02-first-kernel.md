# 3.2 第一个 Kernel

## 要说明的问题

读者需要先看到一个最小可运行例子，才能把前面的概念和实际写法联系起来。

在这之前，最好先能认出最基本的张量形状和输入输出关系。只要知道一个张量大概长什么样、怎么传进去、怎么拿出来比对，后面的第一个 kernel 就不会突然变成陌生的语法题。

这一节在结构上和第 5 章很像：先给一个可判断的基准点，再去谈后续变化。
如果没有这个基准点，后面的编译和验证就很难有统一参照。

## 要点

- 计算定义是什么
- 输入输出怎么表达
- kernel 怎么编译
- kernel 怎么执行
- 最小例子要能同时说明“定义、编译、执行、验证”四件事
- 先把逐元素映射、shape 对齐和 dtype 一致这三件事看懂
- 再把这个最小例子当成后面 GEMM、分块和调优的基准模板

## 这里应该解释的现象

- 为什么先用最简单算子
- 为什么这个例子足够说明编译流程
- 为什么它是后续复杂算子的基准模板
- 为什么最小例子比长代码更适合初学者建立心智模型

最小例子的好处，是把变量压到最少。
变量越少，读者越容易看清楚“定义、编译、执行、验证”到底各自发生了什么。

在这个阶段，读者不需要理解复杂的张量调度，只需要知道：
- 输入和输出的 shape 是否一致
- 计算是不是对每个元素做同样的映射
- dtype 会不会影响结果比对
- 结果拿回主机后应该和哪个参考答案比较

如果把这个例子再说白一点，它其实只是在做一件事：让读者看见“输入张量里的一个位置，最后怎么对应到输出张量里的一个位置”。这一步看似简单，但它是后面所有分块、重排和融合的基础，因为只要这个对应关系不稳，后面的优化就很难判断自己改对了没有。

所以这一节不需要广播、不需要切片语法大全，也不需要复杂的控制流。它只要把最基础的元素级映射讲清楚，就已经完成使命了。

对初学者来说，这一节最重要的是先拿到一个“我真的跑起来了”的结果。
有了这个结果，后面再看编译链路就不容易发虚。

## 这一节应该留下的判断标准

- 如果一个 kernel 连最小闭环都跑不通，先不要谈优化
- 如果一个例子不能说明输入、输出和验证，先不要谈复杂调度
- 如果读者看完后还不知道结果该怎么比对，这一节还不够

这三条标准和第 5 章的收口逻辑是一致的：先立住基本结构，再讨论优化空间。
