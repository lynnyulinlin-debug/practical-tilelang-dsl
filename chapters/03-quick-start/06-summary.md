# 3.6 本章总结

## 你应该记住什么

1. 先把环境跑通，再看更深的优化
2. 第一个 kernel 的作用是建立最小闭环
3. 编译流程是理解 TileLang 的关键入口
4. 正确性验证和排错是写 kernel 的固定步骤
5. 先判断问题属于环境、编译还是实现，再决定下一步怎么查

如果只能记住一条，这一条就够了：**先让程序稳定跑起来，再问它为什么这样跑、还能不能跑得更快。**  
这句话看起来简单，但它其实就是后面所有章节的共同前提。

对初学者来说，这一章最后要形成的不是“我学会了多少 API”，而是“我现在知道第一步该看什么”。只要这个顺序建立起来，后面的分块、调度和多后端适配就不会再像纯概念堆叠。

对初学者来说，这一章的收口很简单：
你已经知道怎么开始了，也知道哪里出问题时该往哪找。

## 你还应该能判断什么

- 一个问题是环境问题、编译问题还是实现问题
- 正确性验证和性能验证应该在什么顺序下做
- CPU fallback 为什么不是“次等方案”，而是重要的学习路径
- 哪些错误应该先缩小输入规模再查

## 进入下一章之前

下一章会进入分块和内存层次，开始解释为什么第一个 kernel 之后必须认真处理数据复用和内存访问。

## 写作提醒

这一章要尽量简洁，保持“能上手”的感觉。  
真正的性能和结构优化，会在第 4、5 章展开。  
但这一章已经要让读者建立起最小判断能力，而不是只会照着跑。  
这也是这一章最关键的教育目标：让读者第一次把“跑通”理解成一个有顺序、有层次、可排错的工程动作，而不是只把它当成“命令能执行”。
