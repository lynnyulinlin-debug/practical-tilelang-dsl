# 3.1 环境准备

## 环境准备和最小可执行路径

教程不是在抽象环境里运行的。  
如果环境不明确，就会在还没理解 kernel 之前先卡住。

这一节真正要解决的，不是“环境怎么装”，而是“最小可执行路径是什么”。
只要路径清楚，就能先把 CPU fallback 跑通，再决定要不要切到 GPU。

## 环境准备要抓住的判断

- 需要什么基础依赖
- 需要什么 Python 环境
- 需要什么后端支持
- 没有目标硬件时怎么做回退
- 先确认“能不能跑”，再考虑“跑得快不快”

## 当前仓库的实际口径

- 推荐使用单独的 `tilelang-tvm` conda 环境
- 只验证 CPU 路径时，`pip install apache-tvm` 已足够
- 需要 CUDA / GPU 路径时，再切换到支持 CUDA 的 TVM 包
- 当前机器若没有可用 NVIDIA 驱动，GPU 示例会被跳过
- 本章的环境目标不是“统一所有机器”，而是给出清晰的最小可执行路径

这个口径的价值在于，它把环境判断从“能不能完全一致”改成“能不能先启动”。
这和第 5 章的调优思路一致：先把能控制的部分做成闭环，再看更复杂的部分。

最实用的理解方式是把环境拆成三层：
- Python 层：能不能导入依赖、跑脚本
- TVM 层：能不能完成 build 和后端选择
- 设备层：CPU fallback 或 GPU 驱动是否真的可用

只要这三层都能明确，后面遇到问题时就不会把“装不上”“编不过”和“跑不对”混成一类。

**表：环境依赖与最低要求**

> 这里要说明 Python、TVM、目标后端、驱动版本等依赖的关系，并区分“必须项”和“可选项”。
> 对当前仓库，至少要说明：
> - CPU 验证：Python + `apache-tvm`
> - GPU 验证：Python + CUDA 版 TVM + 可用 NVIDIA 驱动
> - 没有驱动时：只跑 CPU fallback，不阻塞文档阅读

## 环境准备的常见现象

- 为什么同一份代码在不同环境下行为不同
- 为什么先检查安装，再看代码
- 为什么教程必须给出 CPU fallback
- 为什么“环境不同”经常会被误判成“代码有问题”

## 环境准备的判断标准

- 如果 `import tvm` 都过不了，先不要看 kernel 代码
- 如果只有 CPU 环境，就先走 CPU fallback，把接口和验证跑通
- 如果目标是 GPU，再去确认驱动、CUDA 包和后端支持

这三条顺序也很重要：
先确认依赖，再确认后端，最后才进入性能验证。

## 环境准备的常见误区

- 误区 1：环境配置只是安装步骤
- 误区 2：同一份教程必须要求所有机器完全一致
- 误区 3：没有 GPU 就无法学习这一章

实际上，好的教学路径应该允许先在 CPU 上建立心智模型，再逐步过渡到 GPU。

所以这一节最终要交付的，不是某一种安装组合，而是“先能跑，再谈跑得快”的默认顺序。

对初学者来说，这一节学会一个就够了：
如果当前机器只能先跑 CPU，也完全可以先把路线走通。

后面如果把表补上，就能把“必须项 / 可选项 / 回退路径”一眼看清。
