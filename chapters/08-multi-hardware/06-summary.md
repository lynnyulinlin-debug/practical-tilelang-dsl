# 8.6 本章总结

## 你应该记住什么

1. 多后端的核心是逻辑统一、参数分平台调
2. NVIDIA、ROCm、MetaX / Ascend 的差异主要体现在执行粒度、内存层次和矩阵单元映射
3. GEMM 是跨平台迁移的最佳案例
4. benchmark 口径要先统一，再谈性能对比
5. 迁移过程要先保证正确性，再做平台调优
6. 教学上，Notebook 更适合展示迁移记录和对比表，`.py` 更适合保留可复现实验

## 进入下一章之前

下一章会回到 DSL 设计对比，也就是 TileLang 和 Triton 的取舍与选型。

## 本章交付检查

- [ ] 至少有一个平台上的 GEMM baseline
- [ ] 至少有一个平台上的迁移版
- [ ] benchmark 口径统一
- [ ] 参数记录完整
- [ ] 性能对比可复现

## 你还应该能判断什么

- 一个问题是平台差异，还是记录口径差异
- 哪些参数先固定，哪些参数后调整
- 迁移时应该先验证什么，再优化什么
- 哪些平台细节应该进入主文，哪些应该留在记录表里

## 本章最小闭环

如果现在先不追求所有平台都跑通，那么最小闭环应该是：

1. 选一个平台作为 baseline
2. 选一个目标平台作为迁移对象
3. 用同一份 benchmark 模板记录结果
4. 明确正确性检查和性能检查分开做
5. 把迁移前后参数差异写清楚

## 写作提醒

这一章的输出应该是一套迁移模板，而不是几个孤立的平台介绍。  
如果某个平台的细节暂时不够，宁可把迁移流程和记录方式先写稳。

## 常见误区

- 误区 1：跨平台对比就是平台排行榜
- 误区 2：记录表只是附属信息
- 误区 3：所有平台都应该完全共享一套参数

这一章的最终目标，是让迁移成为可重复流程，而不是一次性试验。

## 这一章的收束

你不需要先理解所有平台细节，才开始迁移。
你需要先有一套不会变的流程，再把平台差异当成流程中的可变项处理。

这一章的图表优先级也应该跟着这个原则走：先补第 8.1 的链路图和第 8.5 的迁移记录表，再补各平台的参数检查表，最后补跨平台对比图。

如果后面统一出图出表，建议直接对照 [图表总表](../FIGURE_TABLE_INDEX.md)，按“流程图 -> 平台表 -> 迁移表 -> 对比图”的顺序填。
