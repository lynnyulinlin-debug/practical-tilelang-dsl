# 8.5 GEMM 跨平台迁移案例

## 要解决什么问题

GEMM 是最适合做跨平台迁移的案例，因为它同时涉及计算密度、数据复用、内存层次和 tile 切分。

这一节的目标不是证明某个平台更快，而是把“迁移时要改什么、先改什么、怎么记录”讲成固定流程。

## 要讲什么

### 1. 统一的 GEMM 逻辑

- 按 tile 切分 M / N / K
- 把数据搬到片上缓冲
- 在片上重复计算
- 保持数学表达不变，只变实现参数

### 2. 不同平台上的差异

- NVIDIA：Tensor Core / shared memory / warp 组织
- ROCm：wavefront / local memory / 寄存器压力
- MetaX / Ascend：片上 Buffer / 矩阵计算路径 / 数据搬运
- tile 形状、并行粒度、缓冲映射都需要单独调

### 3. 性能记录方式

- TFLOPS
- GB/s
- latency
- 资源占用
- 版本和驱动信息
- shape、dtype、batch、warmup 次数也要固定

### 4. 迁移步骤

1. 先在一个平台上把 baseline 跑通
2. 固定输入 shape 和 benchmark 口径
3. 迁移到目标平台，先追求正确性
4. 再调整 tile、并行和布局参数
5. 记录每次改动带来的性能变化
6. 用同一套表格汇总结果

### 4b. 最小可执行顺序

- 先只改 `target`
- 再只改最小必要的布局参数
- 然后只改 tile 形状
- 最后才改并行粒度和更激进的优化项

这个顺序的作用，是把“哪个改动带来收益”拆开看，避免一开始就把多个变量混在一起。

### 5. 当前写作口径

- 先讲迁移流程，再讲平台差异
- 先统一 benchmark 字段，再看不同平台的参数
- 如果后面补代码，优先展示“同一逻辑、不同参数”的对照，而不是完全不同的实现

### 6. 验收条件

- 同一份基准输入在至少一个目标平台上跑通
- 迁移前后结果数值一致
- 迁移前后记录表字段一致
- 所有关键参数都能回溯到具体平台

## 小结

跨平台迁移的价值，不在于证明某个平台“最强”，而在于证明同一套方法可以稳定落到多个后端上。

如果没有统一输入 shape、统一 warmup、统一统计方式，跨平台比较就没有意义。
这一节要留下的不是某个 benchmark 数字，而是一个可以重复使用的迁移记录模板。

## 讲义提示

如果要把这一节讲得更实用，建议把内容拆成三栏：

- 左栏：原始 GEMM
- 中栏：目标平台参数
- 右栏：记录表和结果

## 这一节可以进一步追问的问题

- 迁移时最先改的是 target、tile 还是 layout
- benchmark 结果是否可以直接比较
- 哪些差异是平台差异，哪些差异是记录方式差异

## 常见误区

- 误区 1：迁移就是把代码复制到另一个平台
- 误区 2：同一套结果表可以不分平台复用所有字段
- 误区 3：只要性能提高，迁移流程就算完成

跨平台迁移的关键，是把“改了什么”和“为什么改”记录清楚。

读者只要照着这三栏，就能知道迁移时该保留什么、该改什么、该记录什么。
