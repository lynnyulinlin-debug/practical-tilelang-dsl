# 8.5 GEMM 跨平台迁移案例

GEMM 是最适合用来收口这一章的案例，因为它同时把算子逻辑、tile 切分、片上缓冲和平台差异都暴露出来了。  
如果前面的平台章节是在讲“各自怎么想”，那这一节就是把它们放回同一个迁移流程里，看同一份逻辑如何在不同平台上逐步落地。

这一节最重要的价值，不是再讲一次 GEMM，而是把前面所有平台判断收束成一个可执行流程。  
如果读者能在这个案例里看清楚“什么不变、什么要重调、什么必须记录”，后面换别的算子时就知道迁移不是从头开始。

## 要解决什么问题

GEMM 是最适合做跨平台迁移的案例，因为它同时涉及计算密度、数据复用、内存层次和 tile 切分。

这一节的目标不是证明某个平台更快，而是把“迁移时要改什么、先改什么、怎么记录”讲成固定流程。

所以这不是性能排行榜，而是一条迁移链。  
链条里每一步都对应一个验证点：先看逻辑是否稳定，再看参数是否分平台，再看结果是否还能放在同一把尺子上。

如果这个顺序反过来，读者看到的就会是零散的 benchmark，而不是一条可复用的流程。  
所以这一节最重要的不是数值本身，而是数值背后的验证顺序。

## 要讲什么

### 1. 统一的 GEMM 逻辑

- 按 tile 切分 M / N / K
- 把数据搬到片上缓冲
- 在片上重复计算
- 保持数学表达不变，只变实现参数

这一段是在强调一件事：跨平台迁移不是重写算法，而是保持数学表达不变，把实现参数拆到不同平台上分别调。  
只要逻辑稳定，平台差异就可以通过 target、tile、缓冲和并行参数来吸收。

这里的关键其实是“先定骨架，再填参数”。  
如果骨架都变了，后面的平台差异就不再是迁移，而是换了一套实现；只有骨架稳定，迁移才有比较意义。

### 2. 不同平台上的差异

- NVIDIA：Tensor Core / shared memory / warp 组织
- ROCm：wavefront / local memory / 寄存器压力
- MetaX / Ascend：片上 Buffer / 矩阵计算路径 / 数据搬运
- tile 形状、并行粒度、缓冲映射都需要单独调

这几项差异不是“额外麻烦”，而是平台约束本身。  
同一份 GEMM 在不同平台上表现不同，往往不是因为逻辑变了，而是因为硬件组织方式不同，导致 tile 和缓冲映射必须重新匹配。

这一段最值得读者记住的，是“差异来自约束，不只是来自名词”。  
如果把差异都理解成平台术语，就很难判断到底该改 tile、改并行，还是改缓冲映射。

### 3. 性能记录方式

- TFLOPS
- GB/s
- latency
- 资源占用
- 版本和驱动信息
- shape、dtype、batch、warmup 次数也要固定

### 4. 迁移步骤

1. 先在一个平台上把 baseline 跑通
2. 固定输入 shape 和 benchmark 口径
3. 迁移到目标平台，先追求正确性
4. 再调整 tile、并行和布局参数
5. 记录每次改动带来的性能变化
6. 用同一套表格汇总结果

这套步骤的核心，是把迁移过程拆成可验证的阶段。  
先跑通，再固定口径，再改参数，再记录收益，最后回到统一表格，这样才能知道到底是哪一步带来了变化。

这个顺序背后其实有一个很朴素的原则：一次只验证一类变化。  
只要把变化隔离开，迁移就会从“猜哪个参数有用”变成“知道哪个改动在起作用”。

### 4b. 最小可执行顺序

- 先只改 `target`
- 再只改最小必要的布局参数
- 然后只改 tile 形状
- 最后才改并行粒度和更激进的优化项

这个顺序的作用，就是把变量分离开。  
如果一开始就同时改 target、tile、layout 和并行粒度，最后你根本不知道到底是哪一个改动在起作用。

这个顺序的作用，是把“哪个改动带来收益”拆开看，避免一开始就把多个变量混在一起。

换句话说，这一节最重要的经验不是“怎么调得更快”，而是“怎么把改动拆成可解释的最小单位”。  
只要这个单位成立，后续复测、复现和横向比较才有意义。

### 5. 当前写作口径

- 先讲迁移流程，再讲平台差异
- 先统一 benchmark 字段，再看不同平台的参数
- 如果后面补代码，优先展示“同一逻辑、不同参数”的对照，而不是完全不同的实现

### 6. 验收条件

- 同一份基准输入在至少一个目标平台上跑通
- 迁移前后结果数值一致
- 迁移前后记录表字段一致
- 所有关键参数都能回溯到具体平台

## 小结

跨平台迁移的价值，不在于证明某个平台“最强”，而在于证明同一套方法可以稳定落到多个后端上。

如果没有这一层案例，前面的平台章节很容易停留在各自的注意事项；有了 GEMM 之后，读者才能把这些注意事项收拢成一条迁移流程。

如果没有统一输入 shape、统一 warmup、统一统计方式，跨平台比较就没有意义。
这一节要留下的不是某个 benchmark 数字，而是一个可以重复使用的迁移记录模板。

这也是为什么 8.5 应该写成“流程 + 记录 + 比较”三段式，而不是单纯的案例展示。  
流程告诉读者先做什么，记录告诉读者改了什么，比较告诉读者结果为什么能放在一起看。

## 讲义提示

如果要把这一节讲得更实用，建议把内容拆成三栏：

- 左栏：原始 GEMM
- 中栏：目标平台参数
- 右栏：记录表和结果

## 这一节可以进一步追问的问题

- 迁移时最先改的是 target、tile 还是 layout
- benchmark 结果是否可以直接比较
- 哪些差异是平台差异，哪些差异是记录方式差异

## 常见误区

- 误区 1：迁移就是把代码复制到另一个平台
- 误区 2：同一套结果表可以不分平台复用所有字段
- 误区 3：只要性能提高，迁移流程就算完成

跨平台迁移的关键，是把“改了什么”和“为什么改”记录清楚。

读者只要照着这三栏，就能知道迁移时该保留什么、该改什么、该记录什么。

所以这一节不是为了展示一个“完美的跨平台 GEMM”，而是为了给读者一套能复用的迁移顺序。  
只要顺序稳了，后面遇到别的算子也能照着走。
