# 8.3 ROCm 后端

## 要解决什么问题

ROCm 的重点是执行粒度和内存映射的差异，而不是简单换个编译目标。

这节要解决的问题是：同样的算子逻辑，为什么在 AMD 平台上不能直接照搬 NVIDIA 的默认经验。

## 要讲什么

### 1. wavefront 与 warp

- 线程组织方式不同
- 最优并行粒度不同
- 不能直接照搬 NVIDIA 的默认经验

### 2. local memory

- 片上缓冲 / 局部存储
- 和 shared memory 思路的对应关系

### 3. 调度和参数

- block / tile 需要重新评估
- 寄存器压力和向量化偏好要单独看

### 3b. 典型失误

- 直接沿用 NVIDIA 上的 block 设定
- 把 wavefront 组织方式当作细节
- 忽略 local memory 的访问模式
- 只改 tile 不改向量化，最后看不出收益

### 4. 迁移时先看什么

1. 先确认 wavefront 假设是否匹配当前 kernel 设计
2. 再确认 local memory 是否承载了正确的数据块
3. 然后检查向量化与寄存器压力是否平衡
4. 最后再微调 tile 形状和调度细节

### 4b. 收敛顺序

1. 先确认执行粒度
2. 再确认数据块能否正确落到 local memory
3. 再确认向量化是否和访问方向一致
4. 再确认寄存器压力没有把并发压垮
5. 最后才做更细的 tile 和调度调参

### 4. 当前写作口径

- 先讲 wavefront 和 local memory 的对应关系
- 再讲为什么参数不能直接照搬
- 最后讲迁移时应该先验证什么，再优化什么

### 5. 不要默认照搬的经验

- 不要默认 NVIDIA 上的 warp 经验可以直接套用
- 不要默认同样的 tile 形状在 ROCm 上有同样的收益
- 不要把 local memory 只看成 shared memory 的同名替换
- 不要在没有验证执行粒度前就先做复杂优化

### 6. 这一节应该留下什么

- 一个 AMD 平台上判断 wavefront 的思路
- 一个 local memory 是否在承载热数据的判断方法
- 一个向量化与寄存器压力的平衡顺序

## 小结

ROCm 版本的核心不是“翻译 CUDA”，而是重新审视并行粒度和内存布局。
这和第 8.1 节的“先统一逻辑，再分平台调参”是一致的。

## 讲义提示

如果想把这一节讲深一点，建议围绕“执行粒度”展开：

- wavefront 是怎么组织的
- 数据块怎么放进 local memory
- 调度为什么会影响并发

## 这一节可以进一步追问的问题

- wavefront 假设是否匹配当前 kernel
- local memory 是否真正在承载热数据
- 向量化方向是否和访问方向一致

## 常见误区

- 误区 1：ROCm 只是 NVIDIA 的替代品
- 误区 2：同样的 tile 在不同平台上的收益一定相似
- 误区 3：只要照搬并行参数就能得到类似结果

ROCm 的关键判断，是先确认执行粒度和内存映射，再谈更细的性能调整。

这比单纯讲平台名词更能帮助读者建立迁移心智。
