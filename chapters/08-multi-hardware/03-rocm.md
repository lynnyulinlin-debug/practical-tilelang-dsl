## 要解决什么问题

ROCm 的重点是执行粒度和内存映射的差异，而不是简单换个编译目标。

换句话说，这一节不是在问“能不能编译过去”，而是在问“编译过去以后，执行方式是不是还是对的”。  
这很关键，因为 ROCm 上最常见的错误，不是代码没跑，而是跑了但假设错了。

## 要讲什么

### 1. wavefront 与 warp

- 线程组织方式不同
- 最优并行粒度不同
- 不能直接照搬 NVIDIA 的默认经验

这里的核心不是名词不同，而是并行组织方式不同。  
如果把 wavefront 当成 warp 的同义词，就会在并行粒度、同步方式和资源预算上都做错假设。

所以 wavefront 这一层先讲，不是为了多认识一个平台词，而是为了避免把并行粒度的前提搞错。  
前提一旦错了，后面的 local memory 和 tile 调参就会全部偏航。

所以 wavefront 这一层先讲，不是为了多认识一个平台词，而是为了避免把并行粒度的前提搞错。  
前提一旦错了，后面的 local memory 和 tile 调参就会全部偏航。

### 2. local memory

- 片上缓冲 / 局部存储
- 和 shared memory 思路的对应关系

local memory 的作用并不是“叫法换了”，而是它对应的是 AMD 这条路径下实际可用的片上存储层。  
所以它既要看能不能承载热数据，也要看访问模式会不会把收益吃掉。

这一点和 NVIDIA 上的 shared memory 很像，但不能直接等同。  
读者在这里最需要建立的是“同样是片上存储，但它们在不同平台上的约束边界不同”。

这一点和 NVIDIA 上的 shared memory 很像，但不能直接等同。  
读者在这里最需要建立的是“同样是片上存储，但它们在不同平台上的约束边界不同”。

### 3. 调度和参数

- block / tile 需要重新评估
- 寄存器压力和向量化偏好要单独看

这一步的关键，是把“能不能照搬”变成“哪些参数需要重估”。  
block、tile、向量化、寄存器压力在 ROCm 上往往不是 NVIDIA 的简单镜像，它们要按 wavefront 和 local memory 的实际约束重新平衡。

也就是说，ROCm 上最重要的不是先把参数调得漂亮，而是先确认这些参数是不是在正确的约束下被解释。  
如果约束没对齐，调得再细也只是把误差压得更小。

如果约束没对齐，调得再细也只是把误差压得更小。

### 3b. 典型失误

- 直接沿用 NVIDIA 上的 block 设定
- 把 wavefront 组织方式当作细节
- 忽略 local memory 的访问模式
- 只改 tile 不改向量化，最后看不出收益

### 4. 迁移时先看什么

1. 先确认 wavefront 假设是否匹配当前 kernel 设计
2. 再确认 local memory 是否承载了正确的数据块
3. 然后检查向量化与寄存器压力是否平衡
4. 最后再微调 tile 形状和调度细节

### 4b. 收敛顺序

1. 先确认执行粒度
2. 再确认数据块能否正确落到 local memory
3. 再确认向量化是否和访问方向一致
4. 再确认寄存器压力没有把并发压垮
5. 最后才做更细的 tile 和调度调参

这五步的顺序，其实是在先锁定执行粒度，再看数据有没有被放对位置，最后才进入细调。  
如果一上来就改得很细，往往只是把假设错误的成本藏起来。

这四步和后面的收敛顺序其实是一回事：先把最容易错的前提锁住，再看数据流是否落到正确的存储层，最后才进入细调。  
如果一上来就改得很细，往往只是把假设错误的成本藏起来。

### 4. 当前写作口径

- 先讲 wavefront 和 local memory 的对应关系
- 再讲为什么参数不能直接照搬
- 最后讲迁移时应该先验证什么，再优化什么

### 5. 不要默认照搬的经验

- 不要默认 NVIDIA 上的 warp 经验可以直接套用
- 不要默认同样的 tile 形状在 ROCm 上有同样的收益
- 不要把 local memory 只看成 shared memory 的同名替换
- 不要在没有验证执行粒度前就先做复杂优化

### 6. 这一节应该留下什么

- 一个 AMD 平台上判断 wavefront 的思路
- 一个 local memory 是否在承载热数据的判断方法
- 一个向量化与寄存器压力的平衡顺序

## 小结

ROCm 版本的核心不是“翻译 CUDA”，而是重新审视并行粒度和内存布局。
这和第 8.1 节的“先统一逻辑，再分平台调参”是一致的。

## 常见误区

- 误区 1：ROCm 只是 NVIDIA 的替代品
- 误区 2：同样的 tile 在不同平台上的收益一定相似
- 误区 3：只要照搬并行参数就能得到类似结果

ROCm 的关键判断，是先确认执行粒度和内存映射，再谈更细的性能调整。

这比单纯讲平台名词更能帮助读者建立迁移心智。

如果把这一节压成一句话，那就是：ROCm 上的优化不是“照搬 NVIDIA 后改几个参数”，而是“先重新确认粒度，再重新确认映射，最后才重新调参”。
