# 8.1 统一 DSL，多后端编译

## 要解决什么问题

同一份算子逻辑为什么能落到多个后端，是这章的起点。  
这一节不讲硬件历史，而是讲统一抽象为什么能带来迁移能力。

它的核心是把“我想算什么”和“在哪个后端算”拆开，避免把平台差异提前写死在算法逻辑里。

## 要讲什么

### 1. 统一入口

- `target` 的作用
- 同一份 DSL / TIR 代码如何进入不同后端

### 2. 逻辑一致与实现差异

- 数学结果要一致
- 实现路径可以不同
- 参数不能默认共用
- benchmark 口径不能混用

### 3. 迁移心智模型

- 算法逻辑尽量共享
- 后端配置单独维护
- 性能参数按平台调优
- 先保证可编译，再保证可跑通，最后才追求最优

### 4. 当前写作口径

- 先把统一入口讲清
- 再把“逻辑一致、实现不同”这条原则讲清
- 最后说明为什么迁移记录必须平台分开维护

### 5. 一个最小的迁移模型

```text
same algorithm
  -> same shape / dtype / correctness check
  -> different target / layout / tile / launch config
  -> different benchmark record
```

这个模型的意思是：

- 算法层尽量少改
- 平台层必须显式区分
- 任何性能结论都要带着平台上下文一起看

## 小结

多后端的前提不是“完全不改”，而是“先统一逻辑，再分平台调参数”。
这也是后面所有平台章节都共享的基本方法。

## 这一节可以进一步追问的问题

- 哪些东西应该保持一致，哪些东西应该明确分平台
- benchmark 口径为什么不能被 target 混掉
- 同一个算法为什么要和不同平台的记录表分开看

## 常见误区

- 误区 1：统一 DSL 就意味着所有平台都用同一套参数
- 误区 2：逻辑一致就等于性能也会一致
- 误区 3：只要抽象统一，平台差异就不重要

统一 DSL 的意义，是把可共享的部分先稳定下来，把必须分开的部分显式标出来。
