# 8.1 统一 DSL，多后端编译

第 8 章真正要解决的，不是“能不能支持更多平台”，而是“同一份算法逻辑如何不被平台差异打碎”。  
如果每个后端都要重新写一套算法，那迁移就会变成复制代码；只有先把逻辑和平台参数拆开，统一 DSL 才会真正产生价值。

这一节要先建立的，不是平台知识，而是迁移边界。  
边界一旦清楚，就能分清：哪些东西属于算法本体，哪些东西属于平台适配，哪些东西属于结果记录。

## 统一 DSL 的迁移结论

同一份算子逻辑为什么能落到多个后端，是这章的起点。  
这一节不讲硬件历史，而是讲统一抽象为什么能带来迁移能力。

它的核心是把“我想算什么”和“在哪个后端算”拆开，避免把平台差异提前写死在算法逻辑里。

如果不先做这层拆分，后面每次换平台都会像重写一遍程序；而如果先把抽象层立住，迁移就会退化成调整 target、layout、tile 和 launch config 这类可测参数。

### 1. 统一入口

- `target` 的作用
- 同一份 DSL / TIR 代码如何进入不同后端

这里的关键是入口统一，不是参数统一。  
`target` 负责告诉编译链路应该往哪个后端走，但它并不意味着所有平台都应该共享同一套调参结果。统一入口的意义，是先让“算法怎么写”稳定下来，再让“平台怎么落地”分别处理。

这句话背后其实有一个很强的约束：入口统一解决的是“可编译、可对齐”，不是“性能自动一致”。  
性能之所以还要分平台调，正是因为不同后端对 tile、并行粒度、缓冲映射和资源分配的偏好不同。

### 2. 逻辑一致与实现差异

- 数学结果要一致
- 实现路径可以不同
- 参数不能默认共用
- benchmark 口径不能混用

这一组约束是多后端章节的底层规则。  
数学结果一致，说明你迁移的是同一个算子；实现路径不同，说明你在适配平台；参数和 benchmark 不能混用，说明平台差异必须显式记录，而不能被一个“通用最优”掩盖掉。

这里最值得强调的是“不能混用”这件事。  
如果 benchmark 条件混了，看到的就不是平台差异，而是样本差异；如果参数混了，后面就很难判断收益到底来自哪里。

### 3. 迁移心智模型

- 算法逻辑尽量共享
- 后端配置单独维护
- 性能参数按平台调优
- 先保证可编译，再保证可跑通，最后才追求最优

这个心智模型很重要，因为它决定了迁移顺序。  
先共享逻辑，才能避免重复开发；再分开配置，才能避免平台参数互相污染；最后才追求最优，才能避免在还没跑通时就陷入局部调参。

这节不是在教“怎么把代码搬过去”，而是在教“怎么把迁移拆成三件事”：  
先证明逻辑没变，再证明平台参数不同，最后证明结果还能被统一记录和比较。

### 4. 统一逻辑和平台参数的边界

- 先把统一入口讲清
- 再把“逻辑一致、实现不同”这条原则讲清
- 最后说明为什么迁移记录必须平台分开维护

### 5. 一个最小的迁移模型

```text
same algorithm
  -> same shape / dtype / correctness check
  -> different target / layout / tile / launch config
  -> different benchmark record
```

这个模型的意思是：

- 算法层尽量少改
- 平台层必须显式区分
- 任何性能结论都要带着平台上下文一起看

这个模型再往下收，就会落到算法逻辑、平台参数和记录口径：

1. 算法逻辑：数学表达和正确性检查
2. 平台参数：target、tile、layout、launch config
3. 记录口径：benchmark 字段、版本信息、硬件上下文

这些部分一旦分开，后面的平台章节就能保持同一套叙事节奏，而不会变成各平台自说自话。

这一节要形成一个最基本的判断：  
如果你在迁移时改动的是算法本体，那说明抽象还不够稳；如果你改的是平台参数，那说明迁移正在正确的层次上发生。

## 统一 DSL 的迁移收口

多后端的前提不是“完全不改”，而是“先统一逻辑，再分平台调参数”。
这也是后面所有平台章节都共享的基本方法。

如果要把 8.1 写得更实一点，最终应该能说明：  
“改动的是平台参数，不是算法本体；比较的是同一输入规模、同一硬件和同一统计方式下的结果，不是不同样本。”  
这句话成立，后面的 NVIDIA、ROCm 和专有平台细节才有可比性。

## 统一 DSL 的常见误区

- 误区 1：统一 DSL 就意味着所有平台都用同一套参数
- 误区 2：逻辑一致就等于性能也会一致
- 误区 3：只要抽象统一，平台差异就不重要

统一 DSL 的意义，是把可共享的部分先稳定下来，把必须分开的部分显式标出来。

如果还要继续追问，就顺着这三个误区往下看：哪些东西应该保持一致，哪些东西应该明确分平台；benchmark 口径为什么不能被 target 混掉；以及同一个算法为什么要和不同平台的记录表分开看。  
这也是后面每个平台章节的共同出发点：先把共同逻辑固定，再把平台差异单独拉出来看。
