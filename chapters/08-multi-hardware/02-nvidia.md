# 8.2 NVIDIA 后端

NVIDIA 这一节放在最前面，是因为它最成熟，也最容易把“参数怎么影响性能”讲清楚。  
先把指令链路、资源占用和形状对齐分开看，后面看 ROCm 或专有平台时就会少很多误判。

这一节要建立的，不是对 CUDA 的熟悉度，而是对“性能为什么会掉”的判断顺序。  
如果只记住 Tensor Core 这个词，却没看见指令链路、资源占用和 tile 形状之间的关系，那后面的调参还是在猜。

## NVIDIA 的路径检查

先把最成熟、最常见的后端讲清楚，后面再对比其他平台。

这一节不是在讲 CUDA 入门，而是在讲 TileLang 代码到 NVIDIA 平台时，哪些层次最影响最终性能。

这一节的任务是把“跑得慢”拆成几个具体问题：指令没传顺、资源先顶满、或者形状没对齐。  
这些问题分开了，后面的优化顺序就不会乱。

### 1. PTX / SASS

- PTX 作为中间表示
- SASS 作为最终机器指令
- 编译链路对性能的影响

这一层要说明，TileLang 代码并不是直接落到硬件上，而是先经过中间表示，再落到最终指令。  
所以性能问题有时不是出在算法，而是出在中间表示能不能把 tile、布局和资源约束传得足够清楚。

对初学者来说，可以先把 PTX 看成 NVIDIA 后端里的“可读中间层”，SASS 则是更接近最终硬件的落地结果。  
这章不需要把 PTX 当成 ISA 教程来学，只要看它有没有把形状、布局和资源约束正确传下去。

### 2. Tensor Core

- 矩阵计算单元
- 何时能发挥优势
- 数据布局和 tile 形状的重要性

Tensor Core 不是“加个开关就快”，而是一个对输入形状和布局非常敏感的执行路径。  
如果 tile 形状没对齐、数据没规整、对齐条件没满足，那矩阵单元的优势就不会自动出现。

所以这一节要形成一个实用的习惯：先问当前 shape 是否适合 Tensor Core，再问布局是否规整，最后才问资源层是否允许它跑起来。  
这样看问题，Tensor Core 就不是“高性能标签”，而是一个有条件的路径。

### 3. 性能关注点

- shared memory 容量
- register 使用量
- block 大小
- memory coalescing
- bank conflict

这几项要一起看，因为它们共同决定 NVIDIA 上的资源边界。  
有时候不是算力不够，而是 shared memory 或 register 先把并发压低了；有时候不是 Tensor Core 不行，而是访问模式先把带宽浪费掉了。

如果把这几项拆开看，很容易误判：  
看见 TFLOPS 低，就以为算力不够；看见 occupancy 低，就以为 block 不对；看见访问不合并，就以为 tile 不行。  
实际上，NVIDIA 上最常见的问题，恰恰是这些约束互相叠加。

### 4. 典型失误

- 只看 Tensor Core，不看 tile 是否对齐
- 把 block 设大后忽略 register spill
- 把 shared memory 拉满后忽略占用率下降
- 只看峰值 TFLOPS，不看 latency 和稳定性

### 5. NVIDIA 的路径检查顺序

1. 先确认能否正确编译到 NVIDIA target
2. 再确认基本 shape 和 dtype 是否跑通
3. 然后检查 block / tile 是否和 Tensor Core 形状对齐
4. 最后再调 shared memory、寄存器和并发度

这个顺序很重要：先确认能编译、能跑通，再确认 tile 和 Tensor Core 的形状是否对齐，最后才调资源层。  
顺序反了，常见结果就是把最底层的问题误以为是参数问题。

这一步真正要看的，不是“照着哪个经验抄”，而是指令、资源和形状这三类约束是否同时成立。  
后面补 benchmark 时，也要和第 8.5 节里的字段和测量条件保持一致。

不要默认把别的平台上的 tile 形状直接拿过来，也不要默认 block 越大越好。  
shared memory、Tensor Core 和寄存器压力都要放到同一张检查表里看，性能异常时也先按这个顺序排查。

## NVIDIA 的路径判断

NVIDIA 后端的重点不是“CUDA 基础”，而是“如何让 TileLang 输出更接近硬件极限的实现”。
这一节关注的是映射和约束，不是基础语法。

它真正要传达的，是 NVIDIA 上的优化顺序：先指令链路，再资源占用，再访存和形状。  
只要这个顺序成立，后续平台上的判断顺序也会更稳定。

## NVIDIA 的关键约束

如果想把这一节讲深一点，建议把“为什么快”拆成指令链路、资源占用和形状对齐：

- 指令层：PTX / SASS
- 资源层：shared memory / register / occupancy
- 形状层：tile / 对齐 / coalescing

## NVIDIA 的常见误区

- 误区 1：NVIDIA 后端就是 CUDA 教程
- 误区 2：只要启用了 Tensor Core 就结束了
- 误区 3：越接近峰值 TFLOPS 就一定越适合当前场景

如果还要继续追问，就顺着这三个误区往下看：这一步的性能瓶颈更像指令、资源还是形状问题；Tensor Core 是否真的适合当前形状；以及 block 和 shared memory 的配置是否已经逼近资源上限。  
这节要教的是 NVIDIA 上的参数判断顺序，而不是 CUDA 语法本身。  
把这些部分区分开，就能看懂很多性能问题为什么不是单一因素造成的。  
NVIDIA 上的优化不是“把参数调满”，而是“先确认路径，再确认资源，最后确认形状”。  
这个顺序稳定了，后面的平台差异就会更好理解。
