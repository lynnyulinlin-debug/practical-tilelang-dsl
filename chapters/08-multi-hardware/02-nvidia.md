# 8.2 NVIDIA 后端

NVIDIA 这一节之所以放在最前面，是因为它最成熟，也最容易把“参数怎么影响性能”这件事讲清楚。  
读者只要把 NVIDIA 的指令层、资源层和形状层分开理解，后面看 ROCm 或专有平台时就会少很多误判。

这一节真正要建立的，不是对 CUDA 的熟悉度，而是对“性能为什么会掉”的分层判断。  
如果读者只记住 Tensor Core 这个词，却没看见指令链路、资源占用和 tile 形状之间的关系，那后面所有调参都还是在猜。

## 要解决什么问题

先把最成熟、最常见的后端讲清楚，后面再对比其他平台。

这一节不是在讲 CUDA 入门，而是在讲 TileLang 代码到 NVIDIA 平台时，哪些层次最影响最终性能。

换句话说，这一节的任务是把“跑得慢”拆成三类问题：指令没传顺、资源先顶满、或者形状没对齐。  
只要这三类问题能分开，后面的优化顺序就不会乱。

## 要讲什么

### 1. PTX / SASS

- PTX 作为中间表示
- SASS 作为最终机器指令
- 编译链路对性能的影响

这一步要让读者知道，TileLang 代码并不是直接落到硬件上，而是先经过中间表示，再落到最终指令。  
所以性能问题有时不是出在算法，而是出在中间表示能不能把 tile、布局和资源约束传得足够清楚。

这也是为什么 PTX / SASS 要先讲：它们决定的是“编译链条有没有把抽象传到位”。  
如果这条链条没传通，后面再怎么调 shared memory 或 block 大小，也只能在错误的基础上微调。

对初学者来说，可以先把 PTX 看成 NVIDIA 后端里的“可读中间层”，SASS 则是更接近最终硬件的落地结果。我们在这一章不需要把 PTX 当成 ISA 教程来学，只需要知道：如果 PTX 里的形状、布局和资源约束已经和预期不一致，后面的最终指令通常也不会自己变好。也就是说，PTX 的价值在于帮你确认编译器有没有把上层意图正确传下去。

### 2. Tensor Core

- 矩阵计算单元
- 何时能发挥优势
- 数据布局和 tile 形状的重要性

Tensor Core 不是“加个开关就快”，而是一个对输入形状和布局非常敏感的执行路径。  
如果 tile 形状没对齐、数据没规整、对齐条件没满足，那矩阵单元的优势就不会自动出现。

所以这一节要让读者形成一个很实用的习惯：先问当前 shape 是否适合 Tensor Core，再问布局是否规整，最后才问资源层是否允许它真正跑起来。  
这样看问题，Tensor Core 就不是“高性能标签”，而是一个要满足条件的路径。

### 3. 性能关注点

- shared memory 容量
- register 使用量
- block 大小
- memory coalescing
- bank conflict

这几项要一起看，因为它们共同决定 NVIDIA 上的资源边界。  
有时候不是算力不够，而是 shared memory 或 register 先把并发压低了；有时候不是 Tensor Core 不行，而是访问模式先把带宽浪费掉了。

如果把这几项拆开看，读者很容易误判：  
看见 TFLOPS 低，就以为算力不够；看见 occupancy 低，就以为 block 不对；看见访问不合并，就以为 tile 不行。  
实际上，NVIDIA 上最常见的问题，恰恰是这三类约束互相叠加。

### 3b. 典型失误

- 只看 Tensor Core，不看 tile 是否对齐
- 把 block 设大后忽略 register spill
- 把 shared memory 拉满后忽略占用率下降
- 只看峰值 TFLOPS，不看 latency 和稳定性

### 4. 迁移时先看什么

1. 先确认能否正确编译到 NVIDIA target
2. 再确认基本 shape 和 dtype 是否跑通
3. 然后检查 block / tile 是否和 Tensor Core 形状对齐
4. 最后再调 shared memory、寄存器和并发度

这个顺序很重要：先确认能编译、能跑通，再确认 tile 和 Tensor Core 的形状是否对齐，最后才调资源层。  
如果顺序反了，常见结果就是把最底层的问题误以为是参数问题。

这一点也解释了为什么第 8 章不能直接从“调到最快”开始写。  
在 NVIDIA 这种成熟平台上，最容易出现的不是没有优化手段，而是优化顺序不对，导致读者把结构问题、资源问题和形状问题混成一团。

### 4b. 收敛顺序

1. 先保证输出正确
2. 再保证 tile 和硬件单元匹配
3. 再看访存是否合并
4. 再看 shared memory 是否足够但不过量
5. 最后看寄存器压力和并发是否平衡

### 4. 当前写作口径

- 这一节优先讲落地路径和性能关注点
- 不是要重讲 CUDA 语法，而是要解释为什么 TileLang 的参数在 NVIDIA 上要这样选
- 如果后面补 benchmark，要和第 8.5 节的统一口径一致

### 5. 不要默认照搬的经验

- 不要默认把别的平台上的 tile 形状直接拿过来
- 不要默认 block 越大越好
- 不要默认 shared memory 越多越好
- 不要默认“Tensor Core 一定优于普通路径”，要看 shape 和对齐条件

### 6. 这一节应该留下什么

- 一个 NVIDIA 上的参数检查顺序
- 一个判断 Tensor Core 是否值得用的思路
- 一个看到性能异常时的第一轮排查方向

## 小结

NVIDIA 后端的重点不是“CUDA 基础”，而是“如何让 TileLang 输出更接近硬件极限的实现”。
换句话说，这一节关注的是映射和约束，不是基础语法。

它真正要传达的，是 NVIDIA 上的优化顺序：先指令链路，再资源占用，再访存和形状。  
只要这个顺序成立，读者在后续平台上也会知道先看什么。

## 讲义提示

如果想把这一节讲深一点，建议把“为什么快”拆成三层：

- 指令层：PTX / SASS
- 资源层：shared memory / register / occupancy
- 形状层：tile / 对齐 / coalescing

## 这一节可以进一步追问的问题

- 这一步的性能瓶颈更像指令、资源还是形状问题
- Tensor Core 是否真的适合当前形状
- block 和 shared memory 的配置是否已经逼近资源上限

## 常见误区

- 误区 1：NVIDIA 后端就是 CUDA 教程
- 误区 2：只要启用了 Tensor Core 就结束了
- 误区 3：越接近峰值 TFLOPS 就一定越适合当前场景

这节要教的是 NVIDIA 上的参数判断顺序，而不是 CUDA 语法本身。

读者一旦能把这三层区分开，就能看懂很多性能问题为什么不是单一因素造成的。

如果把这一节压成一句话，那就是：NVIDIA 上的优化不是“把参数调满”，而是“先确认路径，再确认资源，最后确认形状”。  
这个顺序一旦稳定，后面的平台差异就会更好理解。
