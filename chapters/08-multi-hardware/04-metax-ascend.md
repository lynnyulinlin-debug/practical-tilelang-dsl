# 8.4 MetaX / Ascend 后端

## 要解决什么问题

这一节把另一类平台放进来，重点讨论它们与 GPU 的映射关系，而不是硬件历史背景。

它更像是在回答：如果目标平台有自己的片上结构和矩阵路径，TileLang 的内存层次和 tile 结构该怎么对应。

## 要讲什么

### 1. 片上缓冲映射

- shared 类缓冲如何对应到片上结构
- fragment 类存储如何对应到计算片段

### 2. 矩阵计算单元

- 专用矩阵路径
- tile 形状和数据对齐

### 3. 编译和运行差异

- target 差异
- 运行时开关
- 平台专有配置

### 3b. 典型失误

- 只看矩阵计算单元，不看数据搬运
- 只调 tile，不看片上缓冲是否足够
- 把专有配置当成主问题，而不是把它当成最后一步
- 忽略运行时开关会改变可用路径

### 4. 迁移时先看什么

1. 先确认 tile 是否和片上结构对齐
2. 再确认矩阵路径是否覆盖了热点计算
3. 然后检查数据搬运是否是主要瓶颈
4. 最后再调专有配置和运行时参数

### 4b. 收敛顺序

1. 先确认是否能映射到目标平台的矩阵路径
2. 再确认片上缓冲能否承载热数据
3. 再确认 tile 是否适配数据流
4. 再确认运行时开关是否打开了正确的路径
5. 最后再做更细的平台专有调参

### 4. 当前写作口径

- 先讲映射关系，再讲差异
- 不要先铺平台名词，要先说明 TileLang 的抽象为什么能落到这些结构上
- 如果后面补表格，要和第 8.5 节的统一 benchmark 字段一致

### 5. 不要默认照搬的经验

- 不要默认 GPU 上的缓存层次可以直接映射成同样的优化顺序
- 不要默认 tile 形状只要和矩阵计算单元对齐就一定够好
- 不要先追求平台术语的完整覆盖，先追求路径和瓶颈的正确识别

### 6. 这一节应该留下什么

- 一个“tile 与片上结构对齐”的判断框架
- 一个“矩阵路径是否吃到热点”的检查顺序
- 一个“先路径，后专有配置”的优化次序

## 小结

这类平台更强调显式内存和矩阵计算映射，因此 TileLang 的抽象通常更容易直接落到性能关键路径上。
这一点和 GPU 平台的共同点是：都要把 tile 和片上资源绑定起来看。

## 讲义提示

如果想把这一节讲深一点，建议把重点放在“映射”而不是“平台名词”：

- 哪些数据进片上缓冲
- 哪些计算走矩阵路径
- 哪些配置只是在收尾阶段才需要

## 这一节可以进一步追问的问题

- tile 是否已经对齐片上结构
- 热点计算是否真正走到了矩阵路径
- 运行时开关是否影响了主路径

## 常见误区

- 误区 1：专有配置是迁移最重要的难点
- 误区 2：只要有矩阵单元，就自动能得到高性能
- 误区 3：平台专有结构只是另一个名字而已

先把路径和映射确认好，再去调专有配置，通常更稳。

这样读者才能把它当作可迁移方法，而不是平台介绍。
