# 8.4 MetaX / Ascend 后端

这一节不是为了补充一组新平台名词，而是为了把“片上结构 + 矩阵路径 + 运行时开关”这三件事放进同一个判断框架里。  
对这类平台来说，迁移的重点往往不是单个算子语法，而是数据到底能不能顺着目标硬件的结构走下去。

这一节真正要建立的，不是平台名词，而是迁移时的判断顺序。  
如果读者先去看配置，而不是先看映射，很多本来是结构问题的东西就会被误判成参数问题。

## 要解决什么问题

这一节把另一类平台放进来，重点讨论它们与 GPU 的映射关系，而不是硬件历史背景。

它更像是在回答：如果目标平台有自己的片上结构和矩阵路径，TileLang 的内存层次和 tile 结构该怎么对应。

换句话说，这一节不是在问“有没有专有词”，而是在问“TileLang 的抽象能不能顺着这些结构真正落下去”。  
这也是为什么它和 NVIDIA / ROCm 节一样，必须先谈路径，再谈参数。

## 要讲什么

### 1. 片上缓冲映射

- shared 类缓冲如何对应到片上结构
- fragment 类存储如何对应到计算片段

这里最先要确认的，不是配置项，而是映射关系。  
如果 shared 类缓冲和片上结构没对上，后面的矩阵路径再强也很难把数据流真正压住。

这一步其实是在先确认“数据该停在哪一层”。  
只要停靠层错了，后面的矩阵路径和运行时开关都只能修补错误，而不是解决问题。

### 2. 矩阵计算单元

- 专用矩阵路径
- tile 形状和数据对齐

这一步和 NVIDIA 上的 Tensor Core 类似，但平台语义和运行时路径并不相同。  
重点仍然是先看 tile 是否对齐、数据是否规整，而不是先追求专有名词上的覆盖。

这里最容易犯的错，就是把“有矩阵路径”误当成“已经能高性能”。  
实际上，路径存在只是前提，对齐、规整和数据搬运方式才决定它能不能真正吃到热点。

### 3. 编译和运行差异

- target 差异
- 运行时开关
- 平台专有配置

运行时开关在这里尤其重要，因为它会直接影响主路径是否被打开。  
所以这一节不能只看静态编译结果，还要把运行时路径也纳入迁移判断。

这意味着 MetaX / Ascend 上的迁移，不能只盯着编译输出看，还要把“运行时有没有走到想要的路径”作为必须检查的一步。  
否则你以为打开了矩阵路径，实际上跑的还是另一条路径。

### 3b. 典型失误

- 只看矩阵计算单元，不看数据搬运
- 只调 tile，不看片上缓冲是否足够
- 把专有配置当成主问题，而不是把它当成最后一步
- 忽略运行时开关会改变可用路径

### 4. 迁移时先看什么

1. 先确认 tile 是否和片上结构对齐
2. 再确认矩阵路径是否覆盖了热点计算
3. 然后检查数据搬运是否是主要瓶颈
4. 最后再调专有配置和运行时参数

这个顺序的意思很明确：先确认数据和 tile 是否能落到目标平台的结构上，再确认矩阵路径是否真的吃到热点，最后才去碰平台专有配置。  
如果直接从专有配置开始，很容易把本应由结构解决的问题误判成参数问题。

换句话说，这一节的判断逻辑和前面两个平台是一样的：先确认最底层的映射关系，再看热点有没有被接住，最后才轮到专有配置。  
只要这个顺序不乱，平台词再多也不会压过方法论。

### 4b. 收敛顺序

1. 先确认是否能映射到目标平台的矩阵路径
2. 再确认片上缓冲能否承载热数据
3. 再确认 tile 是否适配数据流
4. 再确认运行时开关是否打开了正确的路径
5. 最后再做更细的平台专有调参

这五步其实是在把“路径是否正确”拆成更小的可验证单元。  
如果这五步都过不了，说明问题还停留在映射层，没必要急着进细调。

### 4. 当前写作口径

- 先讲映射关系，再讲差异
- 不要先铺平台名词，要先说明 TileLang 的抽象为什么能落到这些结构上
- 如果后面补表格，要和第 8.5 节的统一 benchmark 字段一致

### 5. 不要默认照搬的经验

- 不要默认 GPU 上的缓存层次可以直接映射成同样的优化顺序
- 不要默认 tile 形状只要和矩阵计算单元对齐就一定够好
- 不要先追求平台术语的完整覆盖，先追求路径和瓶颈的正确识别

### 6. 这一节应该留下什么

- 一个“tile 与片上结构对齐”的判断框架
- 一个“矩阵路径是否吃到热点”的检查顺序
- 一个“先路径，后专有配置”的优化次序

## 小结

这类平台更强调显式内存和矩阵计算映射，因此 TileLang 的抽象通常更容易直接落到性能关键路径上。
这一点和 GPU 平台的共同点是：都要把 tile 和片上资源绑定起来看。

换句话说，这一节要让读者看到的，是“先路径、后参数”的迁移顺序。  
只要路径对了，平台专有配置才有意义；路径没对，配置再多也只是增加噪声。

如果把这一节压成一句话，那就是：这类平台的优化先看映射，再看热点，最后才看开关和专有参数。  
这个顺序一旦成立，读者就能把它和 NVIDIA / ROCm 的判断方式接上。

## 讲义提示

如果想把这一节讲深一点，建议把重点放在“映射”而不是“平台名词”：

- 哪些数据进片上缓冲
- 哪些计算走矩阵路径
- 哪些配置只是在收尾阶段才需要

## 这一节可以进一步追问的问题

- tile 是否已经对齐片上结构
- 热点计算是否真正走到了矩阵路径
- 运行时开关是否影响了主路径

## 常见误区

- 误区 1：专有配置是迁移最重要的难点
- 误区 2：只要有矩阵单元，就自动能得到高性能
- 误区 3：平台专有结构只是另一个名字而已

先把路径和映射确认好，再去调专有配置，通常更稳。

这样读者才能把它当作可迁移方法，而不是平台介绍。
