# 6.3 Shared memory GEMM

## 这一节要解决什么问题

朴素 GEMM 的主要问题是重复读取。  
这一节通过 shared memory 把重复读取变成块内复用。

这也是 TileLang 里“显式控制内存层次”的第一个真正例子：  
不是单纯把代码写得更短，而是把数据移动到更快的层次，再让 block 内多个线程共享它。

## 这一节要讲什么

### 1. 分块思路

- 把 M、N、K 切成 tile
- A 和 B 的子块先搬到 shared memory

### 2. 共享内存中的复用

- 一个 tile 被 block 内多个线程重复使用
- 读一次，多次算

### 3. 访问模式和同步

- 加载阶段
- 计算阶段
- 同步阶段

### 4. 当前实现关注点

- 先保证 tile 划分清楚
- 再保证 shared memory 的读写路径正确
- 最后再看优化后的性能差异

### 5. 这一节应该留下的判断标准

- 如果 A / B tile 会被重复使用，shared memory 通常会有明显收益
- 如果 tile 切得太细，sync 开销和地址计算会变多
- 如果 tile 切得太粗，shared memory 和寄存器压力会抬高

这说明 shared memory 优化的关键不是“用了没有”，而是“tile 形状是否合适”。

## 这一节的结论

shared memory 的价值不在“更快”，而在“把重复访问从慢层搬到快层”。
这一步把 GEMM 从纯访存问题，推进成“访存 + 复用”的综合优化问题。

## 这一节可以进一步追问的问题

- tile 为什么能让 block 内的数据重复利用
- shared memory 的收益为什么会受 tile 形状影响
- 同步和地址计算的成本何时会抵消复用收益

## 常见误区

- 误区 1：shared memory 只要加上去，性能就会显著提升
- 误区 2：tile 形状只影响代码可读性
- 误区 3：把所有中间数据都放进 shared memory 一定是好事

shared memory 优化的本质，是把“重复读慢层”变成“复用快层”。
