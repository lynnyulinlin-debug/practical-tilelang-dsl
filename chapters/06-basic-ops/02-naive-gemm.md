# 6.2 Naive GEMM

如果说 Vector Add 证明的是“链路能跑通”，那朴素 GEMM 证明的就是“跑通之后，为什么性能会立刻拉开”。  
它把教程从最小闭环推进到第一个真正有优化空间的算子上，让读者看到性能问题是如何在一个普通三重循环里出现的。

这一页真正要建立的，是“看懂瓶颈”的习惯。  
朴素 GEMM 不是为了展示算法本身，而是为了把访存重复、数据复用差和内存受限这三件事摆到台面上。

## 为什么先写朴素版

GEMM 是教程里最重要的基础算子之一。  
这里先写最朴素的版本，目的是把后面所有优化的起点说清楚。

在当前仓库里，`examples/04_gemm_basic.py` 已经把 GEMM 的最小验证路径跑通到 CPU fallback：

- 可以先确认矩阵乘法定义没有问题
- 可以确认 `PrimFunc` 和 `tvm.build` 的组合可用
- 可以把 GPU 版本作为后续优化目标，而不是当前阻塞点

## 先看什么

### 1. 朴素实现

- 三重循环
- 全局内存直接读取
- 没有 tile，没有复用

这一版的关键不是“写法简单”，而是“问题暴露得彻底”。  
每个输出元素都要反复读取 A 和 B 的相关数据，但这些读取没有被块内复用承接下来，所以访存很容易先成为瓶颈。

### 2. 朴素版的问题

- 访存次数高
- 数据复用差
- 性能通常受内存限制

这正是朴素 GEMM 的教学价值：它把“为什么需要优化”变成可以直接观察的事实。  
当同一份输入被多次读却没被复用，读者就能自然理解为什么后面的 tile 和 shared memory 会有意义。

### 3. 基准意义

- 朴素版是优化前的参考点
- 后续所有版本都要和它对比

### 4. 当前口径

- 这一节的核心不是“实现细节更多”，而是“数据复用更少”
- 朴素 GEMM 适合作为性能下界
- 只要它足够清楚，后面的 shared memory 和 Tensor Core 优化就更容易解释

### 5. 这一节应该留下的判断标准

- 如果你看见三重循环，就要立刻想到访存重复
- 如果没有 tile，就几乎没有块内复用
- 如果每个输出元素都要单独读 A / B，很容易落在 memory-bound 一侧

朴素 GEMM 的教学价值，就是把“为什么要优化”讲成可以量化的事实。

它不是为了让代码显得不够聪明，而是为了把优化前的边界条件固定下来。  
后面无论是 shared memory 还是 Tensor Core，都是在这个边界上往上推。

## 这一节最后要记住什么

朴素 GEMM 的意义不是“性能差”，而是把优化空间显式地摆出来。
它是后续所有优化版本的对照组。

## 还能追问什么

- 朴素 GEMM 的主要瓶颈为什么首先是访存而不是算力
- 三重循环里哪些访问会重复
- 为什么没有 tile 时很难形成块内复用

## 最容易犯的错

- 误区 1：朴素 GEMM 只是教学，不值得认真分析
- 误区 2：只要是 GEMM，就天然应该快
- 误区 3：性能差只是因为实现不够聪明

朴素 GEMM 的作用，是把“为什么需要优化”变成可以解释的事实，而不是抽象口号。

读者如果能在这一页里看出“同样的数据为什么会被反复读”，后面的 shared memory 和 fragment 才会有明确的动机。
