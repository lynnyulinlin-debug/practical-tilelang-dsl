# 6.2 Naive GEMM

## 这一节要解决什么问题

GEMM 是教程里最重要的基础算子之一。  
这里先写最朴素的版本，目的是把后面所有优化的起点说清楚。

在当前仓库里，`examples/04_gemm_basic.py` 已经把 GEMM 的最小验证路径跑通到 CPU fallback：

- 可以先确认矩阵乘法定义没有问题
- 可以确认 `PrimFunc` 和 `tvm.build` 的组合可用
- 可以把 GPU 版本作为后续优化目标，而不是当前阻塞点

## 这一节要讲什么

### 1. 朴素实现

- 三重循环
- 全局内存直接读取
- 没有 tile，没有复用

### 2. 朴素版的问题

- 访存次数高
- 数据复用差
- 性能通常受内存限制

### 3. 基准意义

- 朴素版是优化前的参考点
- 后续所有版本都要和它对比

### 4. 当前口径

- 这一节的核心不是“实现细节更多”，而是“数据复用更少”
- 朴素 GEMM 适合作为性能下界
- 只要它足够清楚，后面的 shared memory 和 Tensor Core 优化就更容易解释

### 5. 这一节应该留下的判断标准

- 如果你看见三重循环，就要立刻想到访存重复
- 如果没有 tile，就几乎没有块内复用
- 如果每个输出元素都要单独读 A / B，很容易落在 memory-bound 一侧

朴素 GEMM 的教学价值，就是把“为什么要优化”讲成可以量化的事实。

## 这一节的结论

朴素 GEMM 的意义不是“性能差”，而是把优化空间显式地摆出来。
它是后续所有优化版本的对照组。

## 这一节可以进一步追问的问题

- 朴素 GEMM 的主要瓶颈为什么首先是访存而不是算力
- 三重循环里哪些访问会重复
- 为什么没有 tile 时很难形成块内复用

## 常见误区

- 误区 1：朴素 GEMM 只是教学，不值得认真分析
- 误区 2：只要是 GEMM，就天然应该快
- 误区 3：性能差只是因为实现不够聪明

朴素 GEMM 的作用，是把“为什么需要优化”变成可以解释的事实，而不是抽象口号。
