# 6.1 Vector Add

Vector Add 是最简单的算子，但它真正重要的地方，不在于计算本身，而在于它能把整条执行链路单独拎出来看。  
一个最小 kernel 能不能跑通，取决于输入输出定义、编译目标、执行方式和结果验证是否都连上了。只要这条链路有一处断了，后面的 GEMM 和优化版本都没有可靠起点。

## 为什么先从它开始

Vector Add 是最简单的算子，但它足够说明一条最重要的主线：  
一个 kernel 能否跑通，取决于输入输出定义、编译目标、执行方式和结果验证是否完整闭环。

在当前仓库里，`examples/03_vector_add.py` 已经把这条闭环跑通了：

- TE 版本可以 lowered 成 `PrimFunc`
- CPU fallback 可以在 `tilelang-tvm` 环境里执行
- 结果能和 NumPy 对上

这意味着 Vector Add 的价值不是“简单”，而是“足够小，便于确认整个工具链没断”。

## 基础算子的闭环

### 1. 最小算子结构

- 输入张量
- 输出张量
- 逐元素加法

这里不需要复杂的调度，也不需要 tile。  
它的作用只是把“数据从哪里来、到哪里去、怎么算、怎么算完怎么确认”这四个步骤串成一个最小闭环。

### 2. 正确性验证

- 和 NumPy 对比
- 误差阈值
- 为什么先验证正确性，再谈性能

先验证正确性不是保守，而是为了避免把错误的实现误当成性能结果。  
只要结果没对上，再快的 kernel 也没有教学意义。

### 3. 基础性能测量

- 执行时间
- 带宽估算
- 作为后续优化的 baseline

性能测量在这里的角色是建立参照系，不是下结论。  
对于 Vector Add 这种最小算子，真正有价值的是知道“最朴素的正确实现大概处在什么量级”，这样后面的优化才有对照。

### 4. 当前验证口径

- 当前环境没有可用 NVIDIA 驱动时，优先验证 CPU fallback
- GPU 版本只在驱动和后端都可用时再看性能数据
- 这一节的重点是“结果正确”和“流程完整”，不是追求峰值

## Vector Add 的验证闭环

Vector Add 不是为了体现复杂性，而是为了建立所有后续算子都要遵循的最小实践模板。
如果这一步都无法跑通，后面的 GEMM、Attention 和多后端迁移就没有可靠起点。

这一节的价值是把“能写一个 kernel”改写成“能完整验证一个 kernel”。  
这也是后面所有实战章节的共同前提。

如果还想继续往下追问，最值得先确认的三个点是：

- 这段代码的最小闭环到底由哪些步骤组成
- 正确性验证和性能统计分别在验证什么
- CPU fallback 在这里扮演什么角色

## Vector Add 的常见误区

- 误区 1：Vector Add 太简单，不值得单独讲
- 误区 2：只要能算出结果，就说明流程已经完整
- 误区 3：性能指标对最小算子没有意义

Vector Add 的价值在于它足够小，能把“流程是否完整”这件事单独拎出来。
