# 第 6 章导言：基础算子实战：Vector Add 与 GEMM

## 章节定位

这一章开始进入第一轮实战。  
目标不是再讲概念，而是把前面学到的分块、内存和调度方法落到基础算子上。  
这里的叙事顺序很重要：先从最简单的算子看清“正确性怎么保证”，再从 GEMM 看见“性能为什么会拉开差距”，最后才谈优化模板。

这一章的意义，不只是让读者“会做几个算子”，而是让读者第一次把前面章节里的判断真正用起来。  
从整套教程来看，这一章更像是桥建好之后的基础通行层：前面学到的分层、编译和硬件直觉，要先在最基础的算子上走通，后面的复杂 attention 和多后端迁移才有稳的起点。
Vector Add 负责确认流程不会出错，GEMM 负责把性能差异拉开，优化模板则负责把这些经验收成可复用的方法。

## 本章目标

- 完成基础算子从 baseline 到优化版的演进
- 用 GEMM 作为主线案例串起前面的方法
- 让读者第一次看到明确的性能提升
- 为后续高级算子章节建立信心和模板
- 让读者学会判断一个优化是“结构性改进”还是“参数性微调”

这组目标背后的顺序其实很固定：先让结果能对，再让数据流能顺，再让收益能看见，最后才把经验收成模板。  
只要这个顺序成立，第 6 章就不只是“基础算子实战”，而是后面复杂算子的起跑线。

## 本章判断标准

读完这一章，读者应该能判断：

- 一个算子是先补正确性，还是先做结构优化
- 朴素 GEMM 的瓶颈主要来自哪里
- shared memory 优化什么时候已经足够
- 什么时候应该继续走到 fragment / Tensor Core 路径

如果读者读完这一章，已经能在朴素版和优化版之间说清楚“到底是数据流变了，还是只是参数变了”，那这一章的教学目标就算达成了一半。  
剩下的一半，就是把这种判断方式迁移到更复杂的算子里。

## 内容块安排

### 6.1 Vector Add
- 最小算子
- 正确性验证
- 基础性能基准
- 用最小案例建立“先验证，再比较，再优化”的习惯

### 6.2 Naive GEMM
- 朴素实现
- 性能瓶颈
- 让读者看到一个常见事实：最常见的瓶颈往往不是代码复杂，而是数据流不对

### 6.3 Shared memory GEMM
- tile 和数据复用
- shared memory 版本
- 说明第一次结构性优化为什么通常来自数据复用

### 6.4 Tensor Core / Fragment GEMM
- 更接近硬件的实现方式
- 进一步性能提升
- 解释为什么更贴近硬件的表达往往需要更明确的约束

### 6.5 基础优化总结
- 向量化、对齐、布局
- 算子优化模板

### 6.6 本章总结
- 从“能写”走向“能优化”

## 写作提醒

这一章最容易写成“算子列表”，但更好的写法是把它写成一条演进路径：从最小正确性，到朴素性能，再到结构优化。  
读者如果能跟着这条路径理解 GEMM，后面更复杂的算子就会自然接上。

换句话说，第 6 章不是一组独立示例，而是一条把“能跑”推进到“能优化”的连续路径。  
这条路径一旦建立，后面第 7 章的复杂 attention 就不会显得突兀。
