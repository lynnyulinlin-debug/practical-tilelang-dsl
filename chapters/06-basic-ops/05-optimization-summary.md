# 6.5 基础优化总结

前面三个 GEMM 版本其实讲的是同一个故事，只是关注点不同：先把问题跑通，再把重复读取压下去，最后再看能不能让硬件单元更高效地吃数据。  
这一节的作用，就是把这三次变化收成一个可复用的调优模板。

前面三个 GEMM 版本展示了从“能跑”到“跑快”的演进。  
这一节把基础优化方法收束成一个可复用的模板。

如果前面的内容是“做了三次不同的实现”，那么这一节就是“把这三次实现抽象成一套调优顺序”。

## 基础优化怎么归纳

### 向量化、对齐和布局

- 减少单次访存和计算的调度开销
- 让访问更规整
- 减少额外事务
- 行主序 / 列主序
- 适配访问方向

### 版本演进

- 朴素版
- shared memory 版
- fragment / Tensor Core 版
- 以后所有高级算子都能复用这个思路

版本演进背后的顺序其实很固定：先正确性，再复用，再硬件匹配。  
如果把顺序反过来，一上来就追求最复杂版本，通常只会让问题更难定位。

### 当前怎么判断

- 先看有没有正确性闭环
- 再看有没有数据复用
- 然后再看是否已经贴近硬件计算单元
- 最后才讨论更细的布局、对齐和参数搜索

**表：GEMM 三个版本对比表**

> 这里要记录实现复杂度、性能收益和适用场景。
> 这个表最好能直接给出一个选择建议：什么时候停在 shared memory，什么时候值得继续上 fragment。

| 版本 | 主要目标 | 主要收益 | 主要代价 | 适用场景 |
|------|---------|---------|---------|---------|
| 朴素版 | 正确性 | 最容易理解 | 访存多、性能低 | 先验证闭环 |
| shared memory 版 | 数据复用 | 明显减少全局访存 | 需要 tile 和同步 | 大多数基础优化 |
| fragment / Tensor Core 版 | 硬件匹配 | 进一步提高吞吐 | 布局和对齐更复杂 | 目标硬件支持矩阵单元时 |

这张表是本节最重要的落点，后续优先把不同版本的选择条件放到同一张表里。

它真正要帮助做的，不是记住三种写法，而是学会判断“当前阶段该停在哪一层”。  
这比盲目追求更高版本更重要。

## 基础优化的常见偏差

- 误区 1：优化一定要一路走到最复杂版本
- 误区 2：所有版本都应该在同一条标准下比较
- 误区 3：只要有更多优化手段，就应该全部加上

这张表的作用，是先判断“停在哪一层最合理”，不是强行推到最高复杂度。

## GEMM 调优顺序

1. 先确认算子正确
2. 再确认访存模式和 tile 是否合理
3. 再判断是否值得进入 fragment / Tensor Core 路径
4. 最后才做更细的布局、对齐和参数搜索

如果要写 benchmark 记录，最实用的做法是直接沿用统一模板，并把当前版本作为最小样例放在表格后面，作为数据来源和复测入口。

```python
from examples.benchmark_record_template import set_benchmark_record

record = set_benchmark_record(
    scenario="chapter 6 GEMM optimization",
    operator="GEMM",
    platform="NVIDIA",
    target="cuda",
    dtype="float16",
    shape="M=1024,N=1024,K=1024",
    baseline="naive GEMM",
    optimization="shared memory + fragment",
)
```

benchmark 记录里先填的，应该是 baseline 和 optimization 这两个字段：

- baseline 记录的是当前最朴素、最稳定的对照版
- optimization 记录的是这一步真正改变了什么
- 如果后面继续加调优参数，也应该在同一模板下补充，而不是改成另一套格式
