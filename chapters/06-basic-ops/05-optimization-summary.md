# 6.5 基础优化总结

## 这一节要解决什么问题

前面三个 GEMM 版本展示了从“能跑”到“跑快”的演进。  
这一节把基础优化方法收束成一个可复用的模板。

如果前面的内容是“做了三次不同的实现”，那么这一节就是“把这三次实现抽象成一套调优顺序”。

## 这一节要讲什么

### 1. 向量化

- 减少单次访存和计算的调度开销

### 2. 对齐

- 让访问更规整
- 减少额外事务

### 3. 布局

- 行主序 / 列主序
- 适配访问方向

### 4. 作为优化模板

- 朴素版
- shared memory 版
- fragment / Tensor Core 版
- 以后所有高级算子都能复用这个思路

### 5. 当前归纳

- 先看有没有正确性闭环
- 再看有没有数据复用
- 然后再看是否已经贴近硬件计算单元
- 最后才讨论更细的布局、对齐和参数搜索

## 这里应该放的表

**表：GEMM 三个版本对比表**

> 这里要记录实现复杂度、性能收益和适用场景。
> 这个表最好能直接给读者一个选择建议：什么时候停在 shared memory，什么时候值得继续上 fragment。

| 版本 | 主要目标 | 主要收益 | 主要代价 | 适用场景 |
|------|---------|---------|---------|---------|
| 朴素版 | 正确性 | 最容易理解 | 访存多、性能低 | 先验证闭环 |
| shared memory 版 | 数据复用 | 明显减少全局访存 | 需要 tile 和同步 | 大多数基础优化 |
| fragment / Tensor Core 版 | 硬件匹配 | 进一步提高吞吐 | 布局和对齐更复杂 | 目标硬件支持矩阵单元时 |

这张表是本节最重要的落点，后续优先把不同版本的选择条件放到同一张表里。

## 这一节可以进一步追问的问题

- 一个算子现在最缺的是正确性、复用还是硬件匹配
- 这个版本是否已经足够作为稳定 baseline
- 是否值得进入更复杂的布局和参数搜索

## 常见误区

- 误区 1：优化一定要一路走到最复杂版本
- 误区 2：所有版本都应该在同一条标准下比较
- 误区 3：只要有更多优化手段，就应该全部加上

这张表的作用，是帮读者先判断“停在哪一层最合理”，不是强行推到最高复杂度。

## 这里应该补的调优顺序

1. 先确认算子正确
2. 再确认访存模式和 tile 是否合理
3. 再判断是否值得进入 fragment / Tensor Core 路径
4. 最后才做更细的布局、对齐和参数搜索

## 这一节的结论

基础算子的价值，不在于列出多个版本，而在于把一个优化方法论讲透。
第 6 章真正要留下的是：优化顺序，而不是单个代码片段。
读者看完后，应该能自己判断一个 GEMM 该停在 shared memory，还是继续往硬件匹配方向走。
