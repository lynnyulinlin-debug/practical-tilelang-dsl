# 6.4 Tensor Core / Fragment GEMM

当 shared memory 已经把复用问题解决到一定程度后，接下来就要问：能不能再把数据摆成硬件最喜欢的样子。  
这一节讲的就是这一步。它不是简单地“再快一点”，而是把 GEMM 从块级复用进一步推向硬件匹配。

## 为什么要继续往硬件靠

shared memory 已经能把性能拉上来，但如果目标硬件支持矩阵计算单元，还可以进一步把数据组织到更接近计算核心的形态。

这一步的教学重点是：  
同样是 GEMM，真正决定性能的并不只是“有没有矩阵单元”，而是“数据布局是否已经提前适配好了它”。

## 先看什么

### 1. Fragment 的角色

- fragment 是更接近计算单元的中间存储
- 它比 shared memory 更贴近最终算子执行

fragment 的意义在于，它不是单纯缓存数据，而是提前把数据整理成计算单元能直接吃下去的形状。  
这意味着它承接的是“最后一段路”，目标是让计算单元少做解释、少做转换、少做额外搬运。

### 2. Tensor Core 思路

- 让数据布局匹配矩阵单元规格
- 让计算单元吃到合适形状的数据
- 以常见的 `m16n16k16` 一类 tile 规格为例，可以更直观看出数据为什么要先被整理成 fragment

这里最关键的不是某个固定 tile，而是“形状要先对上”。  
Tensor Core 这类单元对输入规格很敏感，所以不是简单把矩阵丢过去就行，而是要先把 tile、对齐和布局整理好。

### 3. 进一步优化的关注点

- tile 形状
- register 压力
- 数据对齐
- 这些因素决定了“能不能用好 Tensor Core”，而不是只看“有没有 Tensor Core”

这一层会直接把前面两章的判断重新拉回来：  
shared memory 负责复用，fragment 负责就近计算，而 Tensor Core 只有在形状、对齐和资源压力都说得过去时，才会真的带来收益。

### 4. 当前仓库的讲法

- 当前仓库先把概念讲清楚，再给出硬件匹配的方向
- 如果没有可用 GPU 驱动，这一节仍然可以作为结构理解来读
- 这里的目标不是在本机复现峰值，而是理解为什么后续实现会朝 fragment 组织靠拢

## 这一节最后要记住什么

fragment / Tensor Core 这一步，是 GEMM 从“块优化”走向“硬件匹配”的关键一步。
在教学上，它帮助读者从“分块为什么有用”过渡到“为什么要配合具体计算单元来分块”。
如果说 shared memory 解决的是“怎么复用数据”，那 fragment 解决的就是“怎么把数据摆成计算单元最喜欢的样子”。

## 还能追问什么

- 为什么矩阵单元对 tile 规格这么敏感
- fragment 的形状为什么不能随便改
- 数据布局和对齐为什么会直接影响单元利用率

## 最容易犯的错

- 误区 1：Tensor Core 一定比 shared memory 版本快很多
- 误区 2：只要硬件支持矩阵单元，就一定应该立刻切过去
- 误区 3：fragment 只是一个更小的缓存层

硬件匹配的前提是形状和布局先对上，而不是“有硬件就用硬件”。

所以这一节要让读者看到的，不只是 Tensor Core 更快，而是“更快”背后需要更严格的前置条件。  
这就是从块优化走向硬件匹配的真正分界线。
