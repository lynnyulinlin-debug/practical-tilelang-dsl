# 10.4 逐步替换与优化

## 逐步替换为什么要分轮做

真实迁移不会一步到位，通常要经历一轮一轮替换和优化。

这一节要把“逐步替换”讲成可执行的方法，而不是经验性的描述。

## 逐步替换每轮要写什么

### 1. 第一步：先替换出来

- 先保证能跑
- 先保证结果正确

### 2. 第二步：再做局部优化

- shared memory
- layout
- pipeline

### 3. 第三步：再做融合

- 中间张量消除
- 多阶段合并

### 4. 第四步：再看整体收益

- 单点收益
- 累积收益
- 代价和复杂度

### 5. 每轮都要固定记录

- 改动点
- 为什么改
- 改完后的收益
- 引入的新风险
- 是否值得保留

## 逐步替换的判断原则

- 先讲控制变量，再讲收益累积
- 不要把所有优化揉成一次性重写
- 这一节应该能照着做自己的迁移迭代表

这里还要额外强调一件事：每一轮优化都要能回头解释。
如果一轮改动之后只知道“更快了”，却说不清“改了什么、为什么快、代价是什么”，那这一轮就还不够像工程迭代。

## 每一轮固定记录什么

- 改动点
- 为什么改
- 带来的收益
- 新引入的风险

如果项目里有多个后端，还要顺手记录这轮改动是不是跨平台可复用。
这样后面做多后端适配时，就不会把本来可以共用的优化拆散。

## 逐步替换的收口

逐步替换的关键是控制变量，不要一口气把所有优化一起做。
如果没有阶段边界，后面就很难判断收益到底来自哪里。

这节真正要传达的，不是“优化次数越多越好”，而是“每一轮都要能被解释、被复核、被复用”。

## 常见误区

- 误区 1：一次性重写更有效
- 误区 2：只要结果变快，就不需要记录过程
- 误区 3：阶段边界没有必要

逐步替换的核心，是让收益可归因、风险可追踪。

## 逐步替换检查顺序

这一节最适合画成迭代表：

- 第几轮
- 改了什么
- 为什么改
- 收益多少
- 代价是什么

这样能把“优化”理解成一个阶段化工程，而不是一次性魔法。
