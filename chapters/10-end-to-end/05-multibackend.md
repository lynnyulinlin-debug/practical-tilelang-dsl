# 10.5 多后端适配

## 要解决什么问题

这一节不是再讲后端名词，而是讲同一套迁移结果如何在不同平台上落地。

它承接第 8 章，但视角更工程化：关注的是“同一套迁移结果如何变成可复用的多后端交付物”。

更具体地说，这一节要解决的不是“平台差异有没有”，而是“差异出现时，先改什么、后改什么、哪些东西尽量不动”。
如果没有这个顺序，多后端适配很容易退化成重新写一遍实现，最后只剩下平台介绍，没有迁移方法。

## 要写的内容

### 1. target 选择

- 不同平台的编译目标
- 后端入口的差异

### 2. 参数迁移

- 哪些 tile 参数可以复用
- 哪些必须重调

### 3. 布局和同步

- 访问模式差异
- shared memory / 片上缓冲差异
- 同步方式差异

### 4. 适配顺序

1. 先确认目标平台是否可编译
2. 再确认目标平台是否可正确运行
3. 然后只改必要的布局和 tile 参数
4. 最后再补平台专有优化

这个顺序的核心，是先把“功能闭环”做出来，再把“性能闭环”做出来。
对于多后端项目来说，能编译、能跑通、能对齐结果，比一开始追求某个平台的极限性能更重要。

如果顺序反过来，常见后果是：

- 还没确认正确性，就开始调平台专有参数
- 还没固定公共参数，就先做局部优化
- 还没区分复用边界，就把每个平台都写成独立实现

### 4. 当前写作口径

- 先讲可以复用什么，再讲必须重调什么
- 不要把多后端适配写成重新发明一遍实现
- 这一节的重点是交付可迁移参数，而不是平台介绍

这一节还要补一个判断：哪些差异属于“平台约束”，哪些差异只是“当前实现习惯”。
前者必须记录，后者尽量收敛到统一模板里。

换句话说，这一节真正要输出的不是平台清单，而是适配边界清单。
读者看完后，应该能知道“哪些字段能直接抄、哪些字段需要重新测、哪些字段最好统一命名”。

### 5. 一个现实的边界

- 不是所有参数都能跨平台复用
- 不是所有 layout 都有同样收益
- 不是所有平台都应该追求同一个最优点

**表：多后端迁移参数表**

> 这里要记录 target、tile、吞吐、带宽、显存峰值、版本信息等统一字段。

如果表格要落成最终版本，建议至少补上这些列：

- 后端名称
- 编译目标
- tile 参数
- 布局策略
- 同步方式
- 主要瓶颈
- 是否复用前一版参数
- 备注

这样表格就不只是记录结果，而是记录“为什么这一版能在该平台上成立”。

## 小结

多后端适配的核心不是“完全不改”，而是“逻辑统一、参数分平台调”。
它是第 8 章方法在真实项目中的延续。

## 这一节可以进一步追问的问题

- 哪些参数是平台无关的
- 哪些参数必须重新评估
- 多后端适配的第一步应该先看什么

## 常见误区

- 误区 1：多后端适配就是复制粘贴
- 误区 2：同一套参数可以直接跨平台复用
- 误区 3：只要有多后端能力，就应该所有平台都做满

多后端适配的价值，是让同一条迁移路线能在不同目标上复用，而不是强行消灭差异。

另一个常见误区，是把多后端适配写成“平台性能排行榜”。
这一章关心的是迁移边界和交付结构，而不是谁永远最快。

## 讲义提示

这一节最适合强调“先能跑，再谈好”：

- 先跑通目标平台
- 再把参数收口
- 最后才做平台专有优化

这比一开始就追求极限性能更适合工程迁移。
