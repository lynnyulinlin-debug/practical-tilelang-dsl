# 10.5 多后端适配

## 要解决什么问题

这一节不是再讲后端名词，而是讲同一套迁移结果如何在不同平台上落地。

它承接第 8 章，但视角更工程化：关注的是“同一套迁移结果如何变成可复用的多后端交付物”。

## 要写的内容

### 1. target 选择

- 不同平台的编译目标
- 后端入口的差异

### 2. 参数迁移

- 哪些 tile 参数可以复用
- 哪些必须重调

### 3. 布局和同步

- 访问模式差异
- shared memory / 片上缓冲差异
- 同步方式差异

### 4. 适配顺序

1. 先确认目标平台是否可编译
2. 再确认目标平台是否可正确运行
3. 然后只改必要的布局和 tile 参数
4. 最后再补平台专有优化

### 4. 当前写作口径

- 先讲可以复用什么，再讲必须重调什么
- 不要把多后端适配写成重新发明一遍实现
- 这一节的重点是交付可迁移参数，而不是平台介绍

### 5. 一个现实的边界

- 不是所有参数都能跨平台复用
- 不是所有 layout 都有同样收益
- 不是所有平台都应该追求同一个最优点

## 小结

多后端适配的核心不是“完全不改”，而是“逻辑统一、参数分平台调”。
它是第 8 章方法在真实项目中的延续。

## 这一节可以进一步追问的问题

- 哪些参数是平台无关的
- 哪些参数必须重新评估
- 多后端适配的第一步应该先看什么

## 常见误区

- 误区 1：多后端适配就是复制粘贴
- 误区 2：同一套参数可以直接跨平台复用
- 误区 3：只要有多后端能力，就应该所有平台都做满

多后端适配的价值，是让同一条迁移路线能在不同目标上复用，而不是强行消灭差异。

## 讲义提示

这一节最适合强调“先能跑，再谈好”：

- 先跑通目标平台
- 再把参数收口
- 最后才做平台专有优化

这比一开始就追求极限性能更适合工程迁移。
