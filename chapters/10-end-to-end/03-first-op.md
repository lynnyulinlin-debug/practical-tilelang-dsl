# 10.3 第一个可迁移算子

## 要解决什么问题

这一节只做一件事：选一个最容易落地的算子，把迁移过程真正走通。

它是整章的“第一个闭环点”：如果这里跑不通，后面就很难证明迁移流程成立。

这也是整章里最需要“少而稳”的地方。
第一个算子不是用来展示最高收益的，而是用来验证迁移链条是否能够闭合。

## 选择原则

第一个算子最好满足：

- 结构稳定
- 输入输出清晰
- 容易验证正确性
- 能代表后续算子的共性

还要补一个隐含条件：它最好不是整个子图里最复杂的那一个。
如果一开始就选最复杂算子，后面的 baseline 对齐、误差定位和性能归因都会变得很重。

## 推荐优先顺序

1. `GEMM`
2. `RMSNorm`
3. `Softmax`
4. `Attention` 的局部子块

## 这一节要写的内容

### 1. baseline

- 原始实现是什么
- 在当前系统里怎么跑
- 有什么瓶颈

### 2. TileLang 版本

- 计算表达
- 分块策略
- 验证方法

### 3. 对比结果

- 正确性
- 延迟
- 吞吐
- 显存占用

### 4. 当前写作口径

- 先讲为什么这个算子适合作为第一个迁移目标
- 再讲 baseline 和 TileLang 版本怎么对照
- 最后把对比维度固定住，方便后续章节复用

这里的“对照”不只是数值对照，还包括结构对照。
读者需要知道 baseline 的瓶颈在哪里，TileLang 版本到底改了哪一层，才能理解为什么这个闭环有意义。

### 5. 一个最小闭环

1. 先把 baseline 跑通
2. 再把 TileLang 版本跑通
3. 再做正确性对齐
4. 最后补 benchmark

这四步的重点是先证明“可行”，再证明“值得优化”。
如果跳过正确性对齐，后面的性能数字就没有足够的可信度。

这个闭环的关键不是一次做到最好，而是先证明迁移流程真的能闭合。

## 小结

第一个可迁移算子的价值，不是收益最大，而是把迁移方法打通。
它应该帮助读者建立“先跑通，再优化，再扩展”的节奏感。

换句话说，这一节是整章的起跑线，不是终点线。
它要让读者看到方法是怎么闭合的，而不是只看到最后快了多少。

## 这一节可以进一步追问的问题

- 为什么这个算子最适合作为第一个闭环
- 迁移的 baseline 和 TileLang 版本分别证明了什么
- 为什么先闭合流程比先追求最优更重要

## 常见误区

- 误区 1：第一个算子必须是收益最高的
- 误区 2：只要算子简单，就不需要认真比较
- 误区 3：迁移流程可以跳过 baseline

第一个闭环的意义，是证明迁移方法可行，而不是证明它已经最优。

## 讲义提示

这节最适合把“为什么选它”单独列成一小段：

- 它为什么容易跑通
- 它为什么容易验证
- 它为什么能代表后续算子

只要这三点讲清楚，读者就会理解为什么它是第一个闭环点。
