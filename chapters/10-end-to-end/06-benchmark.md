# 10.6 端到端 benchmark

## 端到端 benchmark 要解决的判断

端到端项目的最终判断标准不是某个算子单点多快，而是整体链路有没有真实收益。

这一节要把“项目有没有价值”落到统一指标上，而不是只看局部优化感觉。

这里的重点不是把指标堆得越多越好，而是把指标分层后各自看懂。
如果单算子、子图和端到端三层混在一起，很容易误把局部收益当成整体收益。

## 端到端 benchmark 的指标

- token/s
- latency
- throughput
- 峰值显存
- 平均显存
- 迁移前后加速比
- 小规模趋势扫描结果

## benchmark 记录

- 直接复用 [QUICKREF.md](../../QUICKREF.md) 里的 benchmark 模板
- 端到端场景只需要在其基础上补充模型名、子图名和系统级指标

模板里最好明确三件事：

1. 测试对象是什么
2. 统计口径是什么
3. 这次数据能说明哪一层的结论

只有这三件事都清楚，后面的表和图才不会变成孤立数字。

```python
from examples.benchmark_record_template import set_benchmark_record

record = set_benchmark_record(
    scenario="chapter 10 end-to-end migration",
    operator="subgraph",
    platform="multi-backend",
    target="cuda",
    dtype="float16",
    shape="subgraph specific",
    baseline="original implementation",
    optimization="iterative migration + backend tuning",
)
```

如果启用了小规模扫描，记录里还应该额外补上：

- `sweep_enabled`
- 不同 `seq_len` 下的 latency
- 不同 `seq_len` 下的 speedup

## benchmark 的对比维度

- 单算子前后
- 子图前后
- 端到端前后
- 单后端与多后端

这四类对比不要混着写。
单算子前后主要看局部结构是否成立，子图前后主要看中间张量是否减少，端到端前后主要看业务链路是否受益，多后端对比主要看迁移边界是否稳定。

## benchmark 的记录顺序

1. 先记录单算子指标
2. 再记录子图级指标
3. 然后记录端到端指标
4. 最后才做跨后端比较

如果项目里启用了 `sweep`，还应该把规模扫描插在单算子和端到端之间，作为趋势验证的一层。

这个顺序的目的，是先把局部事实钉住，再看整体结论是否成立。
如果反过来先写端到端结论，后面往往很难解释收益到底来自哪里。

## 端到端 benchmark 的判断标准

- 先统一指标，再统一统计方式
- 不要把局部 benchmark 当作最终结论
- 如果有趋势扫描，优先看趋势是否稳定，而不是只看一个 shape 的最优值
- 如果数据不完整，也要先把表格结构定下来

## 端到端 benchmark 的收束

端到端 benchmark 的价值在于把单点优化收益转化成工程收益。
如果没有整体指标，前面的所有优化都很难证明其业务价值。
如果有趋势扫描，还能知道这种收益是否只是某个 shape 的偶然结果。

这一节还要一起记住：

- 单算子、子图和端到端指标分别说明什么
- 哪些指标应该固定口径
- 为什么跨后端比较一定要先统一统计方式

常见误区也要放在一起看：

- 误区 1：单点 benchmark 就足够说明项目价值
- 误区 2：不同层级的指标可以混着看
- 误区 3：只要性能提高，口径就不重要
- 误区 4：某个 shape 上的最优值可以直接推广到所有规模

如果要把这节落成最终表格，建议分成三层：

- 算子级
- 子图级
- 端到端级

这样会明白 benchmark 不是只记录一个数字，而是记录一条完整链路。
