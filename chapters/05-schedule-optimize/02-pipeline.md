# 5.2 流水线：T.Pipelined 与双缓冲

如果并行层级是在决定“谁来做”，那流水线就是在决定“什么时候做”。  
这一节最重要的直觉，是把加载和计算看成两段可以交错的工作，而不是必须严格串行的一条直线。

这一节可以直接视为本章里“异步 / 重叠”的主讲位置：它想解决的不是把同步删掉，而是把原本必须等待的阶段尽量并起来。

很多 kernel 慢，不是因为算不动，而是因为在等数据。  
并行已经大致对齐之后，性能仍然上不去，往往就说明问题不在“谁来做”，而在“做的时候是不是还在等”。

双缓冲是这个思路最典型的实现方式。  
它的做法并不神秘，只是提前准备下一批数据，让当前批次的计算可以连续推进。

这个思路对 GEMM 特别有效，因为 GEMM 天然就有可重叠的加载与计算阶段。  
只要数据流足够稳定，流水线就能把很多原本停在中间的时间片收掉。

## 双缓冲要处理的等待

当数据搬运和计算完全串行时，计算单元在等，内存通道也在等，最后看到的就是一连串空泡。  
双缓冲的目的，就是把这部分等待提前吸收掉：当前块在算，下一块在搬，两个阶段彼此交错。

这种交错并不是无代价的。  
它需要额外的缓冲区，也需要更复杂的调度；如果资源压力本来就很高，或者算子根本没有明显的加载-计算重叠空间，收益就可能被抵消。

真正该记住的，不是双缓冲这个名字，而是它背后的判断：如果 kernel 主要卡在等数据，流水线通常比继续堆并行更值；如果加载和计算天然能重叠，就应该优先考虑双缓冲。

## 流水线和并行的衔接

这一节和上一节是连着的。  
并行负责把协作结构摆对，流水线负责把时间结构压平。

也正因为如此，流水线不应该被看成一个独立技巧，而应该被看成调优顺序中的第二步。

读完这一节，至少应该能回答三件事：

1. 为什么很多 kernel 慢的根源其实是“在等”
2. 为什么双缓冲能把等待压缩掉一部分
3. 为什么流水线不是所有算子都同样有效

这些问题说清楚以后，后面的 layout 才会有位置。
