# 5.1 并行策略：T.Parallel 的层级选择

## 要说明的问题

并行不是越多越好，而是要匹配工作负载和硬件层次。

## 要点

- Grid / Block / Thread 的差异
- 不同层级的同步和通信成本
- 为什么并行粒度会影响 occupancy

## 这里应该解释的现象

- 为什么并行不是越细越好
- 为什么线程数很多，吞吐却不一定高
- 为什么选择错误的并行层级会把后续优化空间挤掉

## 这一节应该留下的判断标准

- 如果工作负载适合 block 内协同，就不要先把问题拆得过碎
- 如果同步频繁，过细的并行粒度可能带来额外开销
- 如果 occupancy 已经被资源压住，单纯加并行度通常不会有明显收益

## 常见误区

- 误区 1：更多线程一定更快
- 误区 2：并行层级只影响代码写法，不影响性能结构
- 误区 3：先把并行拉满，再考虑其他优化

并行层级是调优的入口选择，不是越靠前越好，也不是越细越好。
