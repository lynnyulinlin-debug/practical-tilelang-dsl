# 5.1 并行策略：T.Parallel 的层级选择

并行策略往往是第一个看起来很技术、实际却最容易选错的地方。  
很多 kernel 不是因为并行不够才慢，而是因为并行拆得太细，把本来可以在 block 内完成的协作，提前拆成了更贵的同步和调度。

这一节要先纠正一个常见误区：并行不是越早拉满越好，而是要先看工作负载需要怎样的协作，再决定把计算切到哪一层。

## 先定边界

Grid、Block、Thread 不是简单的命名差异，而是三种不同的协作边界。  
Grid 更像任务分发，Block 更像局部协同，Thread 更像最细粒度的执行单元。  
一旦把协作放错层级，后面的同步、通信和资源占用都会跟着偏掉。

不同层级的同步和通信成本也完全不一样。  
有些协作在 block 内就能完成，硬拆到更细的粒度，只会把本来可以局部消化的工作变成更贵的跨层协调。  
这也是为什么并行层级不是一个“代码怎么写”的问题，而是一个“协作放在哪里最划算”的问题。

并行粒度还会直接影响 occupancy。  
线程数看起来更多，不代表吞吐一定更高；如果寄存器、shared memory 或同步关系先把资源卡住，再继续加并行通常只会让调度更拥挤。

## 为什么会选错

并行策略之所以容易出错，是因为它看起来像性能问题，实际上常常是结构问题。  
很多 kernel 表面上像是“线程不够”，真正的问题却是协作边界放错了：本来可以在局部解决的事，被拆成了更远的同步和更重的调度。  
一旦这种边界错了，后面的 tile、流水线和布局优化都只能在错误的结构上修修补补。

所以这一节真正要留下的，不是“线程越多越好”这种老结论，而是一个更稳的判断：  
如果工作负载适合 block 内协同，就不要先把问题拆得过碎；如果同步频繁，过细的并行粒度往往只会放大额外开销；如果 occupancy 已经被资源压住，单纯加并行度通常不会带来明显收益。

## 这一节应该留下什么

并行层级是调优的入口选择，不是越靠前越好，也不是越细越好。  
这一节真正要留下的判断，是“把并行放在哪里最划算”，而不是“开了多少线程”。

换句话说，这一节承担的是“先定边界”的角色。
