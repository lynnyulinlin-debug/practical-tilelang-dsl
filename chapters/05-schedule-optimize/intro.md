# 第 5 章导言：调度优化与性能分析

## 章节定位

这一章解决“怎么把已经写对的 kernel 调得更快”。  
它是从结构化表达走向系统化调优的过渡章。  
前面几章已经把 kernel 组织起来了，这一章开始追问：当代码已经正确，性能为什么还卡住，应该先改哪里，哪些改动只是表面热闹，哪些才是真正的收益来源。

可以把这一章理解成一次真实的排查过程：你拿到一个能跑的 kernel，先发现它慢，再一步步分辨问题到底出在并行边界、等待时间、访问路径，还是参数空间。

如果把第 1 章那座桥和第 4 章的 tile / memory 层次连起来看，第 5 章就是开始回答：这些结构为什么最终会表现成访存是否合并、等待是否被藏起来、布局是否把通信边界压小了。也就是说，这一章不是再补一套新概念，而是把前面建立的结构感翻成性能判断。

## 本章目标

- 选择合适的并行层级
- 用流水线和双缓冲隐藏延迟
- 用布局和 swizzle 改变访问模式
- 用 Roofline 和 profiler 形成诊断闭环
- 让读者知道“调优的顺序”和“调优的停止点”同样重要

这几个目标不是平行罗列的，而是有前后顺序的。  
先看并行层级，再看流水线，再看布局，最后才是 Roofline 和 autotuning。

## 本章判断标准

读完这一章，读者应该能判断：

- 先改并行层级，还是先改流水线
- 先改布局，还是先看算术强度
- 某个 kernel 是该继续调参，还是应该停下来验证当前结构已经足够好
- autotuning 应该在哪里介入，哪里不该盲搜

## 内容块安排

### 5.1 并行策略
- Grid / Block / Thread
- 并行层级选择
- 先从并行层级入手，因为错误的并行边界会让后面的优化都打折扣

### 5.2 流水线与双缓冲
- 计算与加载重叠
- latency hiding
- 解释为什么“把时间藏起来”比“单纯加资源”更重要

### 5.3 内存布局优化
- 行主序 / 列主序
- `T.annotate_layout`
- `swizzle`
- 说明布局不是修饰项，而是访问模式和硬件路径的改写

### 5.4 Roofline 与性能分析
- 算术强度
- memory-bound / compute-bound
- profiler 入口
- 把分析工具变成判断工具，而不是只看图表的工具

### 5.5 自动调优
- 搜索空间
- AutoTVM
- 参数评估

### 5.6 本章总结
- 给出一个调优闭环模板

如果把这一章当成故事来读，它的推进顺序就是：先定边界，再压等待，再改路径，再判断瓶颈，最后把候选配置筛成可执行方案。

## 本章最小深度要求

这一章至少要让读者回答：

1. 为什么要先看并行层级，再看流水线，再看布局
2. 为什么 Roofline 不是结果本身，而是判断工具
3. 为什么调优不能只看单次 benchmark
4. 为什么要知道什么时候该停

如果只是记住几个优化名词，这章还不够。
