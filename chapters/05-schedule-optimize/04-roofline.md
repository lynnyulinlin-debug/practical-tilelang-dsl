# 5.4 Roofline 模型与性能分析

前面几节讲的是怎么改，这一节讲的是先怎么判断。  
Roofline 的价值不在于给出一个漂亮的图，而在于帮你把“当前 kernel 更像卡在内存还是卡在计算”这件事说清楚。

如果把第 5 章前半段看成一次真实的调优排查，那么 Roofline 就是这次排查里的分流点。  
它先把问题粗分成两类：更像数据搬运受限，还是更像算力受限。  
只要这一步判断错了，后面的并行、流水线、布局和自动调优就很容易走偏。

Roofline 之所以重要，是因为它把“看起来都慢”的 kernel 拆成了可解释的两种情况。  
有的 kernel 是算术强度太低，做一点计算就要搬很多数据；有的 kernel 是算术强度已经不低了，继续压访存的收益反而有限。  
这两种情况在外观上都可能表现为“慢”，但真正该改的方向完全不同。

## Roofline 的分流

算术强度是 Roofline 里最关键的概念。  
它不是一个抽象名词，而是一个判断工具：如果同样做一批计算，却要反复搬运更多数据，那么问题就更偏内存；如果搬运量相对不高，瓶颈就更可能转向计算单元。

这也是为什么 Roofline 不能单独替代 profiler。  
Roofline 负责告诉你“更像哪一类瓶颈”，profiler 负责告诉你“时间具体花在什么地方”。  
前者给方向，后者给证据，两者合起来，调优才不会变成拍脑袋。

换个更直观的说法，Roofline 更像是一张路线图，而不是终点图。  
它告诉你下一步先看哪一边：如果更像内存受限，就优先想办法减少流量和提高复用；如果更像计算受限，就别继续把时间花在访存微调上。  
这一步建立起来以后，后面的调优动作才有先后顺序。

读完这一节，至少应该能回答三件事：

1. 当前 kernel 更像 memory-bound 还是 compute-bound
2. Roofline 和 profiler 各自负责什么
3. 为什么调优不能一上来就盲目改参数

这三件事一旦说清楚，后面的 autotuning 才不会变成盲搜，而是会变成有方向的筛选。

这一节真正要传达的，不是一个固定阈值，而是一种判断顺序。  
先用 Roofline 判断大方向，再用 profiler 找证据，最后决定到底先改哪一层。  
这也是第 5 章的收口逻辑：先判断，再行动。

如果没有这一步，后面的优化很容易变成“哪里都改一点”，但没有真正解决主瓶颈。  
所以 Roofline 的作用不是给一个绝对答案，而是帮你决定下一步先改哪里。

如果把前面三节看成是在调整结构、时间和路径，那么 Roofline 就是在给这些动作排队。  
它不负责告诉你该怎么改代码，而是先告诉你哪一类改动更有可能见效。  
这也是为什么它应该出现在并行、流水线、布局之后，作为这一章前半段的判断收束。
